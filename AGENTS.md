# 项目开发流程

记住本项目的开发流程：

**修改源码** → **执行 clean** → **执行 install** → **执行 build** → **执行 package** → **执行 e2e 脚本** → **用户手动启动 e2e 生成的测试项目** → **阅读测试项目生成的代码** → **从测试项目提取 SDD 流程的缺陷** → **纳入当前 feature 修复或者加入未来 feature**

## 流程说明

1. **修改源码** - 在源代码中进行功能开发或缺陷修复
2. **执行 clean** - 清理之前的构建产物
3. **执行 install** - 安装依赖
4. **执行 build** - 构建项目
5. **执行 package** - 打包项目
6. **执行 e2e 脚本** - 运行端到端测试生成测试项目
7. **用户手动启动 e2e 生成的测试项目** - 验证生成的项目
8. **阅读测试项目生成的代码** - 检查输出质量
9. **从测试项目提取 SDD 流程的缺陷** - 识别问题和改进点
10. **纳入当前 feature 修复或者加入未来 feature** - 分类处理缺陷

## 注意事项

- 按用户要求执行流程，不可错乱顺序（如：不能直接修改测试项目）
- e2e 测试是验证 SDD 流程正确性的关键步骤
- 缺陷应及时记录并分类处理

## E2E 脚本说明

有两个端到端测试脚本可供选择：

- `sdd-e2e.sh` - 多模式测试脚本（支持 SDD/SDDU 模式切换和对比测试）
- `sddu-e2e.sh` - SDDU 专用测试脚本（纯 SDDU 模式，更专注）

注意：`@sdd-*` 系列旧命令虽仍可正常使用，但官方已推荐使用 `@sddu-*` 新命令。
