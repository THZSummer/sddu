# ADR-016: Agent 命令别名机制 - @sdd-* 和 @sddu-* 共存实现

**Feature**: FR-SDDU-RENAME-001 插件改名 SDDU  
**状态**: PROPOSED  
**创建日期**: 2026-04-06  
**决策者**: SDD Plan Agent  

---

## 1. 背景

本次改名涉及 18 个 Agent 的名称变更。需要设计一种机制，使新旧 Agent 名称可以同时存在并正常工作。

### Agent 改名清单

| 旧 Agent | 新 Agent | 类型 |
|---------|---------|------|
| `sdd` | `sddu` | Master Coordinator |
| `sdd-help` | `sddu-help` | Help Assistant |
| `sdd-0-discovery` | `sddu-0-discovery` | 需求挖掘 (编号) |
| `sdd-discovery` | `sddu-discovery` | 需求挖掘 (短名) |
| `sdd-1-spec` | `sddu-1-spec` | 规范编写 (编号) |
| `sdd-spec` | `sddu-spec` | 规范编写 (短名) |
| `sdd-2-plan` | `sddu-2-plan` | 技术规划 (编号) |
| `sdd-plan` | `sddu-plan` | 技术规划 (短名) |
| `sdd-3-tasks` | `sddu-3-tasks` | 任务分解 (编号) |
| `sdd-tasks` | `sddu-tasks` | 任务分解 (短名) |
| `sdd-4-build` | `sddu-4-build` | 任务实现 (编号) |
| `sdd-build` | `sddu-build` | 任务实现 (短名) |
| `sdd-5-review` | `sddu-5-review` | 代码审查 (编号) |
| `sdd-review` | `sddu-review` | 代码审查 (短名) |
| `sdd-6-validate` | `sddu-6-validate` | 验证 (编号) |
| `sdd-validate` | `sddu-validate` | 验证 (短名) |
| `sdd-docs` | `sddu-docs` | 目录导航生成器 |
| `sdd-roadmap` | `sddu-roadmap` | Roadmap 规划专家 |

---

## 2. 方案对比

### 方案 A: 双定义（推荐）

在 `opencode.json` 中为每个 Agent 创建两份定义，指向同一 prompt 文件。

**配置示例**:
```json
{
  "agent": {
    "sdd": {
      "description": "[Deprecated] Use @sddu instead. SDDU Master Coordinator.",
      "prompt": "{file:.opencode/agents/sddu.md}"
    },
    "sddu": {
      "description": "SDDU Master Coordinator. 规格驱动开发统一框架主协调器。",
      "prompt": "{file:.opencode/agents/sddu.md}"
    },
    "sdd-spec": {
      "description": "[Deprecated] Use @sddu-spec instead.",
      "prompt": "{file:.opencode/agents/sddu-spec.md}"
    },
    "sddu-spec": {
      "description": "SDDU Spec Agent. 规范编写专家。",
      "prompt": "{file:.opencode/agents/sddu-spec.md}"
    }
  }
}
```

**优点**:
- 实现简单，只需修改配置文件
- 新旧名称完全等价
- 无需修改代码逻辑
- OpenCode 原生支持

**缺点**:
- 配置文件膨胀（约 36 个 agent 定义）
- 需要手动维护双份定义的一致性

### 方案 B: 代码层别名

在代码中实现别名映射，只在 `opencode.json` 中定义新名称。

**优点**:
- 配置文件简洁
- 别名逻辑集中管理

**缺点**:
- 需要修改 OpenCode 插件框架（不可行）
- 增加代码复杂度
- OpenCode 不支持运行时别名

### 方案 C: Prompt 层重定向

旧 Agent 的 prompt 文件中添加重定向指令。

**优点**:
- 配置文件只需定义新 Agent

**缺点**:
- 用户体验差（调用旧名称会提示使用新名称）
- 需要额外调用步骤
- 不符合"平滑迁移"目标

---

## 3. 决策

**我们决定**: 采用 **方案 A - 双定义**

### 具体实施

1. **配置文件结构**:
   - 在 `/opencode.json` 中定义所有 36 个 Agent（18 旧 + 18 新）
   - 在 `/.opencode/plugins/sdd/opencode.json` 中同样定义
   - 旧 Agent 描述以 `[Deprecated]` 开头

2. **Prompt 文件**:
   - 只保留新名称的 prompt 文件（`sddu*.hbs`）
   - 旧 Agent 指向新 prompt 文件
   - 不保留旧的 prompt 文件副本

3. **描述规范**:
   - 旧 Agent: `[Deprecated] Use @sddu-xxx instead. <原描述>`
   - 新 Agent: `<完整描述>`

4. **Agent 分组**:
   ```
   主协调器:
   - sdd → sddu

   工作流 Agent (6 阶段):
   - sdd-0-discovery / sdd-discovery → sddu-0-discovery / sddu-discovery
   - sdd-1-spec / sdd-spec → sddu-1-spec / sddu-spec
   - sdd-2-plan / sdd-plan → sddu-2-plan / sddu-plan
   - sdd-3-tasks / sdd-tasks → sddu-3-tasks / sddu-tasks
   - sdd-4-build / sdd-build → sddu-4-build / sddu-build
   - sdd-5-review / sdd-review → sddu-5-review / sddu-review
   - sdd-6-validate / sdd-validate → sddu-6-validate / sddu-validate

   工具 Agent:
   - sdd-docs → sddu-docs
   - sdd-roadmap → sddu-roadmap

   帮助 Agent:
   - sdd-help → sddu-help
   ```

---

## 4. 后果

### 正面影响

✅ **平滑迁移**: 用户使用旧名称无任何感知  
✅ **灵活选择**: 用户可在任意时刻切换到新名称  
✅ **实现简单**: 只需修改配置文件，无需代码变更  
✅ **OpenCode 兼容**: 使用平台原生功能

### 负面影响

❌ **配置文件膨胀**: agent 定义数量翻倍（18→36）  
❌ **维护负担**: 需要确保新旧定义指向正确文件  
❌ **潜在混淆**: 用户可能不确定使用哪个名称

### 缓解措施

1. **配置注释**: 在 `opencode.json` 中添加注释说明双定义原因
2. **帮助文档**: 在 `@sddu-help` 中提供完整的改名对照表
3. **代码生成**: 使用脚本生成双份定义，确保一致性
4. **定期审查**: 在 v1.y 版本时审查使用情况，计划 v2.0 清理
