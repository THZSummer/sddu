# ADR-019: 迁移脚本设计 - 自动化迁移工具

**Feature**: FR-SDDU-RENAME-001 插件改名 SDDU  
**状态**: PROPOSED  
**创建日期**: 2026-04-06  
**决策者**: SDD Plan Agent  

---

## 1. 背景

为帮助用户从旧版本平滑迁移到新版本，需要提供自动化迁移工具。迁移涉及：

1. 插件包名变更：`opencode-sdd-plugin` → `opencode-sddu-plugin`
2. 配置文件更新：`opencode.json` 中的 agent 定义
3. 工作空间目录（可选）：`.sdd/` → `.sddu/`
4. 自定义脚本更新：用户项目中的 `sdd-*` 引用

### 迁移场景

| 场景 | 复杂度 | 用户群体 |
|------|--------|---------|
| 仅更新插件包 | 低 | 所有用户 |
| 更新配置文件 | 中 | 自定义配置的用户 |
| 迁移工作空间 | 中 | 有 `.sdd/` 目录的用户 |
| 更新自定义脚本 | 高 | 有自定义集成的用户 |

---

## 2. 方案对比

### 方案 A: 单一迁移脚本（推荐）

提供一个综合迁移脚本，处理所有迁移场景。

**脚本结构**:
```bash
#!/bin/bash
# migrate-sdd-to-sddu.sh

# 步骤 1: 备份
backup_existing_config()

# 步骤 2: 更新插件包
update_plugin_package()

# 步骤 3: 更新配置文件
update_opencode_json()

# 步骤 4: 迁移工作空间（可选）
migrate_workspace()

# 步骤 5: 验证
verify_migration()
```

**优点**:
- 一站式解决方案
- 用户体验好
- 可控制迁移顺序
- 支持回滚

**缺点**:
- 脚本复杂度高
- 需要处理多种场景
- 测试工作量大

### 方案 B: 多个专用脚本

为每个迁移场景提供独立脚本。

**脚本列表**:
- `migrate-plugin.sh` - 更新插件包
- `migrate-config.sh` - 更新配置文件
- `migrate-workspace.sh` - 迁移工作空间

**优点**:
- 每个脚本职责单一
- 用户可选择需要的步骤
- 易于测试和维护

**缺点**:
- 用户需要运行多个脚本
- 步骤间依赖需要文档说明
- 体验分散

### 方案 C: 交互式迁移工具

提供交互式 CLI 工具，引导用户完成迁移。

**优点**:
- 用户友好
- 可跳过不需要的步骤
- 实时反馈

**缺点**:
- 开发成本高
- 需要额外的依赖
- 可能引入新问题

---

## 3. 决策

**我们决定**: 采用 **方案 A - 单一迁移脚本**

### 具体实施

#### 3.1 脚本位置

```
/tools/
├── migrate-sdd-to-sddu.sh    # Linux/Mac
└── migrate-sdd-to-sddu.ps1   # Windows
```

#### 3.2 脚本功能

```bash
#!/bin/bash
# migrate-sdd-to-sddu.sh
# SDD to SDDU Migration Script
# Version: 1.0.0

set -e

# 配置
WORKSPACE_ROOT="${1:-.}"
BACKUP_DIR="$WORKSPACE_ROOT/.sddu-migration-backup-$(date +%Y%m%d-%H%M%S)"
DRY_RUN=false
VERBOSE=false

# 帮助信息
show_help() {
  cat << EOF
SDD to SDDU Migration Script

Usage: ./migrate-sdd-to-sddu.sh [OPTIONS] [WORKSPACE_ROOT]

Options:
  -h, --help        Show this help message
  -d, --dry-run     Show what would be done without making changes
  -v, --verbose     Verbose output
  --skip-backup     Skip backup step (not recommended)
  --skip-workspace   Skip workspace directory migration

Arguments:
  WORKSPACE_ROOT    Root directory of your project (default: current directory)

Examples:
  ./migrate-sdd-to-sddu.sh                    # Migrate current directory
  ./migrate-sdd-to-sddu.sh /path/to/project   # Migrate specific directory
  ./migrate-sdd-to-sddu.sh --dry-run          # Preview changes
EOF
}

# 步骤 1: 备份现有配置
backup_existing_config() {
  echo "📦 Step 1: Backing up existing configuration..."
  mkdir -p "$BACKUP_DIR"
  cp -r "$WORKSPACE_ROOT/.sdd" "$BACKUP_DIR/" 2>/dev/null || true
  cp "$WORKSPACE_ROOT/opencode.json" "$BACKUP_DIR/" 2>/dev/null || true
  cp "$WORKSPACE_ROOT/package.json" "$BACKUP_DIR/" 2>/dev/null || true
  echo "✅ Backup saved to: $BACKUP_DIR"
}

# 步骤 2: 更新插件包
update_plugin_package() {
  echo "📦 Step 2: Updating plugin package..."
  if [ -f "$WORKSPACE_ROOT/package.json" ]; then
    # 使用 npm 或手动修改
    echo "   Please run: npm uninstall opencode-sdd-plugin && npm install opencode-sddu-plugin"
  fi
}

# 步骤 3: 更新配置文件
update_opencode_json() {
  echo "📝 Step 3: Updating opencode.json..."
  # 使用 sed 或 node 脚本更新
  # 保留旧 agent 定义，添加新定义
}

# 步骤 4: 迁移工作空间（可选）
migrate_workspace() {
  echo "📁 Step 4: Migrating workspace directory..."
  if [ -d "$WORKSPACE_ROOT/.sdd" ] && [ ! -d "$WORKSPACE_ROOT/.sddu" ]; then
    mv "$WORKSPACE_ROOT/.sdd" "$WORKSPACE_ROOT/.sddu"
    echo "✅ .sdd/ renamed to .sddu/"
  elif [ -d "$WORKSPACE_ROOT/.sdd" ] && [ -d "$WORKSPACE_ROOT/.sddu" ]; then
    echo "⚠️  Both .sdd/ and .sddu/ exist, skipping..."
  else
    echo "ℹ️  No .sdd/ directory found, skipping..."
  fi
}

# 步骤 5: 验证
verify_migration() {
  echo "🔍 Step 5: Verifying migration..."
  # 检查关键文件是否存在
  # 检查配置语法
}

# 回滚功能
rollback() {
  echo "🔄 Rolling back migration..."
  if [ -d "$BACKUP_DIR" ]; then
    cp -r "$BACKUP_DIR/.sdd" "$WORKSPACE_ROOT/" 2>/dev/null || true
    cp "$BACKUP_DIR/opencode.json" "$WORKSPACE_ROOT/" 2>/dev/null || true
    echo "✅ Rollback complete"
  else
    echo "❌ No backup found, cannot rollback"
  fi
}

# 主流程
main() {
  echo "🚀 Starting SDD to SDDU migration..."
  backup_existing_config
  update_plugin_package
  update_opencode_json
  migrate_workspace
  verify_migration
  echo "✅ Migration complete!"
}

# 参数解析
while [[ $# -gt 0 ]]; do
  case $1 in
    -h|--help) show_help; exit 0 ;;
    -d|--dry-run) DRY_RUN=true; shift ;;
    -v|--verbose) VERBOSE=true; shift ;;
    --skip-backup) SKIP_BACKUP=true; shift ;;
    --skip-workspace) SKIP_WORKSPACE=true; shift ;;
    *) WORKSPACE_ROOT="$1"; shift ;;
  esac
done

main
```

#### 3.3 迁移检查清单

脚本执行前后验证：

**迁移前检查**:
- [ ] 备份目录创建成功
- [ ] 原配置文件存在

**迁移后验证**:
- [ ] `package.json` 中包名已更新
- [ ] `opencode.json` 语法正确
- [ ] 新 Agent `@sddu-*` 可调用
- [ ] 旧 Agent `@sdd-*` 仍可调用（向后兼容）
- [ ] 工作流可正常运行

#### 3.4 回滚机制

如果迁移失败，提供回滚功能：

```bash
# 回滚命令
./migrate-sdd-to-sddu.sh --rollback

# 或手动回滚
rm -rf .sddu
mv .sddu-migration-backup-*/.sdd ./
cp .sddu-migration-backup-*/opencode.json ./
```

## 4. 后果

### 正面影响

✅ **用户友好**: 一键迁移，降低采用门槛  
✅ **安全性**: 自动备份，支持回滚  
✅ **一致性**: 确保所有用户迁移步骤一致  
✅ **可验证**: 脚本包含验证步骤

### 负面影响

❌ **开发成本**: 需要编写和测试迁移脚本  
❌ **维护负担**: 迁移脚本需要随版本更新  
❌ **边缘情况**: 可能无法覆盖所有用户场景

### 缓解措施

1. **充分测试**: 在多种场景下测试迁移脚本
2. **文档说明**: 提供详细的迁移指南和故障排除
3. **手动备选**: 提供手动迁移步骤作为备选
4. **反馈渠道**: 建立用户反馈渠道收集迁移问题
