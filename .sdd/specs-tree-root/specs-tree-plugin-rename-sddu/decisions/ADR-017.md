# ADR-017: 目录改名策略 - .sdd/ 是否强制改名

**Feature**: FR-SDDU-RENAME-001 插件改名 SDDU  
**状态**: PROPOSED  
**创建日期**: 2026-04-06  
**决策者**: SDD Plan Agent  

---

## 1. 背景

当前工作空间目录为 `.sdd/`，包含所有 specs-tree、状态文件和文档。改名时需决策：

1. 是否强制将 `.sdd/` 改名为 `.sddu/`？
2. 如何处理现有用户的工作空间？
3. 新 Feature 应该使用哪个目录？

### 现状分析

- `.sdd/` 已成为现有用户的标准工作空间
- 包含大量历史 Feature 和状态文件
- 强制改名会导致：
  - 现有工作空间"断裂"
  - 状态文件路径失效
  - 用户需要手动迁移

---

## 2. 方案对比

### 方案 A: 不强制改名（推荐）

保持 `.sdd/` 目录支持，同时支持新建 `.sddu/` 目录。

**实现方式**:
```typescript
// 代码中同时支持两种目录名
const WORKSPACE_DIRS = ['.sdd', '.sddu'];

function findWorkspaceDir(basePath: string): string | null {
  for (const dir of WORKSPACE_DIRS) {
    if (existsSync(path.join(basePath, dir))) {
      return path.join(basePath, dir);
    }
  }
  return null;
}
```

**优点**:
- 现有用户完全不受影响
- 无迁移成本
- 用户可选择是否使用新目录名
- 品牌名 SDDU 与目录名 .sdd 可共存

**缺点**:
- 代码需要支持双目录
- 文档需要说明两者等价
- 可能造成新用户的困惑

### 方案 B: 强制改名

要求所有用户将 `.sdd/` 改名为 `.sddu/`。

**优点**:
- 命名统一
- 代码逻辑简化
- 品牌形象一致

**缺点**:
- 现有用户需要手动迁移
- 迁移过程可能出错
- 用户抵触情绪高
- 历史 Git 记录中的路径失效

### 方案 C: 提供迁移脚本

提供自动化工具将 `.sdd/` 迁移到 `.sddu/`，但不强制。

**优点**:
- 用户有选择权
- 降低迁移成本
- 可记录迁移状态

**缺点**:
- 需要开发迁移工具
- 迁移脚本需要充分测试
- 仍需代码支持双目录（迁移前）

---

## 3. 决策

**我们决定**: 采用 **方案 A + C - 不强制改名 + 提供迁移脚本**

### 具体实施

1. **代码层面**:
   - 在路径解析函数中同时支持 `.sdd` 和 `.sddu`
   - 优先使用已存在的目录
   - 都不存在时，创建 `.sddu`（新默认）

2. **目录优先级**:
   ```
   1. 如果 .sddu/ 存在 → 使用 .sddu/
   2. 如果 .sdd/ 存在 → 使用 .sdd/
   3. 都不存在 → 创建 .sddu/
   ```

3. **迁移脚本** (P1 优先级):
   - 创建 `.tool/migrate-workspace.js` 脚本
   - 功能：将 `.sdd/` 重命名为 `.sddu/`
   - 包含完整性检查
   - 支持回滚

4. **文档说明**:
   - 明确 `.sdd/` 和 `.sddu/` 等价
   - 推荐新项目使用 `.sddu/`
   - 现有项目可继续使用 `.sdd/`

5. **specs-tree 目录命名**:
   - 现有 Feature: `specs-tree-sdd-*` 保持不变
   - 新 Feature: 使用 `specs-tree-sddu-*` 前缀

---

## 4. 后果

### 正面影响

✅ **用户友好**: 现有用户无需任何操作  
✅ **灵活性**: 用户可按需选择目录名  
✅ **渐进迁移**: 通过脚本支持自愿迁移  
✅ **历史兼容**: 现有 Feature 目录保持不变

### 负面影响

❌ **命名不一致**: 品牌 SDDU 与目录 .sdd 不匹配  
❌ **代码复杂度**: 需要支持双路径  
❌ **文档维护**: 需要说明两者关系  
❌ **长期债务**: 双路径支持需要长期维护

### 缓解措施

1. **代码封装**: 将路径解析封装为单一函数，减少重复逻辑
2. **文档清晰**: 在 `.sdd/README.md` 中明确说明命名策略
3. **测试覆盖**: 为双路径支持编写完整测试
4. **清理计划**: 在 v2.0 时评估是否移除 `.sdd/` 支持
