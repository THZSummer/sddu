# ADR-015: 向后兼容策略 - 双名称并存

**Feature**: FR-SDDU-RENAME-001 插件改名 SDDU  
**状态**: PROPOSED  
**创建日期**: 2026-04-06  
**决策者**: SDD Plan Agent  

---

## 1. 背景

插件改名涉及从 `@sdd-*` 到 `@sddu-*` 的 Agent 名称变更。现有用户已依赖旧名称建立工作流，直接切换会导致：

1. 现有用户工作流立即断裂
2. 自动化脚本失效
3. 用户抵触情绪高
4. 紧急回滚风险

需要制定向后兼容策略，确保平滑迁移。

---

## 2. 方案对比

### 方案 A: 双名称并存（推荐）

在 `opencode.json` 中同时保留新旧 Agent 定义：

```json
{
  "agent": {
    "sdd": {
      "description": "[Deprecated] Use @sddu instead",
      "prompt": "{file:.opencode/agents/sddu.md}"
    },
    "sddu": {
      "description": "SDDU Master Coordinator",
      "prompt": "{file:.opencode/agents/sddu.md}"
    }
  }
}
```

**优点**:
- 现有用户工作流不受影响
- 迁移成本低，用户可逐步适应
- 技术实现简单
- 可观察新旧名称使用情况

**缺点**:
- 配置文件体积增加约 2 倍
- 需要维护双份定义
- 可能造成用户困惑

### 方案 B: 硬切换

直接替换所有 `@sdd-*` 为 `@sddu-*`。

**优点**:
- 配置简洁
- 品牌统一快速

**缺点**:
- 用户工作流断裂
- 高抵触风险
- 可能丢失用户

### 方案 C: 渐进式切换

先发布过渡版本添加新名称，后续版本逐步废弃。

**优点**:
- 用户有充足适应时间
- 可收集使用数据

**缺点**:
- 发布周期长
- 版本管理复杂

---

## 3. 决策

**我们决定**: 采用 **方案 A - 双名称并存**

### 具体实施

1. **配置文件**: 在 `opencode.json` 中定义双份 Agent
   - 旧 Agent (`sdd`, `sdd-help` 等) 标记为 deprecated
   - 新旧 Agent 指向同一 prompt 文件
   - 描述中明确说明推荐使用新名称

2. **工具函数**: 保留 `sdd_update_state.js`，同时创建 `sddu_update_state.js`
   - 两者功能完全相同
   - 旧工具不输出警告（避免干扰）

3. **目录支持**: 代码同时支持 `.sdd/` 和 `.sddu/` 目录
   - 不强制用户迁移工作空间
   - 新 Feature 可使用任一路径

4. **文档策略**:
   - 新文档默认使用 `@sddu-*`
   - 旧文档保留但添加更新提示
   - 发布改名对照表

5. **废弃时间表**:
   - v1.x (当前): 双名称并存，无警告
   - v1.y (下一版本): 旧名称有 console 警告
   - v2.0 (大版本): 移除旧名称支持

---

## 4. 后果

### 正面影响

✅ **用户体验**: 现有用户无感知迁移，可按需切换到新名称  
✅ **采用率**: 降低新名称采用门槛  
✅ **风险控制**: 如发现问题可快速回退  
✅ **数据收集**: 可通过日志分析新旧名称使用比例

### 负面影响

❌ **维护成本**: 需要维护双份定义（约 2 倍配置）  
❌ **用户困惑**: 用户可能不确定使用哪个名称  
❌ **技术债务**: 需要在 v2.0 时清理旧定义

### 缓解措施

1. **清晰文档**: 发布改名对照表，明确推荐新名称
2. **Help Agent**: 在 `@sddu-help` 中说明新旧映射关系
3. **日志记录**: 记录使用的 Agent 名称，用于使用分析
4. **废弃警告**: 在 v1.y 版本开始，旧名称输出 deprecation 警告

---

- [ ] 配置文件语法正确
- [ ] 文档中包含新旧名称映射说明
