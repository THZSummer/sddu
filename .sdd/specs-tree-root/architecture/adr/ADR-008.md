## 1. 背景

### 1.1 问题陈述

当前 state.json 中已存在 `history` 字段，但存在以下问题：

1. **格式不统一**: 部分 Feature 的 history 记录不完整
2. **缺少阶段历史**: 只有状态变更历史，没有阶段（phase）级别的记录
3. **自动记录不完整**: 部分状态变更未自动追加历史记录
4. **查询困难**: 无法快速查询特定阶段的历史

### 1.2 决策范围

本 ADR 决定：
- history 字段的统一格式
- 是否新增 phaseHistory 字段
- 历史记录的自动追加机制
- 历史记录的保留策略

### 1.3 现状分析

**现有 history 格式** (来自 state.json):
```json
{
  "history": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "from": "specified",
      "to": "planned",
      "triggeredBy": "@sdd-plan",
      "comment": "技术规划完成"
    }
  ]
}
```

**缺失字段**:
- `actor`: 执行操作的 Agent 名称
- `phase`: 阶段编号
- `phaseName`: 阶段名称

---

## 2. 方案对比

### 2.1 方案 A：扩展现有 history（推荐）⭐

**描述**：
保持现有 `history` 数组结构，增加必要字段，不新增 phaseHistory。

**格式**:
```json
{
  "history": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "from": "specified",
      "to": "planned",
      "fromPhase": 1,
      "toPhase": 2,
      "triggeredBy": "@sdd-plan",
      "actor": "sdd-plan-agent",
      "comment": "技术规划完成"
    }
  ]
}
```

**优点**:
- ✅ 简单：单一数据源，易于维护
- ✅ 向后兼容：在现有结构上扩展
- ✅ 查询灵活：可以过滤 phase 或 status

**缺点**:
- ❌ 冗余：phase 信息可以从 status 推断
- ❌ 统计不便：需要遍历计算阶段历史

**实现复杂度**: 低

---

### 2.2 方案 B：新增 phaseHistory（推荐）⭐

**描述**：
保留现有 `history`（状态变更），新增 `phaseHistory`（阶段变更）。

**格式**:
```json
{
  "history": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "from": "specified",
      "to": "planned",
      "triggeredBy": "@sdd-plan",
      "actor": "sdd-plan-agent",
      "comment": "技术规划完成"
    }
  ],
  "phaseHistory": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "phase": 2,
      "phaseName": "planned",
      "triggeredBy": "@sdd-plan"
    }
  ]
}
```

**优点**:
- ✅ 职责分离：history 记录状态，phaseHistory 记录阶段
- ✅ 统计方便：phaseHistory 可直接用于进度统计
- ✅ 粒度不同：满足不同查询需求

**缺点**:
- ❌ 数据冗余：phase 信息在两个地方存储
- ❌ 同步问题：需要确保两个数组一致

**实现复杂度**: 中

---

### 2.3 方案 C：统一为 timeline

**描述**：
使用单一的 `timeline` 数组，记录所有事件（状态变更、阶段变更、注释等）。

**格式**:
```json
{
  "timeline": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "type": "state_change",
      "data": {
        "from": "specified",
        "to": "planned"
      }
    },
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "type": "phase_change",
      "data": {
        "phase": 2,
        "phaseName": "planned"
      }
    },
    {
      "timestamp": "2026-04-05T10:05:00Z",
      "type": "comment",
      "data": {
        "text": "添加依赖关系"
      }
    }
  ]
}
```

**优点**:
- ✅ 灵活性高：可以记录任意类型事件
- ✅ 时间线清晰：按时间排序查看所有事件
- ✅ 可扩展：未来可以添加更多事件类型

**缺点**:
- ❌ 查询复杂：需要过滤不同类型事件
- ❌ 破坏兼容：需要迁移现有 history
- ❌ 实现复杂：需要新的事件系统

**实现复杂度**: 高

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 B（新增 phaseHistory）**

### 3.2 决策理由

1. **职责分离**: history 和 phaseHistory 各司其职
2. **向后兼容**: 保持现有 history 格式不变
3. **统计便利**: phaseHistory 可直接用于进度报告
4. **规范符合**: 符合 spec v2.0.0 对 FR-003 的要求

### 3.3 Schema 定义

```typescript
// src/state/schema-v2.0.0.ts

// 历史记录条目
interface HistoryEntry {
  timestamp: string;        // ISO 8601
  from: string;             // 原状态
  to: string;               // 新状态
  triggeredBy: string;      // 触发源（@sdd-plan 等）
  actor?: string;           // 执行者（Agent 名称）
  comment?: string;         // 可选注释
}

// 阶段历史条目
interface PhaseHistoryEntry {
  timestamp: string;        // ISO 8601
  phase: number;            // 阶段编号 (1-6)
  phaseName: string;        // 阶段名称
  triggeredBy: string;      // 触发源
  actor?: string;           // 执行者
}

// State Schema v2.0.0
interface StateV2_0_0 {
  feature: string;
  name: string;
  version: '2.0.0';
  status: FeatureStatus;
  phase: number;
  // ... 其他字段
  history: HistoryEntry[];
  phaseHistory: PhaseHistoryEntry[];
  createdAt: string;
  updatedAt: string;
}
```

### 3.4 自动记录机制

```typescript
// src/state/state-manager.ts
class MultiFeatureStateManager {
  async updateState(featureId: string, update: Partial<StateV2_0_0>) {
    const state = await this.loadState(featureId);
    const oldState = { ...state };
    
    // 合并更新
    Object.assign(state, update);
    
    // 检测状态变更
    if (update.status && update.status !== oldState.status) {
      // 追加历史记录
      state.history.push({
        timestamp: new Date().toISOString(),
        from: oldState.status,
        to: update.status,
        triggeredBy: update.triggeredBy || 'manual',
        actor: update.actor,
        comment: update.comment
      });
    }
    
    // 检测阶段变更
    if (update.phase && update.phase !== oldState.phase) {
      // 追加阶段历史
      state.phaseHistory.push({
        timestamp: new Date().toISOString(),
        phase: update.phase,
        phaseName: phaseToName(update.phase),
        triggeredBy: update.triggeredBy || 'manual',
        actor: update.actor
      });
    }
    
    state.updatedAt = new Date().toISOString();
    await this.saveState(featureId, state);
  }
}

function phaseToName(phase: number): string {
  const names: Record<number, string> = {
    1: 'specified',
    2: 'planned',
    3: 'tasked',
    4: 'implementing',
    5: 'reviewed',
    6: 'validated'
  };
  return names[phase] || 'unknown';
}
```

### 3.5 历史记录保留策略

```typescript
interface HistoryRetentionConfig {
  // 最大记录数
  maxEntries: number;        // 默认 1000
  
  // 保留时长 (天)
  retentionDays: number;     // 默认 365
  
  // 是否归档旧记录
  enableArchival: boolean;   // 默认 true
  
  // 归档目录
  archiveDir: string;        // 默认 .sdd/history-archive/
}

// 清理逻辑
async function pruneHistory(state: StateV2_0_0, config: HistoryRetentionConfig) {
  // 限制记录数
  if (state.history.length > config.maxEntries) {
    // 归档旧记录
    if (config.enableArchival) {
      await archiveHistory(state.feature, state.history.slice(0, -config.maxEntries));
    }
    state.history = state.history.slice(-config.maxEntries);
  }
  
  // 清理过期记录
  const cutoff = Date.now() - config.retentionDays * 24 * 60 * 60 * 1000;
  state.history = state.history.filter(h => 
    new Date(h.timestamp).getTime() > cutoff
  );
}
```

### 3.6 查询接口

```typescript
// src/state/query.ts
class StateQuery {
  // 获取完整历史
  async getHistory(featureId: string): Promise<HistoryEntry[]> {
    const state = await this.loadState(featureId);
    return state.history;
  }
  
  // 获取阶段历史
  async getPhaseHistory(featureId: string): Promise<PhaseHistoryEntry[]> {
    const state = await this.loadState(featureId);
    return state.phaseHistory;
  }
  
  // 获取当前阶段
  async getCurrentPhase(featureId: string): Promise<number> {
    const state = await this.loadState(featureId);
    return state.phase;
  }
  
  // 查询在某个阶段的时长
  async getPhaseDuration(featureId: string, phase: number): Promise<number> {
    const history = await this.getPhaseHistory(featureId);
    const phaseEntries = history.filter(h => h.phase === phase);
    
    if (phaseEntries.length < 2) return 0;
    
    const entryTime = new Date(phaseEntries[0].timestamp).getTime();
    const nextEntryTime = new Date(phaseEntries[1].timestamp).getTime();
    
    return nextEntryTime - entryTime;
  }
}
```

## 4. 后果

### 4.1 积极影响

- ✅ 历史记录完整：自动记录所有变更
- ✅ 阶段进度清晰：phaseHistory 便于统计
- ✅ 审计支持：可以追溯所有状态变更
- ✅ 向后兼容：现有 history 格式不变

### 4.2 消极影响

- ⚠️ 存储增加：新增 phaseHistory 数组
- ⚠️ 数据冗余：phase 信息在两个地方存储
- ⚠️ 同步风险：需要确保 history 和 phaseHistory 一致

### 4.3 需要遵循的约定

1. **自动追加**: 所有状态/阶段变更自动追加历史记录
2. **只追加**: 历史记录不可修改（只追加模式）
3. **时间戳**: 使用 ISO 8601 格式
4. **触发源**: 必须记录 triggeredBy 字段
5. **归档策略**: 超过 1000 条或 365 天的记录归档
