## 1. 背景

### 1.1 问题陈述

当前 SDD 工作流插件的状态管理采用**集中式 state.json 设计**，存在以下问题：

1. **单点故障风险**：所有 Feature 状态集中在单一文件中，文件损坏会影响全部 Feature
2. **并发冲突概率高**：多 Agent 同时更新不同 Feature 时需锁定同一文件
3. **目录结构不自然**：状态文件与 Feature 目录分离，不符合"高内聚"设计原则
4. **删除 Feature 时状态残留**：删除目录后，集中式 state.json 中对应条目容易遗漏
5. **缺少状态历史**：无法追溯 Feature 的状态变更历史

### 1.2 决策范围

本 ADR 决定 SDD 工作流状态管理的存储架构：
- 状态文件存储位置（集中式 vs 分布式）
- 状态文件与 Feature 目录的关系
- 聚合查询机制

---

## 2. 方案对比

### 2.1 方案 A：完全分布式存储（推荐）⭐

**描述**：
- 每个 specs 目录拥有独立 `state.json`
- 无集中式状态存储
- 聚合查询通过递归扫描实现

**架构图**：
```
.sdd/specs-tree-root/
├── specs-tree-feature-a/
│   ├── spec.md
│   ├── plan.md
│   ├── tasks.md
│   └── state.json                # Feature A 独立状态
└── specs-tree-feature-b/
    ├── spec.md
    ├── plan.md
    ├── tasks.md
    └── state.json                # Feature B 独立状态
```

**优点**：
- ✅ 无单点故障：单个 state.json 损坏不影响其他 specs
- ✅ 低并发冲突：多 Agent 更新不同 specs 时无锁竞争
- ✅ 高内聚：状态与文档同目录，移动/复制 Feature 时自动携带状态
- ✅ 删除安全：删除 specs 目录时状态自动清除，无残留
- ✅ 扩展性好：支持无限层级嵌套

**缺点**：
- ❌ 聚合查询性能：需要扫描多个文件（100 个 specs 约 50-100ms）
- ❌ 无全局视图：需要递归扫描才能获取全量状态
- ❌ 迁移成本：需要从集中式 state.json 迁移到分布式

### 2.2 方案 B：混合式存储

**描述**：
- 每个 specs 目录拥有独立 `state.json`（详情）
- 保留集中式 `state.json` 作为索引/缓存
- 聚合查询优先读索引，定期同步

**优点**：
- ✅ 聚合查询快：直接读索引文件
- ✅ 向后兼容：保留现有集中式读取逻辑

**缺点**：
- ❌ 数据同步复杂：需要维护两份数据一致性
- ❌ 并发冲突：索引文件仍有单点锁竞争
- ❌ 实现复杂度高：需要双写逻辑和同步机制

### 2.3 方案 C：纯集中式（现有架构）

**描述**：
- 保持现有集中式 `state.json`
- 添加文件锁和事务支持

**优点**：
- ✅ 实现简单：最小化改动
- ✅ 聚合查询快：单文件读取

**缺点**：
- ❌ 单点故障：文件损坏影响全部 specs
- ❌ 并发冲突：多 Agent 更新需锁竞争
- ❌ 低内聚：状态与文档分离

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 A（完全分布式存储）**

### 3.2 决策理由

1. **符合高内聚设计原则**：状态与文档同目录，符合模块化设计最佳实践
2. **消除单点故障**：单个 state.json 损坏不影响其他 specs，系统更健壮
3. **降低并发冲突**：多 Agent 并行更新不同 specs 时无锁竞争，提升并发性能
4. **支持多层嵌套**：天然支持无限层级嵌套结构，符合 TREE.md 规范
5. **删除安全**：删除 specs 目录时状态自动清除，避免脏数据累积
6. **迁移可行**：虽有迁移成本，但可通过工具自动化完成

### 3.3 性能优化措施

为弥补分布式存储的聚合查询性能劣势，采取以下优化：
- **递归深度限制**：最大 10 层，防止栈溢出
- **符号链接跳过**：避免无限循环
- **惰性缓存**：聚合结果缓存 1 分钟
- **并行扫描**：同一层级目录并行扫描

---

## 4. 后果

### 4.1 积极影响

- ✅ 系统可靠性提升：消除单点故障
- ✅ 并发性能提升：多 Agent 并行更新无冲突
- ✅ 可维护性提升：状态与文档高内聚
- ✅ 扩展性提升：支持多层嵌套结构

### 4.2 消极影响

- ⚠️ 迁移工作量：需要从集中式迁移到分布式（预计 1-2 天）
- ⚠️ 聚合查询性能：首次查询需要扫描多个文件（通过缓存缓解）
- ⚠️ 代码重构：状态管理模块需要重构（预计 3-5 天）

### 4.3 需要遵循的约定

1. **状态文件命名**：统一使用 `state.json`
2. **状态文件位置**：必须位于 specs 目录下，与 spec.md 同级
3. **Schema 版本**：使用 v1.3.0 或更高版本
4. **历史记录**：每次状态变更必须追加历史记录
5. **原子写入**：状态更新必须使用原子写入操作
