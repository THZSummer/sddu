## 1. 背景

### 1.1 问题陈述

规范 v2.0.0 要求 **FR-002: 系统必须支持 session.idle 事件触发状态更新**，但当前实现中状态更新完全依赖手动触发。

**现状问题**:
1. **手动操作繁琐**: 每个 Agent 完成后需要手动调用状态更新命令
2. **容易遗漏**: 开发者可能忘记更新状态
3. **状态不一致**: state.json 与实际文件状态可能不匹配
4. **缺少兜底机制**: 如果钩子触发失败，没有备用更新方式

### 1.2 决策范围

本 ADR 决定：
- session.idle 事件的监听和处理机制
- 文件变更检测策略
- 状态推断算法
- 更新触发条件（避免过度更新）

### 1.3 使用场景

| 场景 | 触发方式 | 说明 |
|------|----------|------|
| Agent 正常完成 | 钩子触发 | 主要路径，及时更新 |
| Agent 异常退出 | idle 触发 | 兜底路径，检测遗漏 |
| 手动编辑文件 | idle 触发 | 检测外部变更 |
| 批量导入 Feature | 手动触发 | 特殊场景 |

## 2. 方案对比

### 2.1 方案 A：文件监听器（推荐）⭐

**描述**：
使用 `fs.watch` 或 `chokidar` 监听 Feature 目录的文件变化，结合 session.idle 事件触发状态更新。

**架构**:
```
文件变化事件 ─┬─→ 防抖 (500ms) ─→ 状态推断 ─→ StateMachine 验证 ─→ 更新
session.idle ─┘
```

**优点**:
- ✅ 实时性好：文件变化后快速响应
- ✅ 准确性高：基于实际文件状态推断
- ✅ 兜底机制：idle 事件确保不会遗漏

**缺点**:
- ❌ 实现复杂：需要处理文件监听、防抖、去重
- ❌ 资源消耗：持续监听占用系统资源
- ❌ 误触发风险：临时文件编辑可能触发更新

**实现复杂度**: 中

---

### 2.2 方案 B：定期扫描

**描述**：
定期（如每 5 分钟）扫描所有 Feature 目录，检测文件状态与 state.json 是否一致。

**优点**:
- ✅ 实现简单：定时任务 + 扫描逻辑
- ✅ 资源消耗低：非持续监听
- ✅ 可控性好：扫描频率可配置

**缺点**:
- ❌ 延迟高：最坏情况下 5 分钟延迟
- ❌ 性能问题：大量 Feature 时扫描耗时
- ❌ 不够实时：不适合频繁变更场景

**实现复杂度**: 低

---

### 2.3 方案 C：混合模式（推荐优化）

**描述**：
方案 A + 方案 B 结合：
- 文件监听：实时检测变化
- session.idle: 触发实际更新（防抖 + 批量）
- 定期扫描：每小时兜底校验

**优点**:
- ✅ 实时性好：文件变化立即检测
- ✅ 更新频率可控：idle 事件批量处理
- ✅ 兜底机制完善：定期扫描确保一致性

**缺点**:
- ❌ 实现最复杂：需要协调三种机制
- ❌ 调试困难：多种触发路径

**实现复杂度**: 中高

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 C（混合模式）**

### 3.2 决策理由

1. **平衡实时性与性能**: 文件监听确保实时检测，idle 事件批量更新
2. **多层兜底**: 三种机制互为备份，确保状态一致性
3. **符合 VS Code 扩展模式**: session.idle 是 VS Code 标准事件
4. **可扩展**: 未来可以添加更多触发源

### 3.3 实现细节

#### 3.3.1 文件监听器

```typescript
// src/state/auto-updater.ts
import * as chokidar from 'chokidar';
import { EventEmitter } from 'events';

class FileChangeDetector extends EventEmitter {
  private watcher: chokidar.FSWatcher;
  private debounceTimers: Map<string, NodeJS.Timeout> = new Map();
  
  constructor(private specsDir: string) {
    super();
    this.watcher = chokidar.watch(specsDir, {
      ignored: /(^|[\/\\])\../, // 忽略隐藏文件
      persistent: true,
      ignoreInitial: true,
      awaitWriteFinish: {
        stabilityThreshold: 500, // 500ms 防抖
        pollInterval: 100
      }
    });
    
    this.watcher.on('change', (path) => this.onFileChange(path));
    this.watcher.on('add', (path) => this.onFileChange(path));
  }
  
  private onFileChange(filePath: string) {
    const featureId = this.extractFeatureId(filePath);
    if (!featureId) return;
    
    // 防抖：500ms 内只触发一次
    const timer = this.debounceTimers.get(featureId);
    if (timer) clearTimeout(timer);
    
    const newTimer = setTimeout(() => {
      this.emit('featureChanged', featureId, filePath);
      this.debounceTimers.delete(featureId);
    }, 500);
    
    this.debounceTimers.set(featureId, newTimer);
  }
  
  private extractFeatureId(filePath: string): string | null {
    const match = filePath.match(/specs-tree-([^\/]+)/);
    return match ? match[0] : null;
  }
}
```

#### 3.3.2 session.idle 处理

```typescript
// src/state/auto-updater.ts
class StateAutoUpdater {
  private pendingUpdates: Set<string> = new Set();
  private lastUpdateTime: Map<string, number> = new Map();
  
  constructor(
    private detector: FileChangeDetector,
    private stateMachine: StateMachine,
    private stateManager: MultiFeatureStateManager
  ) {
    // 监听文件变化
    this.detector.on('featureChanged', (featureId) => {
      this.pendingUpdates.add(featureId);
    });
    
    // 监听 session.idle
    this.registerIdleHandler();
  }
  
  private registerIdleHandler() {
    // VS Code session.idle 事件
    // 注意：实际实现需要根据 VS Code API 调整
    vscode.authentication.onDidChangeSessions(async () => {
      await this.processPendingUpdates();
    });
    
    // 备用：定时器模拟 idle
    setInterval(() => {
      this.processPendingUpdates();
    }, 60000); // 每 60 秒检查一次
  }
  
  private async processPendingUpdates() {
    const now = Date.now();
    
    for (const featureId of this.pendingUpdates) {
      const lastUpdate = this.lastUpdateTime.get(featureId) || 0;
      
      // 跳过 30 秒内更新过的 Feature
      if (now - lastUpdate < 30000) {
        continue;
      }
      
      try {
        await this.syncFeatureState(featureId);
        this.lastUpdateTime.set(featureId, now);
      } catch (error) {
        logger.warn(`状态同步失败 ${featureId}: ${error.message}`);
      }
    }
    
    this.pendingUpdates.clear();
  }
  
  private async syncFeatureState(featureId: string) {
    // 1. 从文件推断状态
    const inferredState = await this.inferStateFromFiles(featureId);
    
    // 2. 获取当前状态
    const currentState = await this.stateManager.getState(featureId);
    
    // 3. 状态不一致时更新
    if (inferredState && inferredState !== currentState.status) {
      // 4. 验证状态转换
      const validation = await this.stateMachine.validateStageTransition(
        featureId,
        inferredState
      );
      
      if (validation.allowed) {
        // 5. 执行更新
        await this.stateManager.updateState(featureId, {
          status: inferredState,
          phase: stateToPhase(inferredState),
          history: this.createHistoryEntry(inferredState, 'auto-sync')
        });
        
        logger.info(`状态自动更新：${featureId} ${currentState.status} → ${inferredState}`);
      } else {
        logger.warn(`状态转换验证失败 ${featureId}: ${validation.reason}`);
      }
    }
  }
  
  private async inferStateFromFiles(featureId: string): Promise<string | null> {
    const featureDir = path.join(this.specsDir, featureId);
    
    // 根据存在的文件推断状态（优先级从高到低）
    if (await exists(path.join(featureDir, 'validation.md'))) return 'validated';
    if (await exists(path.join(featureDir, 'review.md'))) return 'reviewed';
    if (await exists(path.join(featureDir, 'tasks.md'))) return 'tasked';
    if (await exists(path.join(featureDir, 'plan.md'))) return 'planned';
    if (await exists(path.join(featureDir, 'spec.md'))) return 'specified';
    
    return null; // 无法推断
  }
}
```

#### 3.3.3 配置选项

```typescript
interface AutoUpdateConfig {
  // 防抖时间 (ms)
  debounceDelay: number;        // 默认 500
  
  // 最小更新间隔 (ms)
  minUpdateInterval: number;    // 默认 30000
  
  // idle 检测间隔 (ms)
  idleCheckInterval: number;    // 默认 60000
  
  // 定期扫描间隔 (ms)，0 表示禁用
  scanInterval: number;         // 默认 3600000 (1 小时)
  
  // 最大递归深度
  maxDepth: number;             // 默认 10
  
  // 启用日志
  enableLogging: boolean;       // 默认 true
}
```

### 3.4 状态推断规则

| 存在文件 | 推断状态 | 说明 |
|----------|----------|------|
| validation.md | validated | 验证阶段 |
| review.md | reviewed | 审查阶段 |
| tasks.md | tasked | 任务分解阶段 |
| plan.md | planned | 规划阶段 |
| spec.md | specified | 规范阶段 |
| 无 | null | 无法推断 |

**注意**: 状态推断是单向的（前进），不会自动回退状态。

### 3.5 更新抑制策略

为避免过度更新：

```typescript
// 抑制条件
function shouldSuppressUpdate(featureId: string): boolean {
  // 1. 30 秒内更新过
  if (recentUpdate(featureId, 30000)) return true;
  
  // 2. Agent 正在执行
  if (agentExecuting(featureId)) return true;
  
  // 3. 状态未变化
  if (currentState === inferredState) return true;
  
  // 4. 推断状态不明确
  if (inferredState === null) return true;
  
  return false;
}
```

## 4. 后果

### 4.1 积极影响

- ✅ 状态自动同步：减少手动操作
- ✅ 状态准确性提升：基于实际文件状态
- ✅ 兜底机制：idle 和定期扫描确保一致性
- ✅ 用户体验改善：无需记住更新状态

### 4.2 消极影响

- ⚠️ 资源消耗：文件监听占用系统资源
- ⚠️ 误触发风险：临时编辑可能触发更新
- ⚠️ 实现复杂：需要处理多种触发机制

### 4.3 需要遵循的约定

1. **防抖时间**: 500ms 防抖，避免频繁触发
2. **更新间隔**: 30 秒内不重复更新
3. **单向推断**: 只推断状态前进，不回退
4. **日志记录**: 所有自动更新操作记录日志
