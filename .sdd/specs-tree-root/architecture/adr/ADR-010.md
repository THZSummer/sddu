## 1. 背景

### 1.1 问题陈述

当前项目中存在多个 State Schema 版本混用：

| Schema 版本 | 使用 Feature | 状态 |
|-------------|--------------|------|
| v1.2.5 | 旧 Feature | 已废弃 |
| v1.2.11 | sdd-multi-module | 已验证 |
| v2.0.0 | 新 Feature | 目标版本 |

**问题**:
1. **版本不统一**: 不同 Feature 使用不同 Schema，导致兼容性问题
2. **字段不一致**: 不同版本的字段定义不同
3. **迁移困难**: 没有自动化工具迁移旧版本到新版本
4. **读取复杂**: 代码需要处理多个版本的读取逻辑

### 1.2 决策范围

本 ADR 决定：
- Schema v2.0.0 的最终定义
- 迁移策略（一次性 vs 渐进式）
- 兼容性处理（多版本共存）
- 迁移工具实现

## 2. 方案对比

### 2.1 方案 A：一次性迁移（推荐）⭐

**描述**：
提供迁移工具，一次性将所有 Feature 的 state.json 迁移到 v2.0.0。

**流程**:
```
1. 扫描所有 Feature 目录
2. 检测 Schema 版本
3. 对非 v2.0.0 的版本执行迁移
4. 验证迁移结果
5. 备份旧版本（可选）
```

**优点**:
- ✅ 统一版本：迁移后所有 Feature 使用相同 Schema
- ✅ 简化代码：只需支持 v2.0.0
- ✅ 长期维护成本低

**缺点**:
- ❌ 风险高：一次性迁移可能影响大量 Feature
- ❌ 回滚困难：迁移后回滚复杂
- ❌ 需要停机：迁移期间可能影响使用

**实现复杂度**: 中

---

### 2.2 方案 B：渐进式迁移（推荐）⭐

**描述**：
支持多版本共存，逐个 Feature 迁移到 v2.0.0。

**策略**:
- 读取：支持 v1.2.5/v1.2.11/v2.0.0
- 写入：统一使用 v2.0.0
- 迁移：按需触发（首次访问时或手动触发）

**优点**:
- ✅ 风险低：逐个迁移，影响范围小
- ✅ 回滚容易：可以单独回滚某个 Feature
- ✅ 无需停机：迁移过程不影响使用

**缺点**:
- ❌ 代码复杂：需要支持多版本读取
- ❌ 迁移周期长：可能需要数周完成
- ❌ 临时状态：迁移期间版本混杂

**实现复杂度**: 中高

---

### 2.3 方案 C：混合模式（推荐）

**描述**：
方案 A + 方案 B 结合：
- 渐进式：默认按需迁移
- 批量工具：提供批量迁移命令
- 自动检测：首次访问旧版本时自动迁移

**架构**:
```typescript
class SchemaMigrator {
  // 自动迁移（首次访问）
  async loadState(featureId: string): Promise<StateV2_0_0> {
    const state = await this.readState(featureId);
    
    if (state.version !== '2.0.0') {
      // 自动迁移
      const migrated = await this.migrate(state, '2.0.0');
      await this.saveState(featureId, migrated);
      return migrated;
    }
    
    return state;
  }
  
  // 批量迁移
  async migrateAll(dryRun = false): Promise<MigrationReport> {
    const features = await this.scanAllFeatures();
    const report = { total: 0, migrated: 0, errors: [] };
    
    for (const feature of features) {
      const state = await this.readState(feature.id);
      
      if (state.version !== '2.0.0') {
        report.total++;
        
        if (!dryRun) {
          try {
            const migrated = await this.migrate(state, '2.0.0');
            await this.saveState(feature.id, migrated);
            report.migrated++;
          } catch (error) {
            report.errors.push({ feature: feature.id, error: error.message });
          }
        }
      }
    }
    
    return report;
  }
}
```

**优点**:
- ✅ 灵活性高：支持自动和手动迁移
- ✅ 风险可控：可以 dry-run 预览
- ✅ 渐进完成：不强制一次性完成

**缺点**:
- ❌ 实现最复杂：需要支持多种迁移模式
- ❌ 测试工作量大：需要测试各种场景

**实现复杂度**: 高

## 3. 决策

### 3.1 选定方案

**决定采用：方案 C（混合模式）**

### 3.2 决策理由

1. **风险与效率平衡**: 自动迁移方便，批量工具高效
2. **向后兼容**: 支持多版本读取，平滑过渡
3. **用户友好**: dry-run 预览，用户可以控制迁移
4. **符合规范要求**: 符合 spec v2.0.0 对 FR-006 的要求

### 3.3 Schema 定义

```typescript
// src/state/schema-v2.0.0.ts

export type FeatureStatus = 
  | 'specified' | 'planned' | 'tasked' 
  | 'implementing' | 'reviewed' | 'validated' | 'completed';

export interface StateV2_0_0 {
  // 基本信息
  feature: string;                    // Feature ID (specs-tree-xxx)
  name: string;                       // Feature 名称
  version: '2.0.0';                   // Schema 版本
  
  // 状态信息
  status: FeatureStatus;              // 当前状态
  phase: number;                      // 当前阶段 (1-6)
  level: number;                      // 嵌套层级 (1-10)
  parent: string | null;              // 父 Feature ID
  
  // 文件引用
  files: {
    spec: string;
    plan: string;
    tasks: string;
    readme: string;
    review?: string;
    validation?: string;
  };
  
  // 依赖关系
  dependencies: {
    on: string[];                     // 依赖的 Feature ID 列表
    blocking: string[];               // 被哪些 Feature 阻塞
  };
  
  // 历史记录
  history: HistoryEntry[];
  phaseHistory: PhaseHistoryEntry[];
  
  // 时间戳
  createdAt: string;                  // ISO 8601
  updatedAt: string;                  // ISO 8601
}

export interface HistoryEntry {
  timestamp: string;
  from: string;
  to: string;
  triggeredBy: string;
  actor?: string;
  comment?: string;
}

export interface PhaseHistoryEntry {
  timestamp: string;
  phase: number;
  phaseName: string;
  triggeredBy: string;
  actor?: string;
}
```

### 3.6 兼容性层

```typescript
// src/state/compatibility.ts
class CompatibilityLayer {
  // 读取：支持多版本
  async readState(featureId: string): Promise<StateV2_0_0> {
    const raw = await this.readRawState(featureId);
    
    switch (raw.version) {
      case '1.2.5':
      case '1.2.11':
        return await this.migrate(raw, '2.0.0');
      case '2.0.0':
        return raw;
      default:
        throw new Error(`不支持的 Schema 版本：${raw.version}`);
    }
  }
  
  // 写入：统一 v2.0.0
  async writeState(featureId: string, state: StateV2_0_0): Promise<void> {
    // 验证版本
    if (state.version !== '2.0.0') {
      throw new Error('写入必须使用 v2.0.0 Schema');
    }
    
    await this.saveState(featureId, state);
  }
}
```

### 3.7 验证策略

```typescript
// 迁移后验证
async function validateMigration(state: StateV2_0_0): Promise<ValidationResult> {
  const errors = [];
  
  // 必需字段验证
  if (!state.feature) errors.push('缺少 feature 字段');
  if (!state.name) errors.push('缺少 name 字段');
  if (state.version !== '2.0.0') errors.push('版本不是 2.0.0');
  
  // 状态验证
  const allowedStatuses = ['specified', 'planned', 'tasked', 'implementing', 'reviewed', 'validated'];
  if (!allowedStatuses.includes(state.status)) {
    errors.push(`无效状态：${state.status}`);
  }
  
  // 阶段验证
  if (state.phase < 1 || state.phase > 6) {
    errors.push(`无效阶段：${state.phase}`);
  }
  
  // 依赖验证
  if (!Array.isArray(state.dependencies.on)) {
    errors.push('dependencies.on 必须是数组');
  }
  
  return {
    valid: errors.length === 0,
    errors
  };
}
```

## 4. 后果

### 4.1 积极影响

- ✅ Schema 统一：所有 Feature 使用相同版本
- ✅ 代码简化：只需支持 v2.0.0
- ✅ 向后兼容：旧版本可以读取和迁移
- ✅ 功能增强：支持 history 和 phaseHistory

### 4.2 消极影响

- ⚠️ 迁移工作量：需要迁移所有现有 Feature
- ⚠️ 兼容层复杂：需要支持多版本读取
- ⚠️ 数据丢失风险：迁移过程可能丢失字段

### 4.3 缓解措施

1. **备份**: 迁移前自动备份旧版本
2. **验证**: 迁移后验证数据完整性
3. **回滚**: 提供回滚机制
4. **日志**: 详细记录迁移过程

### 4.4 需要遵循的约定

1. **版本统一**: 新写入必须使用 v2.0.0
2. **自动迁移**: 首次访问旧版本时自动迁移
3. **备份**: 迁移前创建备份
4. **验证**: 迁移后必须验证
