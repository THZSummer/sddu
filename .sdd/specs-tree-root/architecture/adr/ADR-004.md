# ADR-004: 辅导模式采用 4 级分类

## 1. 背景

### 1.1 问题陈述

Discovery Agent 需要适应用户不同的需求成熟度，提供不同程度的辅导干预。需要确定辅导模式的分级策略。

**用户状态多样性**：
- 有的用户只有一个模糊想法
- 有的用户已经明确了痛点
- 有的用户已有初步方案
- 有的用户只需要确认范围边界

### 1.2 决策范围

本 ADR 决定：
- 辅导模式的分级数量
- 每级的定义和特征
- 各级的干预程度和引导方式
- 级别判断方式（自动 vs 手动）

---

## 2. 方案对比

### 2.1 方案 A：4 级分类（推荐）⭐

**描述**：
将用户需求成熟度分为 4 个阶段：

| 级别 | 名称 | 特征 | 干预程度 |
|------|------|------|----------|
| 1 | 想法阶段 | 只有模糊想法，不清楚具体问题 | 高干预 |
| 2 | 痛点阶段 | 明确了痛点，但不知道如何解决 | 中干预 |
| 3 | 方案阶段 | 有了初步方案，需要验证可行性 | 低干预 |
| 4 | 执行阶段 | 方案明确，只需要确认范围和边界 | 最低干预 |

**各级引导策略**：

```
想法阶段：
  - 引导式提问："您想解决什么问题？"
  - 5 Whys 深度挖掘
  - 帮助从模糊到清晰

痛点阶段：
  - 深度挖掘痛点："为什么这是个问题？"
  - 量化影响："这个问题造成了什么损失？"
  - 识别受影响的用户

方案阶段：
  - 验证方案可行性
  - 识别潜在风险
  - 对比替代方案

执行阶段：
  - 确认范围边界
  - 定义成功标准
  - 明确 MVP 范围
```

**优点**：
- ✅ 覆盖完整：从想法到执行的全流程
- ✅ 粒度适中：4 级既不太粗也不太细
- ✅ 易于理解：用户容易理解自己所在阶段
- ✅ 自动判断可行：基于输入内容可以较准确判断

**缺点**：
- ❌ 边界模糊：某些情况难以明确归类
- ❌ 自动判断可能不准：需要用户确认或手动调整

---

### 2.2 方案 B：3 级分类

**描述**：
简化为 3 级：初期 / 中期 / 后期

| 级别 | 名称 | 特征 |
|------|------|------|
| 1 | 初期 | 想法和痛点阶段合并 |
| 2 | 中期 | 方案阶段 |
| 3 | 后期 | 执行阶段 |

**优点**：
- ✅ 更简单：用户更容易理解
- ✅ 实现简单：判断逻辑更少

**缺点**：
- ❌ 粒度太粗：初期内部差异大，难以精准辅导
- ❌ 辅导效果差：想法阶段和痛点阶段需要不同的引导策略

---

### 2.3 方案 C：5 级或更多分类

**描述**：
更细粒度的分类，如：
1. 意识萌芽
2. 问题定义
3. 痛点确认
4. 方案探索
5. 方案确认
6. 执行准备

**优点**：
- ✅ 更精准：可以更细致地匹配辅导策略

**缺点**：
- ❌ 过于复杂：用户难以理解
- ❌ 判断困难：级别之间差异太小，难以准确判断
- ❌ 实现复杂：需要维护多套辅导逻辑

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 A（4 级分类）**

### 3.2 决策理由

1. **完整性**：覆盖从想法到执行的完整流程
2. **实用性**：每级都有明确的辅导策略差异
3. **可判断性**：基于用户输入可以较准确判断级别
4. **简洁性**：4 级是复杂度的甜点，既不太粗也不太细

### 3.3 级别判断逻辑

```typescript
function detectCoachingLevel(userInput: string): CoachingLevel {
  // 基于关键词和输入特征判断
  
  // 想法阶段特征：短、模糊、疑问句
  if (userInput.length < 30 && 
      (userInput.includes('想') || userInput.includes('可以') || 
       userInput.includes('吗') || userInput.includes('?'))) {
    return '想法阶段';
  }
  
  // 痛点阶段特征：包含问题、痛点、困扰等词
  if (userInput.includes('痛点') || userInput.includes('问题') ||
      userInput.includes('困扰') || userInput.includes('困难')) {
    return '痛点阶段';
  }
  
  // 方案阶段特征：包含方案、实现、技术、设计等词
  if (userInput.includes('方案') || userInput.includes('实现') ||
      userInput.includes('技术') || userInput.includes('设计')) {
    return '方案阶段';
  }
  
  // 执行阶段：详细的实现计划、范围确认
  return '执行阶段';
}
```

### 3.4 各级干预策略详情

| 级别 | 干预程度 | 提问数量 | 引导深度 | 示例提问 |
|------|----------|----------|----------|----------|
| 想法阶段 | 高 | 7-10 个 | 深度引导 | "能具体说说您遇到的问题吗？" |
| 痛点阶段 | 中 | 5-7 个 | 痛点挖掘 | "这个问题对用户的影响有多大？" |
| 方案阶段 | 低 | 3-5 个 | 验证确认 | "这个方案的技术风险是什么？" |
| 执行阶段 | 最低 | 2-3 个 | 边界确认 | "MVP 的最小范围是什么？" |

### 3.5 用户手动覆盖

虽然提供自动判断，但允许用户手动指定：

```
系统：检测到您处于 [想法阶段]，将采用引导式辅导模式。
      如需调整，请输入：/coaching-level [想法 | 痛点 | 方案 | 执行]

用户：/coaching-level 痛点
系统：已切换到 [痛点阶段] 辅导模式
```

## 4. 后果

### 4.1 积极影响

- ✅ 辅导精准：不同阶段获得合适的引导
- ✅ 用户体验好：不会感到过度干预或支持不足
- ✅ 适应性强：覆盖各种需求成熟度
- ✅ 可扩展：未来可以增加更多级别

### 4.2 消极影响

- ⚠️ 判断可能不准：自动判断有时需要用户手动调整
- ⚠️ 实现复杂度：需要维护 4 套引导策略
- ⚠️ 测试工作量大：需要测试各级别的辅导效果

### 4.3 需要遵循的约定

1. **级别名称固定**：使用"想法/痛点/方案/执行"四阶段命名
2. **默认自动判断**：优先自动判断，允许手动覆盖
3. **引导策略一致**：同一级别的引导策略应保持一致
4. **持续优化**：根据用户反馈调整判断逻辑
