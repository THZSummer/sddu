## 1. 问题陈述

### 1.1 背景

参考用户真实项目 (`/home/usb/workspace/Product/open-app/`) 的实践，发现当前 SDD 工作流从 spec 阶段直接开始存在以下问题：

- **直接进行 spec 效果不好**，对用户的真实需求挖掘理解不够
- **用户需求理解不深入**，导致规范质量不稳定
- **缺少结构化的需求挖掘流程**
- **需求假设未显性化**，后期变更成本高

### 1.2 核心问题

**能力封闭**：SDD 工作流缺少前置的需求挖掘阶段，导致：

- 规范返工率高
- 开发过程中需求变更频繁
- 交付质量受影响

### 1.3 影响

- 开发流程不规范，容易跳过关键需求澄清
- 团队协作时需求理解不同步
- 质量问题难以追溯到需求源头

## 4. 功能需求

### 4.1 Discovery Agent (FR-001)

**FR-001: 系统必须创建 @sdd-discovery Agent，提供 7 步需求挖掘工作流**

#### 7 步工作流

1. **问题空间探索** - 背景理解、痛点挖掘、业务价值澄清
2. **用户画像与场景** - 目标用户定义、用户场景还原、用户旅程地图
3. **需求分类与优先级** - MoSCoW 分类、需求归类、优先级排序
4. **竞品与方案调研** - 竞品对标、替代方案评估
5. **风险与假设识别** - 关键假设、潜在风险
6. **成功标准定义** - 定性指标、定量指标、验收标准
7. **范围边界划定** - MVP/V1/未来版本规划

#### 验收标准

- [ ] AC-001.1: Agent 必须提供完整的 7 步工作流
- [ ] AC-001.2: 支持辅导模式（根据用户水平调整干预程度）
- [ ] AC-001.3: 输出需求挖掘报告到 `.sdd/specs-tree-root/[feature]/discovery.md`
- [ ] AC-001.4: 辅导模式包含 4 种用户水平（想法/痛点/方案/执行阶段）

### 4.2 状态机更新 (FR-002)

**FR-002: 系统必须更新状态机支持 discovery 阶段**

#### 状态流转

```
drafting → discovered → specified → planned → tasked → implementing → reviewed → validated
             ↓
      (新增状态)
```

#### 验收标准

- [ ] AC-002.1: 状态机必须支持 `discovered` 状态
- [ ] AC-002.2: 状态流转必须经过验证
- [ ] AC-002.3: 智能入口必须检查 discovery 前置条件（推荐但不强制）

### 4.3 智能入口更新 (FR-003)

**FR-003: 系统必须更新 @sdd 智能入口**

#### 验收标准

- [ ] AC-003.1: 必须支持 `@sdd discovery [name]` 命令
- [ ] AC-003.2: 状态流转图必须包含 discovery
- [ ] AC-003.3: 尝试跳过 discovery 直接 spec 时必须提示推荐（但允许）
- [ ] AC-003.4: 阶段跳转验证表必须更新

### 4.4 输出模板 (FR-004)

**FR-004: 系统必须提供需求挖掘报告模板**

#### 模板结构

1. 问题定义
2. 用户画像
3. 需求清单（MoSCoW 分类）
4. 推荐方案
5. 成功指标
6. 范围边界
7. 风险与假设

#### 验收标准

- [ ] AC-004.1: 必须提供标准模板
- [ ] AC-004.2: 模板必须包含所有核心章节
- [ ] AC-004.3: 支持 Markdown 格式

- **NFR-006**: Agent 定义必须放在 `dist/templates/agents/` 目录
