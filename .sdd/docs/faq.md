# OpenCode SDDU 常见问题 (FAQ)

> **文档版本**: SDDU-FAQ-1.0  
> **更新日期**: 2026-04-06  
> **状态**: 已发布

## 🎯 FAQ 概览

这份 FAQ 文档回答关于 SDDU (Specification-Driven Development Ultimate) 的常见问题。包括关于 SDD 到 SDDU 变迁的信息。

---

## 1. 通用问题

### Q1: 什么是 SDDU？它与 SDD 有什麼区别？
**A**: SDDU 指 **Specification-Driven Development Ultimate**（规范驱动开发终极版），这是对 SDD (Specification-Driven Development) 的功能增强和品牌升级。主要区别：

- **命令系统**: SDDU 提供了 `@sddu-*` 系列命令并兼容舊的 `@sdd-*` 系列命令
- **新增阶段**: SDDU 在原本 6 个阶段基础上新增了阶段 0：Discover (需求挖掘)
- **品牌升级**: 提供更好的视觉识别、用户体验和功能集
- **向后兼容**: 完全保护现有投资，SDD 所有功能依然有效

### Q2: SDDU 跟 SDD 是不同产品还是同一产品？
**A**: 这是同一个产品的不同版本。SDDU 是 SDD 的升级版，提供增强功能的同时保持完全向后兼容。您可以：

- 继续使用原有的 `@sdd-*` 命令不变  
- 尝試新的 `@sddu-*` 命令获得增强功能
- 在同一项目中混合使用兩种命令

### Q3: 为什么需要改名 SDD 到 SDDU？
**A**: 改名的原因是为了：
- 标明这是一个重大升级版本 (Ultimate)
- 避免與其他類似產品混淆
- 為未來增加更多功能提供清晰的品牌定位
- 支持更豐富的工作流和協作模式

---

## 2. 關於雙版本兼客

### Q4: SDDU 是替代 SDD 嗎？现有的项目會受影响嗎？
**A**: SDDU 並非直接替代 SDD，而是提供了一個平穩的升級途徑：

✅ **現有項目完全兼容**：您不需要对现有项目進行任何更改
✅ **舊命令繼續可用**：所有 `@sdd-*` 命令仍然正常工作  
✅ **數據結構統一**：兩種系統共享相同的 `.sdd/` 目錄結構
✅ **漸進式採用**：您可以選擇性地開始使用 `@sddu-*` 命令

### Q5: SDD 會不會很快被棄用？
**A**: 
- 短期內不會：我們保證至少 24 個月的兼容支持期
- 長期支持：對於仍有大量投資的 SDD 項目，我們將繼續維護兼容性
- 自由選取：您可以根據項目的實際情況來決定何時遷移
- 混合使用：可以在同一項目中混合使用兩種命令

### Q6: 我應該繼續用 `@sdd-*` 命令還是在開始使用 `@sddu-*` 命令？
**A**: 
- **現有項目**：建議繼續使用 `@sdd-*`，以保護您的學習和配置投資
- **新項目**：推薦使用 `@sddu-*`，以獲得最新功能和最佳支援
- **混合模式**：您也可以在同一項目中依功能選擇性地使用不同命令

### Q7: 倘若兩個系統共享相同資料，混合使用會出問題嗎？
**A**: 不會。SDD 和 SDDU 兩個系統設計時就考慮了兼容性：
- 同樣的檔案格式（spec.md, plan.md, tasks.md 等）
- 相同的狀態機制（.state.json）
- 共享相同的目錄結構（.sdd/specs-tree-root/）
- 開発狀態和工作進度可互相認得

## 3. 關於新功能和階段

### Q8: SDDU 新增的階段 0 (Discovery) 是什麼？
**A**: 階段 0 - Discover (需求挖掘) 是一個新的初始階段，專門用於：
- 深入挖掘用戶需求和期望
- 驗識潛在的邊界條件和使用場景
- 為規範階段提供充分的需求基礎
- 可能的話，探索不同解決方案的選項

您可以使用 `@sddu-discovery` 或 `@sdd-discovery` 來啟動需求挖掘。

### Q9: 既然新增了階段 0，那麽其他階段會調整嗎？
**A**: 階段數字已進行相應調整：
- 原「階段 3」變成了「階段 4」
- 原「階段 4」變成了「階段 5」  
- 原「階段 5」變成了「階段 6」
- 原「階段 6」變成了「階段 7」
- 這是一個內部調整，對於開發者來說，您只需關注功能本身

## 4. 關於遷移

### Q10: 我如何將現有 SDD 機能遷移到 SDDU 機能？
**A**: 多數情況下您不需要做任何遷移：

- **資料結構不變**：您現有的 `.sdd/` 目錄和檔案不需要更動
- **命令可繼續**：原有的 `@sdd-*` 指令依然有效
- **漸進昇級**：當您的專案下次修改時，您可以選擇使用 `@sddu-*` 命令

如果您確實需要手動遷移至完整 SDDU 體驗，請參見 `.sdd/docs/migration-guide.md`。

### Q11: 達到 SDDU 功能時是否需要重新學習？
**A**: 不需要完全重新學習：

- **核心概念 unchanged**：規範驅動開發的方法論未改變
- **大部分命令相似**：只是前綴從 `@sdd-` 改為 `@sddu-`
- **平滑過渡**：您可以逐步適應新的命令和功能
- **幫助支援**：使用 `@sddu-help` 來了解新增功能

---

## 5. 關於設定和安裝

### Q12: 需要重新安裝才能使用 SDDU？
**A**: 不需要完全重新安裝。根據您的 SDDU 插件版本：
- 如果已安裝最新版本，`@sddu-*` 命令應該已經可用
- 否則需要更新插件以獲取最新功能
- 但不影響您現有的 SDD 環境

### Q13: opencode.json 配置會自動更新嗎？
**A**: 
- 插件配置會在更新時自動處理兼容性
- 新舊命令均可識別和執行
- 建議檢查您的 `opencode.json` 是否支援最新功能
  
使用 `@sddu-help` 來查看最新的配置建議。

---

## 6. 關於錯誤和問題解決

### Q14: 如果 `@sddu-*` 和 `@sdd-*` 命令產生不同的結果怎麼辦？
**A**: 請知悉這兩個版本是設計為了保持一致的行為：

- **數據處理一致**：兩系統以相同方式解析和處理 `.sdd/` 檔案
- **輸出格式相同**：產生相同結構的 output files
- **邏輯算法相同**：核心工作流沒有變化

如果發現不一致，請提 issues 以便修復。

### Q15: 如何知道我是用 SDD 還是 SDDU 已執行？
**A**: 您可通過以下方式辨識：

- **指令本身**：以 `@sddu-` 開頭的是 SDDU，以 `@sdd-` 開頭的是 SDD
- **狀態輸出**：SDDU 可能在某些訊息中標記為 "SDDU 模式"
- **新增功能**：使用階段 0 功能通常是 SDDU 系統

執行 `@sdd-help` 或 `@sddu-help` 以獲得更多詳細資訊。

---

## 7. 關於未來發展

### Q16: 未來的新功能會只在 SDDU 中提供嗎？
**A**: 我們的策略是：

- **核心功能**：主要新功能將集中在 `@sddu-*` 命令
- **SDD 維護**：現有 SDD 功能將繼續維護以支援兼容
- **逐漸轉移**：鼓勵用戶逐漸遷移到 SDDU 獲得更好體驗

### Q17: 有什麼工具可以幫助我從 SDD 轉換到 SDDU？
**A**: 我們提供：
- 逐步遷移指南（`.sdd/docs/migration-guide.md`）
- 自動化的向後兼容機制
- 標準化的工作流以減少轉換複雜性
- 交互使用的靈活性

目前暫無全自動遷移工具，因為大多數情況下無需強制遷移。

## 8. 支持和聯繫

### Q18: 遇到問題時，我可以去哪里尋求協助？
**A**: 您有以下幾種尋求協助的管道：

- 使用 `@sddu-help` 或 `@sdd-help` 獲取內建說明
- 閱讀 `.sdd/docs/migration-guide.md` 瞭解詳細資訊
- 檢視 `.sdd/ROADMAP.md` 瞭解產品發展路徑
- 在項目社群中發問獲得協助

---

### Q19: 什麼時候是開始使用 SDDU 的最佳時機？
**A**: 
- **立即開始**：新的項目可以立即採用 SDDU
- **下個週期**：現有項目的下一個開發週期開始時
- **新功能時**：當您需要使用階段 0 (Discovery) 功能時
- **团队统一**：团队统一决定採用新標准时

您可以根據您的專案時程和需求來靈活選擇。

---

## 9. 關於性能和資源

### Q20: SDDU 是否需要更多的資源？
**A**: 基本上不需要更多的系統資源：
- 檔案存儲：數據結構相同，檔案大小相似
- 計算邏輯：核心算法未改變
- 網路通訊：相同 API 和服務調用
- 處理時間：執行時間與 SDD 相近

新增的 Discovery 階段可能會稍微增加初始的準備時間，這帶來了更深入的需求理解。

## 💡 提示和建議

1. **漸進式適應**：不需要立即切換所有項目到 SDDU
2. **混合使用**：在同一項目中按需混合使用兩種命令
3. **查看新功能**：利用 `@sddu-discovery` 嘗試新的需求挖掘
4. **保持聯繫**：關注後續的 SDDU 更新和新功能
