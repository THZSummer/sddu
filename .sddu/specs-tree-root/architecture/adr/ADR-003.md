# ADR-003: Discovery 阶段可选而非强制

| 元数据 | 值 |
|--------|-----|
| **ADR ID** | ADR-003 |
| **标题** | Discovery 阶段可选而非强制 |
| **状态** | PROPOSED |
| **创建日期** | 2026-04-03 |

## 1. 背景

### 1.1 问题陈述

在 SDD 7 阶段工作流设计中，需要决定是否强制用户在进入 spec 阶段之前执行 discovery 阶段。

**核心问题**：
- 强制 discovery 可能增加简单需求的流程负担
- 不强制可能导致用户跳过关键的需求挖掘步骤
- 需要在流程规范性和灵活性之间取得平衡

### 1.2 决策范围

本 ADR 决定：
- Discovery 阶段是否强制执行
- 跳过 discovery 的后果和处理方式
- 智能入口的提示策略

---

## 2. 方案对比

### 2.1 方案 A：可选但推荐（推荐）⭐

**描述**：
- Discovery 阶段不强制
- 用户可以直接进入 spec 阶段
- 智能入口会显示推荐提示
- 不阻止用户的跳过行为

**流程示例**：
```
用户：@sdd spec my-feature
系统：⚠️ 建议先执行 @sdd discovery my-feature 进行需求挖掘，是否继续？
用户：是
系统：开始 spec 阶段...
```

**优点**：
- ✅ 灵活性高：适应不同复杂度的需求
- ✅ 向后兼容：不影响现有工作流
- ✅ 用户友好：不强制，减少抵触情绪
- ✅ 渐进式采用：用户可在合适时机尝试 discovery

**缺点**：
- ❌ 可能不被采用：用户可能习惯性跳过
- ❌ 价值难体现：使用率低时难以证明价值
- ❌ 需要额外提示逻辑：增加实现复杂度

---

### 2.2 方案 B：强制执行

**描述**：
- 必须先完成 discovery 才能进入 spec
- 没有 discovery.md 时拒绝执行 spec
- 严格的阶段依赖检查

**流程示例**：
```
用户：@sdd spec my-feature
系统：❌ 错误：请先执行 @sdd discovery my-feature
```

**优点**：
- ✅ 保证需求挖掘质量
- ✅ 强制规范化流程
- ✅ 实现简单：直接拒绝即可

**缺点**：
- ❌ 灵活性差：简单需求也要走完整流程
- ❌ 用户抵触：可能被视为繁文缛节
- ❌ 破坏向后兼容：现有工作流被打断
- ❌ 学习曲线陡峭：新用户需要理解更多概念

---

### 2.3 方案 C：基于复杂度智能推荐

**描述**：
- 分析用户需求的复杂度
- 复杂需求强制 discovery，简单需求可选
- 使用 NLP 或其他方法判断复杂度

**优点**：
- ✅ 智能适配：根据场景调整
- ✅ 平衡灵活性和规范性

**缺点**：
- ❌ 实现复杂：需要复杂度判断逻辑
- ❌ 判断可能不准确：误判导致用户体验差
- ❌ 边界情况多：难以定义清晰的复杂度阈值

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 A（可选但推荐）**

### 3.2 决策理由

1. **尊重用户选择**：用户最了解自己的需求复杂度，应有权决定流程
2. **降低采用门槛**：不强制可以减少新用户的抵触情绪
3. **向后兼容**：保持与现有 6 阶段工作流的兼容性
4. **渐进式改进**：允许用户逐步接受新的工作方式
5. **简单需求不需要**：对于明确、简单的需求，discovery 可能是多余的

### 3.3 提示策略

虽然不强制，但会通过以下方式鼓励使用：

```markdown
当用户尝试跳过 discovery 直接 spec 时：

⚠️ 注意到您还没有执行 discovery 阶段。

💡 建议：对于复杂或不确定的需求，先运行 `@sdd discovery [feature]` 可以帮助您：
   - 深入挖掘真实需求
   - 显性化关键假设
   - 减少后期返工

❓ 您是否要继续执行 spec 阶段？（输入"是"继续，或先运行 discovery）
```

### 3.4 使用场景建议

| 场景类型 | 建议 |
|----------|------|
| 全新复杂功能 | 强烈推荐 discovery |
| 现有功能迭代 | 可选，根据变更幅度决定 |
| Bug 修复 | 通常不需要 discovery |
| 技术重构 | 可选，根据重构范围决定 |
| 紧急修复 | 可以跳过 |

---

## 4. 后果

### 4.1 积极影响

- ✅ 用户接受度高：不会感到被强制
- ✅ 灵活性好：适应不同场景
- ✅ 采用阻力小：可以逐步推广
- ✅ 保持兼容性：不影响现有工作流

### 4.2 消极影响

- ⚠️ 采用率可能不高：需要持续宣导价值
- ⚠️ 质量提升有限：如果用户不使用的
- ⚠️ 需要额外提示逻辑：增加代码复杂度

### 4.3 需要遵循的约定

1. **提示语设计**：友好、非强制、说明价值
2. **不阻止跳过**：最终决定权在用户
3. **记录跳过行为**：可选记录用户跳过的情况用于分析
4. **定期评估**：收集用户反馈，必要时调整策略

## 5. 验证策略

| 验证项 | 验证方法 | 预期结果 |
|--------|----------|----------|
| 可以跳过 discovery | 直接运行 spec 命令 | 允许执行，显示提示 |
| 提示信息准确 | 检查提示内容 | 包含价值和推荐 |
| 不影响现有流程 | 测试现有 Feature | 行为与之前一致 |
| 用户可以选择 | 提供明确的是/否选项 | 用户有控制权 |
