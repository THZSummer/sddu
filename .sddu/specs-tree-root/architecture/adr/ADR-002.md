# ADR-002: bcrypt 密码加密方案

| 属性 | 值 |
|------|-----|
| **状态** | ACCEPTED |
| **创建日期** | 2026-04-02 |
| **决策者** | SDD Technical Team |
| **相关 Feature** | login-module (FR-AUTH-001) |

---

## 背景

用户密码需要安全存储，防止数据库泄露后密码被破解。需要在多种密码哈希算法中选择合适的方案。

### 安全要求

- 抗暴力破解
- 抗彩虹表攻击
- 每用户唯一盐值
- 可调整计算成本
- 业界广泛验证

---

## 方案对比

### 方案 A: bcrypt（推荐）

**描述**: 基于 Blowfish 的自适应哈希函数，专为密码存储设计。

**优点**:
- ✅ 内置盐值，无需额外存储
- ✅ 可调整 cost factor 适应硬件发展
- ✅ 业界标准，广泛使用验证
- ✅ 抗 GPU 加速破解
- ✅ Node.js 生态成熟（bcrypt 包）

**缺点**:
- ❌ 计算速度较慢（约 100ms/次）
- ❌ 内存使用较低，ASIC 抗性一般
- ❌ 最大密码长度限制 72 字节

**安全参数**:
```typescript
{
  algorithm: 'bcrypt',
  cost: 12,        // 2^12 次迭代
  saltRounds: 12,  // 12 位随机盐
  hashLength: 60   // 输出长度
}
```

**性能**:
- 加密时间：~100ms (cost=12)
- 验证时间：~100ms
- 服务器负载：可接受

---

### 方案 B: Argon2

**描述**: 2015 年密码哈希竞赛获胜者，最新一代密码哈希算法。

**优点**:
- ✅ 抗 GPU/ASIC 攻击最强
- ✅ 可调整内存、时间、并行度
- ✅ 现代算法设计
- ✅ OWASP 推荐

**缺点**:
- ❌ Node.js 实现较新，生态不如 bcrypt 成熟
- ❌ 配置参数复杂（内存、迭代、并行度）
- ❌ 团队熟悉度低
- ❌ 某些环境编译可能有问题

**安全参数**:
```typescript
{
  algorithm: 'argon2id',
  memoryCost: 65536,    // 64MB
  timeCost: 3,          // 3 次迭代
  parallelism: 4        // 4 线程并行
}
```

---

### 方案 C: scrypt

**描述**: 内存密集型哈希函数，设计用于抗硬件加速。

**优点**:
- ✅ 内存密集型，抗 ASIC
- ✅ Node.js 内置支持 (crypto.scrypt)
- ✅ 广泛使用验证

**缺点**:
- ❌ 配置参数复杂 (N, r, p)
- ❌ 性能调优困难
- ❌ 不如 bcrypt 普及

---

### 方案 D: PBKDF2

**描述**: 基于 HMAC 的密钥派生函数，NIST 标准。

**优点**:
- ✅ 标准算法，FIPS 认证
- ✅ 广泛支持
- ✅ 实现简单

**缺点**:
- ❌ 计算密集型但非内存密集型
- ❌ GPU 加速攻击风险高
- ❌ 需要更高迭代次数达到同等安全

---

## 决策

**选择方案 A: bcrypt**

### 决策理由

1. **成熟稳定**: bcrypt 已使用 20+ 年，经过充分验证
2. **生态完善**: Node.js 的 bcrypt 包维护良好，文档丰富
3. **团队熟悉**: 团队有 bcrypt 使用经验
4. **安全足够**: cost=12 提供足够安全保障
5. **性能平衡**: 100ms 加密时间可接受

### 关键配置

```typescript
// 密码加密配置
const BCRYPT_COST = 12;

// 加密
const hash = await bcrypt.hash(password, BCRYPT_COST);

// 验证
const isValid = await bcrypt.compare(password, hash);

// 密码策略
{
  minLength: 8,
  requireLetters: true,
  requireNumbers: true,
  maxAge: null  // 暂不强制定期更换
}
```

### 未来升级路径

如果未来安全要求提高，可考虑：
1. 增加 bcrypt cost 至 14-15
2. 或迁移至 Argon2id（需要用户下次登录时重新哈希）

## 后果

### 正面影响

- ✅ 密码存储安全性高
- ✅ 实现简单，减少 bug
- ✅ 团队快速上手

### 负面影响

- ⚠️ 登录时有约 100ms 加密延迟（可接受）
- ⚠️ 高并发登录时需要足够的 CPU 资源

### 运维要求

| 项目 | 要求 |
|------|------|
| CPU 资源 | 每 100 QPS 登录需要约 10 核 CPU |
| 监控 | 监控密码验证延迟 P95 < 200ms |
| 日志 | 禁止记录明文密码 |
