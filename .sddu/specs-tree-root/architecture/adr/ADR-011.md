## 1. 背景

### 1.1 问题陈述

当前 SDD 插件的类型定义分散在多个文件中：

| 类型文件 | 位置 | 导出类型 |
|----------|------|----------|
| `state/machine.ts` | 状态机核心 | `FeatureStateEnum`, `FeatureState`, `TransitionResult` |
| `state/schema-v2.0.0.ts` | Schema 定义 | `StateV2_0_0`, `WorkflowStatus`, `PhaseHistory` |
| `discovery/types.ts` | Discovery | `DiscoveryStep`, `DiscoveryContext`, `CoachingLevel` |
| `agents/sdd-agents.ts` | Agent 注册 | `AgentIntegrationResult` |
| `utils/tasks-parser.ts` | 工具函数 | `ParsedTask`, `ParallelGroup` |
| `utils/subfeature-manager.ts` | 工具函数 | `SubFeatureMeta` |

**问题**:
1. **导入路径复杂**: 开发者需要知道每个类型的具体位置
2. **循环依赖风险**: 模块间交叉导入可能导致循环依赖
3. **类型发现困难**: 新开发者难以找到可用的类型定义
4. **维护成本高**: 类型变更需要同步更新多个导入点

### 1.2 决策范围

本 ADR 决定：
- 创建统一类型出口文件 `src/types.ts`
- 定义类型 re-export 规则
- 保持原有类型文件不变（向后兼容）
- 提供迁移指南

---

## 2. 决策

### 2.1 方案选择

我们决定采用 **统一类型出口** 方案：

**创建 `src/types.ts` 作为唯一类型导入入口**。

### 2.2 设计原则

1. **单一出口**: 所有公共类型从 `src/types` 导入
2. **Re-export 模式**: 不移动原有类型定义，仅 re-export
3. **向后兼容**: 原有导入路径仍然有效
4. **分类清晰**: 按模块分组导出（State, Discovery, Agent, Utility）

### 2.3 实现细节

```typescript
// src/types.ts

// State Types
export type {
  FeatureStateEnum,
  FeatureState,
  TransitionResult
} from './state/machine';

export type {
  StateV2_0_0,
  WorkflowStatus,
  PhaseHistory
} from './state/schema-v2.0.0';

// Discovery Types
export type {
  DiscoveryStep,
  DiscoveryContext,
  CoachingLevel
} from './discovery/types';

// Agent Types
export type {
  AgentIntegrationResult
} from './agents/sdd-agents';

// Agent Metadata (新增)
export interface AgentMetadata {
  name: string;
  description: string;
  mode: 'subagent' | 'tool';
  promptFile: string;
  category?: 'spec' | 'plan' | 'tasks' | 'build' | 'review' | 'validate' | 'utility';
}

// Utility Types
export type {
  ParsedTask,
  ParallelGroup
} from './utils/tasks-parser';

export type {
  SubFeatureMeta
} from './utils/subfeature-manager';
```

### 2.4 使用示例

**旧方式**:
```typescript
import { FeatureStateEnum } from './state/machine';
import { DiscoveryContext } from './discovery/types';
import { ParsedTask } from './utils/tasks-parser';
```

**新方式**:
```typescript
import type {
  FeatureStateEnum,
  DiscoveryContext,
  ParsedTask
} from './types';
```

## 3. 后果

### 3.1 正面影响

| 影响 | 说明 |
|------|------|
| ✅ 简化导入路径 | 所有类型从单一入口导入 |
| ✅ 减少循环依赖 | 避免模块间交叉导入 |
| ✅ 提高类型发现性 | 开发者知道从 `src/types` 查找 |
| ✅ 便于类型管理 | 集中管理公共类型出口 |

### 3.2 负面影响

| 影响 | 说明 | 缓解措施 |
|------|------|----------|
| ⚠️ 增加 re-export 维护 | 需要保持 re-export 列表更新 | 代码审查时检查 |
| ⚠️ 可能遗漏类型 | 新增类型可能忘记 re-export | 编写类型导出测试 |
| ⚠️ 迁移成本 | 现有代码需要更新导入路径 | 提供迁移指南，保持向后兼容 |

### 3.3 技术影响

- **编译时间**: 无明显影响
- **运行时性能**: 无影响（类型在编译时擦除）
- **包大小**: 无影响
- **向后兼容**: 完全兼容（原有文件不变）

## 4. 替代方案

### 方案 A: 移动类型定义（已否决）

**描述**: 将所有类型定义移动到 `src/types.ts`，删除原有类型文件。

**优点**:
- 代码结构最清晰
- 无重复定义

**缺点**:
- 破坏性变更
- 需要更新所有导入路径
- 风险高

**否决理由**: 破坏性变更，不符合渐进式重构原则。

### 方案 B: 保持现状（已否决）

**描述**: 不做任何改变，维持现有类型分散状态。

**优点**:
- 无变更成本

**缺点**:
- 问题持续存在
- 不利于长期维护

**否决理由**: 不解决问题，技术债务累积。
