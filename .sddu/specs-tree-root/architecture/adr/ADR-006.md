## 1. 背景

### 1.1 问题陈述

当前 StateMachine (`src/state/machine.ts`) 已实现状态转换验证功能，但**未完全集成到 Agent 工作流**中。导致以下问题：

1. **状态更新手动**: Agent 完成后需要手动调用状态更新
2. **验证不自动触发**: 状态转换规则未在每个 Agent 执行后自动验证
3. **历史记录不完整**: 部分状态变更未记录到 history 数组
4. **依赖检查缺失**: 状态变更前未自动检查依赖 Feature 状态

### 1.2 决策范围

本 ADR 决定：
- StateMachine 与 Agent 工作流的集成方式
- 状态更新的触发时机
- 验证和记录的自动化程度
- 错误处理和回滚机制

### 1.3 现状分析

**现有 StateMachine 功能** (已实现):
- ✅ 状态转换规则定义 (`validTransitions`)
- ✅ 状态流转验证 (`canTransition`, `validateStageTransition`)
- ✅ 必需文件检查 (`checkRequiredFiles`)
- ✅ 状态更新 (`updateState`)

**现有 StateMachine 缺口**:
- ❌ 未与 6 个 Agent (@sdd-spec, @sdd-plan, 等) 集成
- ❌ 未在 Agent 完成后自动调用
- ❌ 未与分布式 state.json 集成 (仍使用集中式路径)
- ❌ 未与 session.idle 事件集成

## 2. 方案对比

### 2.1 方案 A：钩子集成模式（推荐）⭐

**描述**：
在每个 Agent 的完成回调中添加状态更新钩子，Agent 任务完成后自动调用 StateMachine。

**架构**:
```
Agent 执行流程:
1. 执行 Agent 任务 (spec/plan/tasks/...)
2. 任务完成，生成文档 (spec.md/plan.md/...)
3. ┌─→ 调用 StateMachine.validateStageTransition()
   │   验证状态转换是否合法
   │
4. ┌─→ 调用 DependencyChecker.check()
   │   检查依赖 Feature 状态
   │
5. ┌─→ 调用 StateManager.updateState()
   │   更新 state.json，追加历史记录
   │
6. 完成，返回成功
```

**优点**:
- ✅ 自动化：Agent 完成后自动更新状态，无需手动操作
- ✅ 一致性：所有 Agent 遵循相同的状态更新流程
- ✅ 验证保证：每次状态变更都经过 StateMachine 验证
- ✅ 历史记录完整：每次变更自动记录到 history

**缺点**:
- ❌ 需要修改 6 个 Agent 的完成逻辑
- ❌ Agent 执行时间增加 (约 50-100ms)
- ❌ 状态更新失败可能影响 Agent 报告

**实现复杂度**: 中

---

### 2.2 方案 B：独立监听器模式

**描述**：
添加独立的 StateChangeListener 监听文件变化，检测到文档创建/修改后自动更新状态。

**架构**:
```
文件系统事件 → StateChangeListener → StateMachine → StateManager
```

**优点**:
- ✅ 解耦：Agent 无需感知状态更新逻辑
- ✅ 灵活：可以监听多个事件源

**缺点**:
- ❌ 延迟：文件变化检测和响应有延迟
- ❌ 不可靠：可能错过某些状态变更
- ❌ 复杂：需要维护文件监听器
- ❌ 调试困难：异步事件流难以追踪

**实现复杂度**: 高

---

### 2.3 方案 C：混合模式

**描述**：
方案 A + 方案 B 结合：
- 主要路径：Agent 钩子触发 (及时)
- 备用路径：session.idle 事件触发 (兜底)

**优点**:
- ✅ 可靠性高：双重保障
- ✅ 及时性好：钩子模式确保及时更新
- ✅ 容错性好：idle 模式修复遗漏

**缺点**:
- ❌ 实现复杂：需要实现两种机制
- ❌ 可能重复更新：需要去重逻辑

**实现复杂度**: 中高

---

## 3. 决策

### 3.1 选定方案

**决定采用：方案 C（混合模式）**

### 3.2 决策理由

1. **可靠性优先**: 双重触发机制确保状态更新不会遗漏
2. **及时性与容错性平衡**: 钩子模式保证及时性，idle 模式作为兜底
3. **渐进实现**: 可以先实现钩子模式，再添加 idle 模式
4. **符合规范要求**: spec v2.0.0 要求 FR-001 (StateMachine 集成) 和 FR-002 (自动更新)

### 3.3 实现细节

#### 3.3.1 钩子集成 (主要路径)

```typescript
// src/agents/sdd-agents.ts (示例)
async function executeAgent(agentName: string, featureId: string) {
  // 1. 执行 Agent 任务
  await agent.execute();
  
  // 2. 生成文档
  await agent.generateDocuments();
  
  // 3. 状态更新钩子
  try {
    await updateFeatureState(featureId, agent.targetState);
  } catch (error) {
    // 记录错误但不阻止 Agent 完成
    logger.warn(`状态更新失败：${error.message}`);
  }
  
  // 4. 返回结果
  return agent.getResult();
}

// 状态更新函数
async function updateFeatureState(featureId: string, targetState: string) {
  const stateMachine = new StateMachine(featureId);
  const stateManager = new MultiFeatureStateManager();
  const dependencyChecker = new DependencyChecker();
  
  // 验证状态转换
  const validation = await stateMachine.validateStageTransition(targetState);
  if (!validation.allowed) {
    throw new Error(`状态转换失败：${validation.reason}`);
  }
  
  // 检查依赖
  const depCheck = await dependencyChecker.check(featureId, targetState);
  if (!depCheck.passed) {
    throw new Error(`依赖检查失败：${depCheck.message}`);
  }
  
  // 更新状态
  await stateManager.updateState(featureId, {
    status: targetState,
    phase: stateToPhase(targetState),
    history: createHistoryEntry(targetState)
  });
}
```

#### 3.3.2 session.idle 触发 (备用路径)

```typescript
// src/state/auto-updater.ts
class StateAutoUpdater {
  private lastKnownState: Map<string, string> = new Map();
  
  async onSessionIdle() {
    // 扫描所有 Feature 目录
    const features = await this.scanFeatures();
    
    for (const feature of features) {
      // 检测文件变更
      const currentState = await this.detectStateFromFiles(feature);
      const lastState = this.lastKnownState.get(feature.id);
      
      if (currentState !== lastState && currentState) {
        // 状态不一致，需要更新
        await this.syncState(feature.id, currentState);
      }
    }
  }
  
  private async detectStateFromFiles(feature: Feature): Promise<string | null> {
    // 根据存在的文件推断状态
    if (await exists('validation.md')) return 'validated';
    if (await exists('review.md')) return 'reviewed';
    if (await exists('tasks.md')) return 'tasked';
    if (await exists('plan.md')) return 'planned';
    if (await exists('spec.md')) return 'specified';
    return null;
  }
}
```

#### 3.3.3 状态与阶段映射

```typescript
function stateToPhase(state: string): number {
  const mapping: Record<string, number> = {
    'specified': 1,
    'planned': 2,
    'tasked': 3,
    'implementing': 4,
    'reviewed': 5,
    'validated': 6,
    'completed': 6
  };
  return mapping[state] || 1;
}
```

```typescript
enum ErrorHandlingStrategy {
  LOG_ONLY = 'log_only',           // 仅记录日志，不阻止
  WARN_USER = 'warn_user',         // 警告用户，提供手动修复命令
  ROLLBACK = 'rollback'            // 回滚 Agent 变更
}

// 默认策略：LOG_ONLY
// 理由：状态更新失败不应阻止 Agent 完成，用户可以手动修复
```

### 3.5 去重逻辑

为避免钩子模式和 idle 模式重复更新：

```typescript
class StateUpdater {
  private recentUpdates: Map<string, number> = new Map(); // featureId -> timestamp
  
  async update(featureId: string, newState: string) {
    const lastUpdate = this.recentUpdates.get(featureId);
    const now = Date.now();
    
    // 5 秒内的更新跳过
    if (lastUpdate && now - lastUpdate < 5000) {
      logger.debug(`跳过重复更新：${featureId}`);
      return;
    }
    
    // 执行更新
    await this.performUpdate(featureId, newState);
    this.recentUpdates.set(featureId, now);
  }
}
```

## 4. 后果

### 4.1 积极影响

- ✅ 状态更新自动化：减少手动操作
- ✅ 状态准确性提升：自动验证和记录
- ✅ 历史记录完整：每次变更都有记录
- ✅ 依赖检查自动化：防止依赖未就绪时推进

### 4.2 消极影响

- ⚠️ Agent 执行时间增加：约 50-100ms (可接受)
- ⚠️ 实现复杂度增加：需要维护两种触发机制
- ⚠️ 调试难度增加：异步事件流需要仔细追踪

### 4.3 需要遵循的约定

1. **钩子位置**: 所有 Agent 的完成回调必须调用状态更新
2. **错误处理**: 状态更新失败不阻止 Agent 完成
3. **去重时间窗口**: 5 秒内跳过重复更新
4. **日志记录**: 所有状态更新操作必须记录日志
