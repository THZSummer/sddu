## 1. 背景

### 1.1 问题陈述

当前 SDD 插件各模块使用不同的错误处理方式：

| 模块 | 错误处理方式 | 问题 |
|------|--------------|------|
| `state/machine.ts` | `throw new Error()` | 无错误码，难以分类处理 |
| `discovery/workflow-engine.ts` | `console.error()` + 返回错误对象 | 日志格式不一致 |
| `utils/tasks-parser.ts` | `throw new Error()` | 无上下文信息 |
| `agents/sdd-agents.ts` | 返回 `{ success, errors }` | 与其他模块不统一 |

**问题**:
1. **错误格式不一致**: 难以统一解析和处理
2. **缺乏错误分类**: 无法按错误类型进行差异化处理
3. **日志格式混乱**: 不同模块输出格式不同
4. **错误上下文缺失**: 难以追踪错误来源

### 1.2 决策范围

本 ADR 决定：
- 创建统一错误基类 `SddError`
- 定义错误码枚举 `ErrorCode`
- 提供错误处理工具函数
- 各模块使用统一错误类型

---

## 2. 决策

### 2.1 方案选择

我们决定采用 **统一错误类层次** 方案：

**创建 `src/errors.ts` 定义错误基类和子类，所有模块使用统一错误类型**。

### 2.2 设计原则

1. **统一基类**: 所有错误继承自 `SddError`
2. **错误码分类**: 按模块分配错误码范围
3. **上下文支持**: 错误包含上下文信息
4. **工具函数**: 提供统一的处理和格式化函数

### 2.3 错误码设计

```typescript
export enum ErrorCode {
  // State Errors (1000-1999)
  STATE_NOT_FOUND = 'STATE_1001',
  STATE_INVALID_TRANSITION = 'STATE_1002',
  STATE_FILE_MISSING = 'STATE_1003',
  STATE_DEPENDENCY_BLOCKED = 'STATE_1004',
  
  // Discovery Errors (2000-2999)

  DISCOVERY_TIMEOUT = 'DISCOVERY_2002',
  DISCOVERY_INVALID_CONTEXT = 'DISCOVERY_2003',

  TOOL_VALIDATION_ERROR = 'TOOL_3003',

```

### 2.4 错误类层次

```typescript
// 错误上下文接口
export interface ErrorContext {
  featureId?: string;
  filePath?: string;
  agentName?: string;
  stepId?: string;
  [key: string]: unknown;
}

// 统一错误基类
export class SddError extends Error {
  constructor(
    message: string,
    public readonly code: ErrorCode,
    public readonly context?: ErrorContext,
    public readonly cause?: Error
  ) {
    super(message);
    this.name = 'SddError';
  }

  toJSON() {
    return {
      name: this.name,
      code: this.code,
      message: this.message,
      context: this.context,
      cause: this.cause?.message
    };
  }
}

// State 错误
export class StateError extends SddError {
  constructor(message: string, code: ErrorCode, context?: ErrorContext, cause?: Error) {
    super(message, code, context, cause);
    this.name = 'StateError';
  }
}

// Discovery 错误
export class DiscoveryError extends SddError {
  constructor(message: string, code: ErrorCode, context?: ErrorContext, cause?: Error) {
    super(message, code, context, cause);
    this.name = 'DiscoveryError';
  }
}

// Tool 错误
export class ToolError extends SddError {
  constructor(message: string, code: ErrorCode, context?: ErrorContext, cause?: Error) {
    super(message, code, context, cause);
    this.name = 'ToolError';
  }
}

// Agent 错误
export class AgentError extends SddError {
  constructor(message: string, code: ErrorCode, context?: ErrorContext, cause?: Error) {
    super(message, code, context, cause);
    this.name = 'AgentError';
  }
}
```

### 2.5 错误处理工具函数

```typescript
// 统一错误包装
export function handleError(error: unknown, context?: ErrorContext): SddError {
  if (error instanceof SddError) {
    return error;
  }
  
  if (error instanceof Error) {
    return new SddError(
      error.message,
      ErrorCode.INTERNAL_ERROR,
      context,
      error
    );
  }
  
  return new SddError(
    String(error),
    ErrorCode.INTERNAL_ERROR,
    context
  );
}

// 错误格式化
export function formatErrorMessage(error: SddError): string {
  return `[${error.code}] ${error.name}: ${error.message}`;
}

// 统一日志
export function logError(error: SddError, level: 'info' | 'warn' | 'error' = 'error'): void {
  const message = formatErrorMessage(error);
  console[level](message);
}
```

### 2.6 使用示例

**旧方式**:
```typescript
// state/machine.ts
if (!state) {
  throw new Error('State not found');
}

// discovery/workflow-engine.ts
console.error('Step failed:', stepId);
return { success: false, error: 'Step failed' };
```

**新方式**:
```typescript
// state/machine.ts
import { StateError, ErrorCode } from './errors';

if (!state) {
  throw new StateError(
    'State not found',
    ErrorCode.STATE_NOT_FOUND,
    { featureId }
  );
}

// discovery/workflow-engine.ts
import { DiscoveryError, ErrorCode } from './errors';

throw new DiscoveryError(
  `Step ${stepId} failed`,
  ErrorCode.DISCOVERY_STEP_FAILED,
  { stepId, featureId: context.featureId }
);
```

## 3. 后果

### 3.1 正面影响

| 影响 | 说明 |
|------|------|
| ✅ 统一错误处理 | 所有模块使用一致的错误类型 |
| ✅ 错误分类清晰 | 通过错误码快速定位问题来源 |
| ✅ 日志格式一致 | 便于日志分析和监控 |
| ✅ 上下文完整 | 错误包含足够的调试信息 |
| ✅ 便于测试 | 可针对特定错误类型编写测试 |

### 3.2 负面影响

| 影响 | 说明 | 缓解措施 |
|------|------|----------|
| ⚠️ 迁移成本 | 现有代码需要更新 | 渐进式迁移，保持向后兼容 |
| ⚠️ 代码量增加 | 需要定义多个错误类 | 控制在 150 行以内 |
| ⚠️ 学习成本 | 新开发者需要熟悉错误体系 | 编写使用文档 |

### 3.3 技术影响

- **编译时间**: 无明显影响
- **运行时性能**: 轻微增加（错误对象创建），可忽略
- **包大小**: 增加约 3KB（压缩后）
- **向后兼容**: 完全兼容（原有 Error 仍然可用）

## 4. 替代方案

### 方案 A: 使用现有错误库（已否决）

**描述**: 使用 `ts-custom-error` 或 `neverthrow` 等第三方库。

**优点**:
- 功能成熟
- 社区支持

**缺点**:
- 增加依赖
- 学习成本
- 过度设计

**否决理由**: SDD 错误处理需求简单，无需第三方库。

### 方案 B: 保持现状（已否决）

**描述**: 不做任何改变，维持现有错误处理方式。

**优点**:
- 无变更成本

**缺点**:
- 问题持续存在
- 不利于错误追踪

**否决理由**: 不解决问题，影响可维护性。
