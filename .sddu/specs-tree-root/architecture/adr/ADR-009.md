## 1. 背景

### 1.1 问题陈述

规范 v2.0.0 要求 **FR-004: 系统必须在状态变更前检查依赖状态**，但当前实现中：

1. **依赖关系图已存在**: `MultiFeatureManager` 已实现依赖图构建
2. **缺少状态级别检查**: 只检查依赖是否存在，不检查依赖状态
3. **缺少自动验证**: 状态变更前不会自动检查依赖

### 1.2 决策范围

本 ADR 决定：
- 依赖检查器的实现策略
- 检查规则（状态前进 vs 回退）
- 与现有依赖图的集成方式
- 错误提示和阻塞机制

### 1.3 现状分析

**已有功能** (来自 sdd-multi-module):
- ✅ 依赖关系图构建 (`buildDependencyGraph`)
- ✅ 循环依赖检测 (`detectCircularDependency`)
- ✅ 依赖就绪检查 (`isDependencyReady`)

**缺失功能**:
- ❌ 状态级别依赖检查（依赖 Feature 的状态 ≥ 当前状态）
- ❌ 状态回退警告
- ❌ 阻塞 Feature 列表生成

---

## 2. 方案对比

### 2.1 方案 A：增强现有依赖检查（推荐）⭐

**描述**：
在现有 `MultiFeatureManager` 的依赖检查基础上，新增状态级别的验证。

**架构**:
```
状态变更前
    │
    ▼
┌─────────────────┐
│ 获取依赖列表    │ (已有)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 检查依赖状态    │ (新增)
│ ≥ 目标状态？    │
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
   是        否
    │         │
    ▼         ▼
  允许     阻止 + 提示
```

**优点**:
- ✅ 复用现有代码：依赖图构建逻辑不变
- ✅ 实现简单：只新增状态比较逻辑
- ✅ 向后兼容：不影响现有功能

**缺点**:
- ❌ 需要修改 `MultiFeatureManager` 接口
- ❌ 性能影响：每次状态变更需要查询多个 state.json

**实现复杂度**: 低

---

### 2.2 方案 B：独立依赖检查器

**描述**：
创建独立的 `DependencyChecker` 类，专门负责状态级别依赖检查。

**架构**:
```
DependencyChecker
├── check(featureId, targetState)
├── getBlockingFeatures(featureId)
└── getDependencyStatus(featureId)
```

**优点**:
- ✅ 职责单一：专注依赖检查
- ✅ 不影响现有代码：独立模块
- ✅ 易于测试：可以独立单元测试

**缺点**:
- ❌ 代码重复：需要重新实现部分依赖图逻辑
- ❌ 数据同步：需要与 `MultiFeatureManager` 保持一致

**实现复杂度**: 中

---

### 2.3 方案 C：混合模式（推荐）

**描述**：
方案 A + 方案 B 结合：
- 依赖图构建：复用 `MultiFeatureManager`
- 状态检查：新增 `DependencyChecker` 封装

**架构**:
```typescript
class DependencyChecker {
  constructor(
    private multiFeatureManager: MultiFeatureManager
  ) {}
  
  async check(featureId: string, targetState: string) {
    // 使用 MultiFeatureManager 获取依赖图
    const graph = await this.multiFeatureManager.getDependencyGraph();
    
    // 获取当前 Feature 的依赖
    const dependencies = graph.dependencies[featureId] || [];
    
    // 检查每个依赖的状态
    for (const depId of dependencies) {
      const depState = await this.multiFeatureManager.getState(depId);
      
      if (!this.isStateReady(depState.status, targetState)) {
        return {
          passed: false,
          message: `依赖 ${depId} 的状态 ${depState.status} 未就绪`,
          blockingFeatures: [depId]
        };
      }
    }
    
    return { passed: true };
  }
  
  private isStateReady(currentState: string, targetState: string): boolean {
    const order = ['specified', 'planned', 'tasked', 'implementing', 'reviewed', 'validated'];
    return order.indexOf(currentState) >= order.indexOf(targetState);
  }
}
```

**优点**:
- ✅ 复用依赖图：避免重复实现
- ✅ 职责分离：检查逻辑独立封装
- ✅ 易于扩展：未来可以添加更多检查规则

**缺点**:
- ❌ 实现稍复杂：需要协调两个模块
- ❌ 性能开销：额外的封装层

**实现复杂度**: 中低

## 3. 决策

### 3.1 选定方案

**决定采用：方案 C（混合模式）**

### 3.2 决策理由

1. **复用优先**: 复用 `MultiFeatureManager` 的依赖图构建
2. **职责分离**: `DependencyChecker` 专注状态检查逻辑
3. **易于扩展**: 未来可以添加更多检查规则（如循环依赖、阻塞检测）
4. **向后兼容**: 不影响现有 `MultiFeatureManager` 的 API

### 3.3 检查规则

#### 3.3.1 状态前进检查

```typescript
// 规则：依赖 Feature 的状态必须 ≥ 目标状态
interface DependencyCheckResult {
  passed: boolean;
  message?: string;
  blockingFeatures?: string[];
  dependencies?: Array<{
    featureId: string;
    currentStatus: string;
    requiredStatus: string;
    satisfied: boolean;
  }>;
}

async function checkDependencyForAdvance(
  featureId: string,
  targetState: string
): Promise<DependencyCheckResult> {
  const graph = await multiFeatureManager.getDependencyGraph();
  const dependencies = graph.dependencies[featureId] || [];
  
  if (dependencies.length === 0) {
    return { passed: true };
  }
  
  const results = [];
  const blockingFeatures = [];
  
  for (const depId of dependencies) {
    const depState = await multiFeatureManager.getState(depId);
    const satisfied = isStateAtLeast(depState.status, targetState);
    
    results.push({
      featureId: depId,
      currentStatus: depState.status,
      requiredStatus: targetState,
      satisfied
    });
    
    if (!satisfied) {
      blockingFeatures.push(depId);
    }
  }
  
  if (blockingFeatures.length > 0) {
    return {
      passed: false,
      message: `以下依赖未就绪：${blockingFeatures.join(', ')}`,
      blockingFeatures,
      dependencies: results
    };
  }
  
  return { passed: true, dependencies: results };
}

function isStateAtLeast(currentState: string, targetState: string): boolean {
  const order = ['specified', 'planned', 'tasked', 'implementing', 'reviewed', 'validated'];
  return order.indexOf(currentState) >= order.indexOf(targetState);
}
```

#### 3.3.2 状态回退警告

```typescript
// 规则：状态回退时警告，检查被依赖 Feature
async function checkDependencyForRegression(
  featureId: string,
  targetState: string
): Promise<DependencyCheckResult> {
  const graph = await multiFeatureManager.getDependencyGraph();
  
  // 获取被当前 Feature 阻塞的 Feature（依赖于当前 Feature）
  const blockedFeatures = graph.blockedBy[featureId] || [];
  
  if (blockedFeatures.length === 0) {
    return { passed: true };
  }
  
  const warnings = [];
  
  for (const blockedId of blockedFeatures) {
    const blockedState = await multiFeatureManager.getState(blockedId);
    
    // 如果被阻塞的 Feature 状态已经超过当前目标状态，警告
    if (isStateBeyond(blockedState.status, targetState)) {
      warnings.push({
        featureId: blockedId,
        currentStatus: blockedState.status,
        message: `回退可能导致 ${blockedId} 依赖不满足`
      });
    }
  }
  
  if (warnings.length > 0) {
    return {
      passed: true, // 回退允许，但警告
      message: '状态回退可能影响依赖 Feature',
      warnings
    };
  }
  
  return { passed: true };
}

function isStateBeyond(currentState: string, targetState: string): boolean {
  const order = ['specified', 'planned', 'tasked', 'implementing', 'reviewed', 'validated'];
  return order.indexOf(currentState) > order.indexOf(targetState);
}
```

#### 3.3.3 循环依赖检测

```typescript
// 复用现有的循环依赖检测
async function checkCircularDependency(featureId: string): Promise<{
  hasCircular: boolean;
  cycle?: string[];
}> {
  const graph = await multiFeatureManager.getDependencyGraph();
  const cycle = detectCircularDependency(featureId, graph.dependencies);
  
  if (cycle) {
    return {
      hasCircular: true,
      cycle
    };
  }
  
  return { hasCircular: false };
}
```

### 3.4 完整检查流程

```typescript
class DependencyChecker {
  async check(
    featureId: string,
    targetState: string,
    isRegression: boolean
  ): Promise<DependencyCheckResult> {
    // 1. 循环依赖检测
    const circular = await this.checkCircularDependency(featureId);
    if (circular.hasCircular) {
      return {
        passed: false,
        message: `检测到循环依赖：${circular.cycle!.join(' → ')}`
      };
    }
    
    // 2. 根据前进/回退选择检查策略
    if (isRegression) {
      return await this.checkForRegression(featureId, targetState);
    } else {
      return await this.checkForAdvance(featureId, targetState);
    }
  }
}
```

## 4. 后果

### 4.1 积极影响

- ✅ 依赖状态保证：防止依赖未就绪时推进
- ✅ 错误提示清晰：明确说明哪些依赖未满足
- ✅ 循环依赖检测：防止循环依赖问题
- ✅ 回退警告：避免破坏依赖关系

### 4.2 消极影响

- ⚠️ 性能影响：每次状态变更需要查询多个 state.json
- ⚠️ 实现复杂：需要协调多个模块
- ⚠️ 严格限制：可能阻碍某些特殊场景

### 4.3 缓解措施

1. **缓存优化**: 缓存依赖图，定期刷新
2. **并行查询**: 并行查询多个依赖状态
3. **超时保护**: 查询超时返回部分结果
4. **手动覆盖**: 提供强制推进选项（需确认）

### 4.4 需要遵循的约定

1. **前进检查**: 状态前进时必须通过依赖检查
2. **回退警告**: 状态回退时警告但不阻止
3. **循环检测**: 检测到循环依赖时阻止
4. **错误提示**: 提供明确的依赖信息和解决建议

## 5. 验证策略

| 验证项 | 验证方法 | 预期结果 |
|--------|----------|----------|
| 前进检查生效 | 依赖未就绪时推进状态 | 被阻止，显示依赖信息 |
| 回退警告生效 | 状态回退时检查 | 显示警告，但允许 |
| 循环依赖检测 | 创建循环依赖 | 被阻止，显示循环路径 |
| 错误提示清晰 | 检查错误消息 | 包含依赖 ID、状态、建议 |
| 性能可接受 | 100 个 Feature 测试 | 检查时间 < 200ms |
| 缓存生效 | 连续多次检查 | 第二次使用缓存 |
