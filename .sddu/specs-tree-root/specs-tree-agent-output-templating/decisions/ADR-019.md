# ADR-019: 模板变量格式为 `<<变量名>>` 并列出语义说明

## 状态
ACCEPTED

## 背景
**开放问题 OQ-004**: 是否需要在 Agent Prompt 中明确列出每个 `<<变量名>>` 的语义说明，还是让 AI 完全自主理解？

变量占位符格式也涉及一个选择：FS spec 中已明确使用 `<<变量名>>` 格式（而非 `{{变量名}}`），以此避免与 Handlebars 模板引擎语法冲突。在变量说明方面，存在两个子选项：

**子选项 A（详尽说明）**: 在 Agent Prompt 的模板引用指令中列出常见变量的语义说明。
**子选项 B（完全自主）**: 不在 Prompt 中说明变量含义，完全依赖 AI 通过变量名自解释。

## 决策
我们决定 **使用 `<<变量名>>` 格式**（与 Spec 一致），并在 Agent Prompt 中 **列出常见变量的语义说明**。

具体包括：
1. 模板占位符使用 `<<变量名>>` 格式
2. 变量名支持中英文混合（如 `<<核心问题>>` 或 `<<core_problem>>`）
3. Agent Prompt 的「输出模板」章节中包含常见变量的语义示例说明
4. 变量说明示例（非穷举）：
   ```
   常见变量说明：
   - <<feature_name>>: 当前 Feature 的名称
   - <<status>>: 当前阶段的状态
   - <<file_path>>: 生成文件的相对路径
   - <<next_step>>: 推荐的下一个操作步骤
   ```
5. 模板文件本身只包含变量占位符，不包含变量说明（说明只在 Prompt 中）

## 理由
1. **避免 Handlebars 冲突**: `<<变量名>>` 不会被 Handlebars 模板引擎解析，确保 `build-agents.cjs` 构建时 output/ 目录文件保持原样
2. **AI 解析友好**: `<<变量名>>` 在视觉上明显且语义清晰，AI 更容易识别和替换
3. **降低解析歧义**: 列出语义说明可以减少 AI 对不同变量含义的理解误差
4. **跨 LLM 兼容性好**: 不依赖特定 LLM 的模板解析能力，使用通用的占位符约定

## 后果
### 正面
- 变量占位符与 Handlebars 语法天然隔离，不需要在构建脚本中做额外转义
- AI 解析质量更高，输出一致性更好
- 模板文件作为纯文本，用户自定义时只需关注内容，无需理解模板引擎

### 负面
- Prompt 指令长度增加（但增加量极小，~10 行）
- 变量说明列表需要随模板变更同步更新（但频率低，可维护）
