# ADR-018: Agent 输出模板解析 — AI-Side 方案

## 状态
ACCEPTED

## 背景
SDDU Agent 输出模板化功能需要确定模板的发现和解析机制。有两个可选路径：

**开放问题 OQ-001**: 模板发现路径查找应在 Agent Prompt（AI 端）实现，还是在插件代码（Node.js 端）实现？

**方案 A（AI-Side）**: Agent Prompt 中包含模板路径、优先级规则和变量替换指令。AI Agent 在运行阶段自主读取模板文件，理解其结构，并按照模板格式生成输出。

**方案 B（Code-Side）**: 新增 Node.js 模块（如 `loadTemplate.js`），在插件初始化时加载模板，通过某种机制注入到 Agent 上下文。

## 决策
我们决定采用 **AI-Side 方案（方案 A）**。

具体决策内容包括：
1. 模板发现和解析完全由 AI Agent 在运行时完成
2. 不新增任何 Node.js 模块
3. Agent Prompt 中明确说明：
   - 模板文件的查找路径和优先级
   - `<<变量名>>` 占位符的替换规则
   - 模板不存在时的回退行为（使用 hardcoded 格式）
4. 内置模板文件放在 `src/templates/agents/output/` 目录，作为纯 Markdown 文本
5. 构建/打包/安装脚本只负责将模板文件复制到目录，不做任何内容处理

## 理由
1. **与 Spec 一致**: Feature Specification 的 Non-Goals 明确「不引入独立的模板渲染引擎（依赖 AI 自主解析）」
2. **架构极简**: 只需创建模板文件 + 修改 Agent Prompt，无需新增 JS 模块、修改 opencode.json 或插件入口
3. **零运行时依赖**: 无需引入新的 npm 包
4. **语义灵活性**: AI 天然理解自然语言占位符（`<<核心问题>>`），无需预定义变量类型
5. **可扩展性**: 未来如需增加校验工具或缓存，可在不改变核心架构的前提下增量添加

## 后果
### 正面
- 维护成本低，基础设施变更最小化
- 不需要修改 opencode.json 配置
- 模板文件是纯文本，修改模板无需重新构建插件

### 负面
- AI Agent 对模板的遵循度取决于 Prompt 指令质量
- 不同 LLM 对 `<<变量名>>` 占位符的解析行为可能有细微差异
- 无法在代码层面做模板校验（留作未来 FR-014）

### 缓解措施
- Agent Prompt 中增加明确的 AI 指令规范，要求 AI 必须替换所有 `<<变量名>>`
- 在模板引用指令末尾强调：输出中不允许出现未替换的 `<<原始占位符>>`
- 内置模板作为安全基线，用户自定义模板出问题时自动回退

## 关联
- **FR-005**: 模板发现路径优先级逻辑（在 Agent Prompt 中定义）
- **FR-006**: AI 自主解析 `<<变量名>>` 占位符
- **NFR-004**: 模板安全性 — 纯文本无注入风险
- **NFR-001**: 模板加载性能 < 50ms — 文件读取在 AI 端以同步方式完成
