# 需求挖掘报告：Agent 输出模板化系统

## 1. 问题定义

### 核心问题
目前 6 个主流程 SDDU Agent（discovery/spec/plan/tasks/build/review/validate）的输出格式**硬编码在各自的 Agent Prompt 中**。修改输出格式需要直接修改 Agent 定义文件，无法灵活适配不同团队、不同项目的输出标准需求。

### 业务价值
- **提效**：模板化后不同团队可快速适配统一的输出标准
- **体验**：用户可根据自身偏好定制 Agent 输出格式
- **可维护性**：输出格式与 Agent Prompt 解耦，降低维护成本
- **生态扩展**：第三方团队可通过自定义模板修改输出格式，无需修改插件源码

### 不做的成本
- 每次需要调整输出格式都必须修改 Agent Prompt 文件，牵一发而动全身
- 不同项目/团队无法差异化输出标准
- 插件生态的灵活性受限

## 2. 用户画像

### 主要用户
| 角色 | 职责 | 与模板系统的关系 |
|------|------|-----------------|
| **普通开发者** | 日常使用 SDDU Agent 工作流 | **消费者**：使用内置默认模板或团队自定义模板的输出 |
| **团队 TL/架构师** | 制定团队输出标准和规范 | **定制者**：编写和维护团队自定义模板 |
| **SDDU 插件作者** | 维护 SDDU 插件 | **提供者**：提供内置默认模板，维护模板基础设施 |

### 用户场景

**场景一：团队统一输出格式**
> 一个前端团队使用 SDDU 进行功能开发，团队 TL 希望所有 Feature 的 spec.md 都包含「性能指标」和「UI 截图」两个 section。TL 编写自定义模板放到 `.sddu/templates/agents/sddu-spec.md.hbs`，此后团队所有成员的 spec 输出都自动遵循该格式。

**场景二：跨项目差异化**
> 公司在 A 项目中使用简洁版输出格式，B 项目中使用详细版输出格式。通过在不同项目的 `.sddu/templates/` 中放置不同的模板文件实现差异化。

**场景三：插件维护升级**
> SDDU 插件发布新版本时修改了内置默认模板，用户项目使用自定义模板覆盖，插件升级不影响用户的模板配置。

## 3. 需求清单

### Must Have（必备）
- [ ] M-01：6 个主流程 Agent 的输出模板化（内置默认模板）
- [ ] M-02：模板文件与 Agent Prompt 分离（`src/templates/agents/` 下独立子目录存放）
- [ ] M-03：用户自定义模板覆盖机制（`.sddu/templates/agents/`）
- [ ] M-04：AI 自主解析 `{{变量名}}` 占位符（模板变量无需预定义，Agent 运行时自动填充）
- [ ] M-05：安装阶段将内置模板复制到 `.opencode/plugins/sddu/` 对应目录
- [ ] M-06：模板语法错误显式报错（不静默降级，提示用户定位问题）
- [ ] M-07：与现有 6 阶段工作流无缝集成（Agent 执行后自动按模板渲染输出）

### Should Have（期望）
- [ ] S-01：内置模板保留当前 Agent Prompt 中定义的输出格式（不重新设计）
- [ ] S-02：模板文件命名与现有 `.hbs` 文件命名对齐（如 `sddu-discovery.md.hbs`）
- [ ] S-03：模板发现路径优先级明确：用户自定义 > 插件内置

### Could Have（惊喜）
- [ ] C-01：提供模板校验工具/命令，用户可提前验证模板正确性
- [ ] C-02：多套内置模板风格（简洁版/详细版等）
- [ ] C-03：模板版本管理（与 Agent 版本对应）

### Won't Have（明确不做）
- [ ] W-01：不支持 Section 级模板（以整个文件为模板单位）
- [ ] W-02：不涉及 Agent Prompt 模板系统（`src/templates/agents/*.hbs` 是提示词模板，与本功能独立）

## 4. 推荐方案

### 方案概述
**三层模板体系 + AI 自主解析变量**：

```
┌────────────────────────────────────────────────────┐
│ 开发阶段 (插件源码)                                 │
│ src/templates/agents/output/                       │
│   ├── sddu-discovery.md.hbs                        │
│   ├── sddu-spec.md.hbs                             │
│   ├── sddu-plan.md.hbs                             │
│   ├── sddu-tasks.md.hbs                            │
│   ├── sddu-build.md.hbs                            │
│   ├── sddu-review.md.hbs                           │
│   ├── sddu-validate.md.hbs                         │
│   └── (与 src/templates/agents/ 原有提示词模板分离) │
├────────────────────────────────────────────────────┤
│ 安装阶段 (插件包)                                   │
│ .opencode/plugins/sddu/templates/output/            │
│   └── ... (同上，构建时从 src 复制)                  │
├────────────────────────────────────────────────────┤
│ 运行阶段 (用户项目)                                  │
│ .sddu/templates/agents/ ← 用户自定义覆盖 (优先级高) │
│   └── ...                                          │
└────────────────────────────────────────────────────┘
```

**模板触发流程**：
1. Agent 开始执行 → 加载 Agent Prompt（含输出模板引用说明）
2. 根据模板发现机制加载模板文件（用户自定义优先）
3. 模板中的 `{{变量名}}` 占位符由 Agent 在输出时自主解析填充
4. Agent 按模板结构生成最终输出文件（discovery.md / spec.md 等）

### 选择理由
- **与现有架构兼容**：Handlebars 语法与现有 `.hbs` 模板一致
- **AI 自主解析 vs 模板引擎**：避免引入复杂的模板渲染引擎，利用 AI 的语义理解能力自动填充变量
- **三层分离**：开发/安装/运行各司其职，互不干扰
- **最小变更**：内置模板照搬现有格式，用户无感迁移

### 差异化优势
- ✅ 无需预定义变量集——AI 智能解析 `{{xxx}}`，天然支持任意自定义变量
- ✅ 无需引入第三方模板引擎——Agent 自身就是渲染器
- ✅ 与现有工作流零侵入——仅在 Agent 输出时引入模板，不改变 Agent 执行逻辑

## 5. 成功指标

### 定性指标
- 用户放置自定义模板到 `.sddu/templates/agents/` 后，Agent 输出自动使用该模板格式
- 模板语法错误可给出明确错误信息，帮助用户定位问题
- 内置模板开箱即用，用户无需任何额外配置

### 定量指标
| 指标 | 目标 | 说明 |
|------|------|------|
| 模板覆盖率 | 6/6 个 Agent | 所有主流程 Agent 均有内置模板 |
| 自定义生效成功率 | 100% | 放置后即时生效 |
| 内置模板格式保留度 | 100% | 与当前硬编码格式完全一致 |
| 错误捕获率 | 100% | 模板语法错误/缺失必须报错 |

## 6. 范围边界

### MVP 范围（最小完整闭环）
- 6 个主流程 Agent 的内置默认模板（放在 `src/templates/agents/output/`）
- 用户自定义模板覆盖机制（`.sddu/templates/agents/`）
- 模板发现路径优先级逻辑（用户自定义 > 插件内置）
- AI 自主解析 `{{变量名}}` 占位符
- 与现有 6 阶段工作流集成
- 安装脚本适配（复制模板到 `.opencode/plugins/`）
- 模板语法错误显式报错

### V1 范围
- (在 MVP 基础上完善后即为完整可用版本)

### 明确不做
- ❌ Section 级模板（仅支持整个文件为单位）
- ❌ 独立的模板渲染引擎（依赖 AI 自主解析）
- ❌ 模板校验工具（留作未来增强）
- ❌ 多套内置模板风格（先做一套默认）
- ❌ docs/roadmap/help 等辅助 Agent 的输出模板化

## 7. 风险与假设

### 关键假设
- **A-01**：Agent（LLM）能够准确理解并填充 `{{xxx}}` 占位符的语义
- **A-02**：用户自定义模板不会引入结构性错误（如缺少关键 section）
- **A-03**：模板文件路径和命名约定对所有用户都足够直观

### 主要风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| AI 对 `{{xxx}}` 解析不稳定或不一致 | 中 | 高 | 在 Agent Prompt 中明确模板规则；内置模板提供清晰的占位符命名 |
| 用户自定义模板结构差异过大导致 Agent 输出质量下降 | 中 | 中 | 内置模板作为安全基线；为用户提供模板编写指南 |
| 安装脚本复制逻辑复杂（区分开发/安装/运行三阶段） | 中 | 中 | 清晰的目录职责划分；安装脚本的单元测试 |
| 与现有 `src/templates/agents/` 路径冲突 | 低 | 高 | 使用独立子目录 `output/` 彻底分离 |

### 验证计划
1. **MVP 验证**：用一个 Agent（如 discovery）做 POC，验证 AI 能否正确解析模板占位符
2. **集成验证**：验证自定义模板覆盖后 Agent 输出是否符合预期
3. **错误注入测试**：故意放错误模板，验证报错机制是否正常工作
4. **安装测试**：验证 build → package → install 完整流程中模板文件正确复制
