## 1. 问题定义

### 1.1 核心问题

> 💡 **核心根因**: P-001（插件缺少分而治之的拆分能力）是本质问题，P-002 和 P-003 是其表象。做好 Feature 拆分，2 和 3 自然迎刃而解。

| 问题 ID | 问题描述 | 用户原话 | 与根因的关系 |
|---------|----------|----------|--------------|
| **P-001** | 插件默认只生成平层目录，不具备 Feature 拆分能力 | "插件不支持拆分 feature，就会造成生成的 feature 巨大" | 🔴 **核心根因** |
| **P-002** | root 缺少顶层规划文档 | "root 没有自己的 spec/plan，作为一个 specs 不完整" | 🟡 表象（因 root 无法拆分为子 Feature 管理） |
| **P-003** | Feature 水平平铺，非树形结构 | "Feature 平级放置，无法表达父子关系" | 🟡 表象（因缺少子 Feature 嵌套机制） |

### 1.2 问题因果链

```
P-001: 插件缺少分而治之的拆分能力 ←── 核心根因
  │
  ├──→ P-002: root 缺少顶层规划（因插件不支持轻量化父级规范）
  │
  └──→ P-003: Feature 水平平铺（因插件不支持子 Feature 嵌套机制）
```

### 1.3 根因深度分析（5 Whys）

**为什么插件缺少分而治之的能力？**
1. Why: 因为 Agent 总是生成完整的 6 阶段文档，不会区分父级和叶子
2. Why: 因为插件不支持父子 Feature 的目录嵌套机制
3. Why: 因为目录扫描只扫描一级 `specs-tree-*`，不递归扫描子目录
4. Why: 因为早期设计关注"多模块并行开发"，而非"层次化拆分管理"
5. **根因**: 插件架构设计时未定义 Feature 拆分粒度和父子关系模型

### 1.4 不做的成本

| 维度 | 影响 |
|------|------|
| **可维护性** | 随着需求复杂度增长，单个 Feature 内容无限膨胀 |
| **协作效率** | 多个 Agent 同时操作同一层级时缺少隔离 |
| **新成员上手** | 缺少顶层规范，新人无法快速理解整体架构 |
| **状态管理** | 当前 state.json 格式不统一，无父子状态管理机制 |
| **Roadmap 执行** | v2.4.0 是 v2.5.0~v2.8.0 的前置依赖 |

### 1.5 业务价值（围绕核心根因 P-001）

| 价值类型 | 具体描述 | 衡量方式 |
|----------|----------|----------|
| **架构合理性** | 分而治之符合高内聚低耦合原则，每个 Feature 关注点单一 | 单一 Feature 职责数 = 1 |
| **可维护性** | 单一职责的 specs 降低认知负担，修改影响范围可控 | 变更影响范围 ≤ 1 个 specs 目录 |
| **协作效率** | 多 Agent 可并行操作不同子 Feature，减少锁冲突 | 并行操作冲突率 = 0 |
| **可扩展性** | 树形结构类比文件系统，支持无限层级嵌套 | 嵌套深度 = 无限制 |

## 2. 用户画像

### 2.1 主要用户

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| **SDDU 开发者**（维护插件源码） | 开发和维护 SDDU 工作流插件 | 插件支持树形结构管理 |
| **架构师**（20%） | 设计整体架构，维护顶层规范 | 产品级视图，快速了解全局 |
| **开发工程师**（80%） | 实现具体 Feature | 清晰的父子关系，独立开发空间 |
| **路由 Agent**（@sddu） | 解析 Discovery 结果并分发任务 | 自动识别树形结构并路由 |

### 2.2 用户场景还原

**场景 A: Discovery/Spec 阶段自动拆分建议**
- 当前: Agent 直接生成一个包含所有内容的巨大 Feature
- 期望: 在 Discovery 或 Spec 阶段识别到需求包含多个独立模块（如前端/后端），输出拆分建议，用户确认后按树形结构生成

**场景 A-1: 不存在一次性生成父子文档**
- 当前: 用户期望 Agent 一次性创建好所有父子目录和文档
- 期望: 每次执行只针对指定 Feature，先生成父级文档（含拆分建议），后续由用户或路由 Agent 逐步创建子级

**场景 B: 父级文档作为子级上下文**
- 当前: 所有细节混在一个 spec 中
- 期望: 父级 spec 描述整体架构，子级 spec 描述具体实现，子级生成时以父级文档为参考输入

**场景 C: 混合状态管理**
- 当前: 只能看到单个 Feature 的状态
- 期望: 父级同时展示自身状态和每个子级的独立状态，手动触发扫描更新

### 2.3 用户旅程地图

```
当前旅程（痛点标注 🔴）:
用户调用 @sddu-discovery → 调用 @sddu-spec → 生成一个巨大的平级 Feature 🔴
  ├── 所有内容混在一个 spec 中 🔴 难以定位
  ├── 无法表达父子关系 🔴
  └── 状态只能看到单个维度 🔴

目标旅程:
用户调用 @sddu-discovery → Agent 识别多模块 → 输出拆分建议
  ├── 用户确认 → 生成父级目录（含拆分建议文档）
  ├── 后续逐步调用 @sddu-spec 创建子级目录
  ├── 父级生成 discovery/spec/plan（无 tasks/build）
  ├── 子级生成完整 6 阶段文档
  ├── 子级 state.json 独立管理自身状态
  └── 父级手动触发扫描，更新自身 + 各子级状态
```

## 4. 推荐方案

### 4.1 方案概述

**轻量化父级 + 完整叶子 + 树形嵌套 + 混合状态 + 向后兼容**

```
specs-tree-root/                          # ← 轻量化父级：仅 discovery/spec/plan
├── discovery.md                          # ← 为什么拆分、拆成什么
├── spec.md                               # ← 父级自身业务/技术架构
├── plan.md                               # ← 子级协作关系和技术规划
├── README.md                             # ← 目录导航
├── state.json                            # ← 自身状态 + 每个子级状态
├── architecture/                         # ← 保留：ADR 文档
├── specs-tree-plugin-rename-sddu/        # ← 保留：原有 11 个 Feature
├── ...                                   # ← 保留
├── specs-tree-[parent-feature]/          # ← 新增：可能也是轻量化父级
│   ├── discovery.md                      # ← 为什么拆分子级
│   ├── spec.md                           # ← 父级自身架构
│   ├── plan.md                           # ← 子级协作关系
│   ├── state.json                        # ← 自身状态 + 子级状态
│   ├── specs-tree-[sub-feature]/         # ← 完整叶子：走完整 6 阶段
│   │   ├── discovery.md
│   │   ├── spec.md
│   │   ├── plan.md
│   │   ├── tasks.md
│   │   ├── build.md
│   │   ├── review.md
│   │   ├── validate.md
│   │   └── state.json                    # ← dependencies.on 可直接引用跨子树路径
│   └── specs-tree-[sub-feature-2]/       # ← 另一个完整叶子
│       └── ...
```

### 4.2 选择理由

1. **全新设计**: 按树形结构从零设计，不受历史包袱限制
2. **渐进式**: 新功能不影响现有历史工作空间
3. **根因解决**: 三个核心问题一次性解决
4. **架构一致**: 与 ADR-001（分布式存储）的设计原则一致

### 4.3 差异化优势

| 对比维度 | 传统方案 | 本方案 |
|----------|----------|--------|
| 历史兼容 | 需要处理旧版结构 | 零历史包袱，全新设计 |
| 状态管理 | 全量扫描 | 手动扫描 + 独立字段 |
| 依赖检查 | 同层依赖 | 跨子树依赖（完整路径引用） |
| 嵌套深度 | 1 层 | 无限（类比文件系统） |

## 7. 风险与假设

### 7.1 关键假设

| 假设 | 验证方式 | 风险等级 |
|------|----------|----------|
| Discovery/Spec 阶段能准确识别拆分时机 | 观察 Agent 输出 | 🟡 中 |
| 无限嵌套在真实场景中有合理深度 | 观察实际使用 | 🟢 低 |
| 手动扫描更新体验可接受 | 用户反馈 | 🟡 中 |

### 7.2 主要风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Agent 拆分建议不准确 | 用户手动调整 | 中 | 提供用户指定能力兜底 |
| 跨子树依赖管理复杂 | 用户理解成本高 | 低 | 提供清晰文档和示例 |
| 新 schema 与现有不完全兼容 | 数据迁移 | 低 | 不需要迁移旧版 |

### 7.3 验证计划

| 验证项 | 方法 | 通过标准 |
|--------|------|----------|
| 树形结构识别 | 创建嵌套目录 | 插件正确读取 |
| 父/叶文档生成 | Agent 创建父/叶 | 文档类型正确 |
| 混合状态管理 | 手动扫描更新 | 父级正确展示自身 + 各子级状态 |
| 跨子树依赖 | 创建跨树依赖 | 检查通过 |

## 8. 现有架构约束

基于已有 ADR 的约束：

| ADR | 决策 | 对 v2.4.0 的影响 |
|-----|------|-----------------|
| **ADR-001** | 完全分布式存储 | ✅ 每个 Feature 独立 state.json，天然支持树形 |
| **ADR-005** | 7 步工作流固定顺序 | ⚠️ 仅叶子完整走 6 阶段，父级仅 discovery/spec/plan |
| **ADR-009** | 混合模式依赖检查 | ✅ 可增强为跨子树依赖检查 |
| **ADR-014** | 打包分发结构优化 | ⚠️ 树形结构需纳入打包范围 |

### 现有 state.json v2.1.0 Schema 设计

| 字段 | 类型 | 说明 |
|------|------|------|
| `selfStatus` | string | 父级自身状态 |
| `childrenStatuses` | object | 每个子级的独立状态（键为子 Feature 名，值为状态对象） |
| `parent` | string | 父级引用（统一字段名） |
| `depth` | number | 当前层级深度 |
| `dependencies.on` | array | 依赖列表（支持完整路径引用跨子树） |
| `phaseHistory` | array | 统一的历史记录字段 |
