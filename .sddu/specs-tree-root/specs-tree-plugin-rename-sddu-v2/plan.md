## 1. 规划概述

### 1.1 背景

V1 插件改名项目（FR-SDDU-RENAME-001）完成了从 SDD 到 SDDU 的品牌升级，但**代码中残留大量 SDD 字眼未彻底替换为 SDDU**。经过实际扫描发现：
- 模板文件中有 47+ 处 `@sdd-*` 引用
- 源码注释中有 "SDD Plugin" 描述
- 类型定义中仍有 `SddConfig`、`SddPhase` 等命名
- 测试文件中保留旧命名（应同步更新）

### 1.2 目标

1. **彻底清理**: 所有源码、模板、测试中的 SDD 字眼替换为 SDDU
2. **不做兼容**: V2 不保留任何向后兼容代码或描述
3. **品牌一致**: 确保代码、模板、文档、测试 100% 使用 SDDU 品牌
4. **降低混淆**: 避免新开发人员被 SDD/SDDU 混合命名搞混
5. **维护简化**: 统一命名减少认知负担

### 1.3 核心原则

**V2 核心原则**:
- ✅ **V2 不做任何 SDD 兼容** - 所有代码、模板、测试全部改为 SDDU
- ✅ **彻底清理** - 不留任何向后兼容描述或代码
- ✅ **一致使用** - 只使用 `@sddu-*`，不存在 `@sdd-*`
- ✅ **tests/* 同步更新** - 所有测试文件必须改为 Sddu* 命名

### 1.4 文件修改策略

**修改策略**:
```
src/ (源码 - 直接修改)
     ↓ 构建/安装时自动生成
.opencode/*, .sdd/*, .sddu/*, opencode.json (生成物 - ❌ 不修改)
dist/* (构建产物 - ❌ 不修改)
```

**分类说明**:

| 类别 | 路径 | 修改策略 | 示例 |
|------|------|----------|------|
| 源码 | `src/**/*.ts` | ✅ 直接修改 | `src/index.ts`, `src/types.ts` |
| 模板 | `src/templates/**/*.hbs` | ✅ 直接修改 | `src/templates/agents/*.hbs` |
| 配置模型 | `src/config/*.ts` | ✅ 直接修改 | `src/config/opencode-config.ts` |
| 测试文件 | `src/**/*.test.ts` | ✅ 直接修改 | `src/types.test.ts`, `src/errors.test.ts` |
| 文档 | `README.md`, `docs/*.md` | ✅ 直接修改 | 源码文档 |
| 脚本 | `install.sh`, `scripts/*` | ✅ 直接修改 | 安装和打包脚本 |
| 插件配置 | `.opencode/*` | ❌ 禁止修改 | 安装时自动生成 |
| 工作空间 | `.sdd/*`, `.sddu/*` | ❌ 禁止修改 | 运行时自动生成 |
| 根目录配置 | `opencode.json` | ❌ 禁止修改 | 安装时自动生成 |
| 构建产物 | `dist/*` | ❌ 禁止修改 | 构建时自动生成 |

**正确做法**:
- ✅ 直接修改 `src/config/opencode-config.ts` 配置模型
- ✅ 修改后运行完整构建流程：`npm run clean` → `npm install` → `npm run build` → `npm run package`
- ✅ 运行 E2E 脚本生成测试项目验证
- ✅ 如不符合预期，重新修改源码并执行完整流程

**错误做法**:
- ❌ 直接编辑 `opencode.json`
- ❌ 直接编辑 `.sdd/*` 或 `.sddu/*`
- ❌ 直接编辑 `.opencode/*`
- ❌ 直接编辑 `dist/*`
- ❌ 跳过任何构建步骤
- ❌ 直接在当前项目安装未经测试的插件

## 3. 技术方案对比

### 方案 A: 彻底清理（推荐）

**描述**: 所有源码、模板、测试中的 SDD 字眼全部替换为 SDDU，不保留任何向后兼容代码或描述。

**优点**:
- ✅ 品牌完全一致，无混淆
- ✅ 长期维护成本低
- ✅ 代码简洁，无冗余
- ✅ 符合 V2 核心原则
- ✅ 测试文件同步更新，确保一致性

**缺点**:
- ❌ 需要修改的文件较多
- ❌ 需要完整测试验证
- ❌ V1 用户需了解变更

**风险评估**:
- 低风险：技术实现简单，直接替换
- 低风险：内部开发人员使用，影响范围可控
- 缓解：完整测试流程确保功能正常

**预估工作量**: 20 小时

---

### 方案 B: 保留向后兼容（不推荐）

**描述**: 在清理的同时保留向后兼容层，测试文件保留旧命名作为兼容测试。

**优点**:
- ✅ V1 用户不受影响
- ✅ 迁移成本低

**缺点**:
- ❌ 代码冗余，维护复杂
- ❌ 违背 V2"不做兼容"原则
- ❌ 增加不必要的复杂性
- ❌ 长期维护成本高

**风险评估**:
- 中风险：代码冗余可能导致混淆
- 中风险：维护成本增加
- 缓解：无必要，V2 明确不做兼容

**预估工作量**: 18 小时（但后续维护成本高）
