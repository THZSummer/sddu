# ADR-001: 批量统一重构方案

## 状态
PROPOSED

## 背景

SDDU 插件的 17 个模板文件（11 个 Agent 指令模板 + 6 个产物输出模板）在格式上存在 10 个维度的不一致，11 个 Agent 缺少显式的四字段职责边界声明。在 discovery 阶段识别出 7 处职责冲突。需要选择一种技术方案来消除这些不一致，使模板维护者和使用者获得可预期的体验。

## 决策

我们决定采用**方案 A：批量统一重构**——一次性对所有 18 个模板文件（含 1 个删除、1 个重命名、1 个新建）进行格式统一和职责边界注入。

核心原则：
1. **骨架优先**：所有指令模板遵循统一章节序列（FR-013），所有产物模板遵循统一双节结构（FR-020）
2. **边界显式**：每个 Agent 模板包含四字段职责声明（负责/输入/输出/不负责），措辞遵循 EC-003（主责而非禁止）
3. **内容不变**：工作流程/核心职责内容保持不变，仅章节重排和命名统一
4. **占位符保留**：所有 `<<变量名>>` 保持原样
5. **原子交付**：一次提交包含全部变更，通过构建验证后交付

备选方案：
- 方案 B（分批渐进重构）：中间状态存在不一致，时间更长，被否决
- 方案 C（模板引擎化）：过度设计，违反 NG2/NG3，被否决

## 后果

### 正面影响
- G1（有章可循）达成：新增模板时可参考 11 个标准化模板
- G2（体验一致）达成：所有模板结构可预期，章节名和顺序统一
- G3（边界分明）达成：每个 Agent 的四字段边界明确，Agent 不越界不推诿
- 回滚成本极低：纯文本模板变更，`git checkout` 即可完整回滚

### 需要关注
- 单次变更 22 项文件操作，需要仔细的 diff 审查
- `build-agents.cjs` 和 `opencode.json` 需要同步更新，确保构建通过
- 新 sddu-docs 模板（FR-010）的工作流程内容需要后续 Feature 补充细化
