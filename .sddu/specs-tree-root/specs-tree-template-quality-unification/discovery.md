# 需求挖掘报告：预置输出模板质量统一

> **discovery 边界**：本文档只做三件事——记录原始痛点、描述具体问题、表达业务诉求。核心评判：这件事是不是原始痛点、问题、诉求——是就写，不是就不写。

## 1. 问题定义

### 核心问题

SDDU 插件当前各 Agent 模板存在两大问题：

#### 问题一：模板风格迥异，不便于理解与维护

17 个模板文件在格式上各写各的——标题格式不统一、章节顺序随心排、代码块反引号数参差不齐、缩进断裂、表格列宽不一。模板本该是团队的统一语言，现状却是打开两个模板就得先猜格式，新增或修改模板时无从参考。

#### 问题二：Agent 边界不清晰，职责不分明

全部 11 个 Agent 的角色定位是叙述式散文，缺少"谁负责什么、谁不负责什么"的显式声明。用户在相邻阶段之间切换时经常困惑：

- discovery vs spec：需求定义权到底归谁？用户不知道哪个阶段输出才算"需求定稿"
- spec vs plan：spec 要不要写技术细节？plan 管不管功能边界？
- plan vs tasks：都在分解任务，分界线在哪？
- review vs validate："一致性检查"几乎同义反复，到底区别是什么？
- roadmap vs docs：都在输出项目全景报告，什么时候该 @ 谁？
- sddu vs sddu-help：用户遇到问题该找谁？

> **审查日期**: 2026-06-13  
> **审查方法**: 逐文件人工审查 + 相邻 Agent 职责比对  
> **涉及文件**: 11 个 Agent 指令模板 + 6 个输出模板（共 17 个）

### 业务诉求

- 打开任意一个 Agent 模板，能一眼找到对应章节，不需要先猜格式
- Agent 之间职责分明，不再出现推诿或越界
- 新增或修改模板有章可循，不靠个人记忆
- 同类问题不会被不同 Agent 重复处理，也不会被遗漏

## 2. 用户场景

1. **维护者修改模板**: 打开两个模板文件对比 → 发现格式不一致 → 不确定以哪个为准 → 反复确认
2. **新增 Agent**: 参考现有模板复制 → 不知道该用哪种标题格式、几个反引号 → 结果又引入新的不一致
3. **用户使用 Agent**: 从 discovery 切换到 spec → 输出格式突变 → 阅读流畅度打断
4. **用户遇到问题**: 不知道该 @sddu 还是 @sddu-help → 两个都试 → 得到两份不同风格的回答

## 3. 问题详细描述

### Q1: 标题格式不统一

4 个 Agent 的标题格式与其余 Agent 不一致：

| Agent | 当前标题 |
|-------|----------|
| sddu-help | `# @sddu-help - SDDU 工作流帮助` |
| sddu-docs | `# @sddu-docs - SDDU 目录导航生成器` |
| sddu-roadmap | `# 🎯 SDDU Roadmap 规划专家 (@sddu-roadmap)` |
| sddu (主入口) | `# @sddu - SDDU 工作流智能入口` |

其余 7 个主流程 Agent 使用 `# 🎯 SDDU 工作流 - 阶段 X/6` 格式。

### Q2: 章节结构不统一

不同 Agent 的章节组织方式各不相同：

- 同样是主流程 Agent，build 和 review 多了 `核心职责` 章节，discovery 和 spec 多了 `辅导模式` 章节，没有规律
- 同样功能的章节用了不同名字：build 叫 `约束条件`，其他 Agent 叫 `规则`
- 4 个辅助 Agent（docs / roadmap / help / sddu）之间几乎没有共同的章节骨架，各自为政

### Q3: 代码块反引号数不一致

- sddu-review 和 sddu-validate 的「执行顺序」代码块使用 4 反引号（````）
- 其余 9 个 Agent 使用 3 反引号（```）

### Q4: 缩进断裂

- sddu-review.md.hbs 的依赖关系节有多余缩进
- sddu-validate.md.hbs 某段落起始有多余空格

### Q5: 异常处理表格格式不统一

- 有的 Agent 根本没有异常处理表格（如 build、tasks）
- 有表格的 Agent 之间列宽不一致，个别表头分隔线列数与列数对不上

### Q6: 依赖关系节格式不一致

7 个主流程 Agent 的依赖关系节，在字段标签（有的写了字段名、有的没有）、缩进、空行间距上各不相同。

### Q7: 输出模板「自动触发文档更新」块位置不一

同一个功能块，在 6 个输出模板中出现在不同位置——有的嵌在输出格式节里面，有的独立成小节，有的跟在末尾。

### Q8: Agent 版本信息缺失

大部分 Agent 模板没有版本信息。只有 sddu-help 和 sddu-roadmap 标了版本。

### Q9: 「输出模板」节覆盖不均

7 个主流程 Agent 都有「输出模板」节，但 4 个辅助 Agent（sddu-help、sddu-docs、sddu-roadmap、sddu）没有。

### Q10: 输出模板结构深度不均

- sddu-discovery.md.hbs 包含「输出格式」+「完成报告」双节
- 其余 5 个输出模板仅「输出格式」单节，没有「完成报告」节

### Q11: Agent 角色边界不清晰

全部 11 个 Agent 的「角色定位」节是叙述式散文，缺少"谁负责什么、谁不负责什么"的显式声明。问题有两个维度：

- **静态维度**：模板中角色声明为叙述式散文，缺少显式边界 → 用户不知道该找谁
- **动态维度**：模板中只有"我能做什么"的正向描述，缺少"我不该做什么"的负向约束 → Agent 自身忘记边界、越权执行

具体冲突：

| 冲突表现 | 涉及 Agent |
|----------|-----------|
| 两者都在定义需求，用户不知道哪个阶段输出才算"需求定稿" | discovery vs spec |
| spec 涵盖验收标准、plan 写技术方案——spec 到底要不要写技术细节？ | spec vs plan |
| 都在分解任务，宏观和原子的分界线模糊 | plan vs tasks |
| review 说"审查代码质量和规范符合性"，validate 说"检查代码与规范的一致性"——几乎同义反复 | review vs validate |
| 都输出项目全景报告，触发场景和输出形态边界模糊 | roadmap vs docs |
| 都是入口/帮助，用户遇到问题不知道该 @ 谁 | sddu vs sddu-help |
| Agent 忘记自己的角色定位，见到任务就抢——比如用户 @sddu-discovery，spec Agent 却越权处理 | 全部 Agent（运行时行为） |
