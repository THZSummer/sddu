# ADR-V2-001: State Schema 验证策略（运行时验证 vs 创建时验证）

## 状态
PROPOSED

## 背景
TreeStateValidator 需要确定 Schema 验证的最佳时机：
- **创建时验证**：在 StateLoader.create() 写入文件前验证，确保写入的文件 100% 合规
- **运行时验证**：在 StateLoader.get() 读取文件时验证，确保读取的状态可用

当前代码中两个时机都存在验证逻辑，但职责边界不清，导致：
1. create() 内部的验证不完整（只调用 validateState 基本检查）
2. get() 内部的 applyReparation() 修复逻辑与 validateNewState() 重复
3. 缺少统一的 ValidationResult 返回格式

## 决策
我们决定 **双时机验证，各司其职**：

### 创建时验证（StateLoader.create → TreeStateValidator.validateNewState）
- **目的**：确保写入的 state.json 100% 符合 v2.1.0 Schema
- **行为**：
  1. StateLoader.create() 构建初始 state 对象
  2. 调用 TreeStateValidator.validateNewState() 进行校验和修复
  3. 使用修复后的 state 写入文件
  4. 如有修复，记录 WARNING 日志

### 运行时验证（StateLoader.get → TreeStateValidator.applyReparation）
- **目的**：确保读取的历史 state.json 可用（向后兼容）
- **行为**：
  1. 读取文件后检查是否存在已知问题（stateHasIssues）
  2. 如存在问题，调用 applyReparation() 修复（仅内存中，不写回文件）
  3. 记录 WARNING 日志提示用户

### 统一接口
- 新增标准化的 `TreeStateValidator.validate()` 方法
- 返回 `ValidationResult { valid, errors, warnings, autoFixed, state }`
- 所有验证场景统一使用此接口

## 后果
### 正面
- 职责清晰：创建时保证写入合规，运行时保证读取可用
- 统一接口减少代码重复
- ValidationResult 格式标准化，便于调用方处理

### 负面
- 需要重构部分现有验证逻辑
- 双时机验证增加代码量

### 风险
- 低：v2 定位为增量修复，不改变核心架构

## 相关决策
- ADR-V2-002: 缺失字段处理策略（决定采用自动填充）
- ADR-V2-005: 历史 state.json 兼容性处理（决定保持兼容）

## 替代方案
| 方案 | 描述 | 未选择原因 |
|------|------|-----------|
| 仅创建时验证 | 只在 create() 时验证 | 无法处理已有历史 state.json |
| 仅运行时验证 | 只在 get() 时验证 | 无法保证写入文件的合规性 |
| 统一验证中间层 | 新建 SchemaEnforcer 中间层 | 对当前问题过度设计 |
