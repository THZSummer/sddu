# ADR-V2-005: 历史 state.json 兼容性处理（保持兼容 vs 强制迁移）

## 状态
PROPOSED

## 背景
现有 11 个 Feature 的 state.json 是在 v2.4.0 阶段创建的，可能缺少 v2.1.0 Schema 要求的某些必填字段（如 version、depth、phaseHistory）。v2 需要决定：
- **保持兼容**：读取时自动修复缺失字段，不修改原文件
- **强制迁移**：提供迁移脚本，一次性更新所有历史 state.json

## 决策
我们决定 **保持兼容，不强制迁移**：

### 兼容性策略
1. **读取时自动修复**（StateLoader.get）
   - 检测已知问题（EC-012/013/014）
   - 在内存中修复，不写回文件
   - 记录 WARNING 日志：`WARNING: Automatic repair(s) applied to state in {featurePath}`

2. **写入时保证合规**（StateLoader.create）
   - 所有新建 state.json 100% 符合 v2.1.0 Schema
   - 不产生新的不合规文件

3. **渐进式更新**
   - 当用户修改某个 Feature 时，该 Feature 的 state.json 会被自动更新为合规版本
   - 不需要一次性迁移所有历史文件

### 为什么不强制迁移
1. **风险**：一次性迁移 11 个 Feature 的 state.json 可能引入未知问题
2. **NFR-001**：明确要求"不破坏现有 11 个 Feature 的正常运行"
3. **不必要**：读取时自动修复已能满足功能需求
4. **渐进式更好**：每次修改自动更新，逐步完成迁移

### 迁移触发时机
历史 state.json 会在以下时机被自动更新为合规版本：
- 用户通过 StateLoader.update() 修改 state
- StateMachine 更新 Feature 状态
- ParentStateManager 更新 childrens 数组

## 后果
### 正面
- 零风险：不触碰现有文件
- 渐进式：自然完成迁移
- 用户无感知：自动修复在后台完成

### 负面
- 短期内存在"双版本"state.json（合规和不合规并存）
- 自动修复逻辑需要长期维护

### 风险
- 低：自动修复的是有明确默认值的字段
- 缓解：WARNING 日志可追踪所有修复操作

## 相关决策
- NFR-001: 不破坏现有 11 个 Feature 的正常运行
- ADR-V2-001: State Schema 验证策略
- ADR-V2-002: 缺失字段处理策略

## 替代方案
| 方案 | 描述 | 未选择原因 |
|------|------|-----------|
| 强制迁移 | 提供迁移脚本一次性更新 | 风险高，违反 NFR-001 |
| 忽略不兼容 | 不处理历史文件 | 功能异常，Schema 验证失败 |
| 双 Schema 支持 | 同时支持旧版和新版 | 增加复杂度，长期维护成本高 |
