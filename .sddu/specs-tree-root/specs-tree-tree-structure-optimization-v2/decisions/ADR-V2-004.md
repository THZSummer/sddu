# ADR-V2-004: 树形测试场景实现方式（脚本生成 vs 手动创建）

## 状态
PROPOSED

## 背景
树形嵌套 E2E 测试场景（FR-120~123）需要创建测试项目结构，有两种实现方式：
- **脚本生成**：使用 bash 脚本自动创建测试项目结构和 state.json
- **手动创建**：手动编写所有测试文件和 state.json

## 决策
我们决定 **采用脚本生成的方式**：

### 实现方案
```
scripts/e2e/tree-scenario/
├── setup.sh                    # 创建测试项目结构
├── validate.sh                 # 运行所有验证
├── verify-childrens.sh         # childrens 数组验证
├── verify-depth.sh             # depth 字段验证
└── verify-cross-tree-deps.sh   # 跨子树依赖验证
```

### 脚本生成优势
1. **可重复**：每次运行 setup.sh 都能创建相同的测试环境
2. **易维护**：修改测试场景只需修改脚本，无需手动更新多个文件
3. **可集成**：可轻松集成到 CI/CD 流程中
4. **清理方便**：EC-107 要求"测试目录已存在时先清理旧数据"，脚本天然支持

### 验证方式
- 验证脚本使用纯 bash + jq（JSON 处理）
- 每个验证脚本返回退出码：0=通过，1=失败
- validate.sh 汇总所有验证结果

### 为什么不选择手动创建
1. **维护成本**：手动创建的文件容易过时，与代码不同步
2. **可重复性差**：不同环境可能创建不一致的测试数据
3. **CI 集成困难**：手动文件需要额外检查机制

## 后果
### 正面
- 测试环境创建一键化
- 易于在 CI/CD 中自动化
- 修改测试场景只需改脚本

### 负面
- bash 脚本调试不如 TypeScript 方便
- 依赖环境中的 jq 工具

### 风险
- 低：脚本实现简单，不依赖复杂逻辑
- 缓解：提供详细的错误输出和验证步骤

## 相关决策
- FR-120~123: 树形嵌套 E2E 测试
- EC-107: 测试项目目录已存在时的处理

## 替代方案
| 方案 | 描述 | 未选择原因 |
|------|------|-----------|
| 手动创建文件 | 直接编写所有测试文件 | 维护成本高，不易集成 CI |
| TypeScript 测试脚本 | 使用 ts-node 运行测试 | 需要额外依赖，脚本更轻量 |
| Jest 集成测试 | 集成到现有 Jest 测试框架 | E2E 场景更适合独立脚本 |
