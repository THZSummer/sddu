# ADR-V2-003: 拆分建议触发规则（基于规则 vs 基于 AI 判断）

## 状态
PROPOSED

## 背景
Discovery 阶段需要识别用户需求中的拆分模式（前后端分离、多端架构等），有两种实现方式：
- **基于规则**：关键词匹配 + 模式识别
- **基于 AI**：使用 LLM 语义分析判断是否需要拆分

## 决策
我们决定 **采用基于规则的方式，保守策略**：

### 规则引擎设计
```typescript
const SPLIT_PATTERNS = [
  {
    id: 'frontend-backend',
    name: '前后端分离',
    keywords: ['前端', '后端', 'frontend', 'backend', 'client', 'server', 'web', 'api'],
    minMatchCount: 2  // 至少匹配 2 个关键词才触发
  },
  {
    id: 'multi-platform',
    name: '多端架构',
    keywords: ['ios', 'android', '移动端', 'pc端', '小程序', 'h5'],
    minMatchCount: 2
  },
  {
    id: 'admin-user',
    name: '管理后台+用户端',
    keywords: ['管理后台', '用户端', 'admin', 'user'],
    minMatchCount: 2
  }
];
```

### 触发条件
- 同一模式匹配 ≥ 2 个关键词时输出拆分建议
- 多个模式同时匹配时输出所有候选（ambiguous=true）
- 未匹配任何模式时不输出建议

### 保守策略理由
1. **可控性**：规则明确，行为可预测，易于调试
2. **低成本**：不需要额外 API 调用，无延迟
3. **渐进式**：后续可基于用户反馈逐步扩展规则
4. **兜底能力**：用户可通过自定义拆分（FR-111）覆盖规则判断

### 为什么不选择 AI 判断
1. **成本**：每次 Discovery 都需要 LLM 调用
2. **一致性**：AI 判断结果可能不稳定
3. **当前需求**：spec 明确要求关键词匹配（FR-110）
4. **可演进**：规则引擎成熟后，可叠加 AI 辅助判断

## 后果
### 正面
- 行为可预测，易于测试
- 无额外 API 成本
- 规则可扩展，逐步完善

### 负面
- 无法处理复杂语义（如"构建一个包含用户管理和商品展示的完整系统"）
- 关键词可能存在误报/漏报

### 风险
- 中：初始规则覆盖有限，可能漏拆或误拆
- 缓解：保守策略 + 用户确认机制 + 自定义拆分兜底

## 替代方案
| 方案 | 描述 | 未选择原因 |
|------|------|-----------|
| 纯 AI 判断 | 使用 LLM 分析需求描述 | 成本高、结果不稳定、当前 spec 未要求 |
| 规则 + AI 混合 | 规则初筛 + AI 精判 | 过度设计，v2 定位为修复增强 |
| 用户手动指定 | 完全依赖用户判断 | 违背自动化目标 |
