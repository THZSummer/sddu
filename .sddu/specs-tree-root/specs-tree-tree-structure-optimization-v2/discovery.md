### 1.4 根因深度分析（5 Whys）

**P-001: 为什么 state.json 缺少必填字段？**
1. Why: State Machine 创建 state 时未强制校验必填字段
2. Why: 初始化逻辑只设置了部分字段（feature, name, status, phase）
3. Why: 开发时假设 Schema 验证会在写入时拦截
4. Why: Schema 验证器（tree-state-validator.ts）只验证 format，不主动填充
5. **根因**: State Machine 初始化逻辑与 Schema 验证器之间存在职责空白

**P-002: 为什么 Discovery 未输出拆分建议？**
1. Why: blog-platform 有前后端分离但未触发拆分
2. Why: 拆分识别规则只检查了"多个独立模块"关键词
3. Why: 规则未覆盖"前端/后端"、"客户端/服务端"等常见拆分模式
4. Why: 规则设计基于假设场景，未基于真实项目特征
5. **根因**: 拆分识别规则缺乏对常见项目模式（前后端分离、多端架构）的覆盖

### 1.5 不做的成本

| 维度 | 影响 |
|------|------|
| **可靠性** | state.json 字段缺失导致 Schema 验证失败，下游 Agent 无法正确读取状态 |
| **智能化** | 用户需要手动拆分 Feature，降低 SDDU 的自动化价值 |
| **可验证性** | 缺少树形嵌套测试意味着核心能力未被充分验证，存在未知 Bug |
| **易用性** | 新用户缺少拆分原则文档和示例项目，上手成本高 |
| **信任度** | E2E 测试覆盖不完整影响用户对插件质量的信心 |

### 1.6 业务价值

| 价值类型 | 具体描述 | 衡量方式 |
|----------|----------|----------|
| **可靠性** | state.json schema 完整性保证，所有必填字段强制填充 | Schema 验证通过率 100% |
| **智能化** | Discovery 阶段自动识别前后端分离等模式，输出拆分建议 | 拆分建议输出率 ≥ 80% |
| **可验证性** | 树形嵌套 E2E 测试覆盖核心场景 | E2E 测试通过率 100% |
| **易用性** | 拆分原则文档 + 树形示例项目，降低新用户上手成本 | 文档完整度评分 ≥ 4/5 |

### 2.1 主要用户

### 2.2 用户场景还原

**场景 A: Feature 创建时 state.json 完整性（P-001）**
- 当前: State Machine 创建 state.json 时只填充部分字段，下游读取时可能因缺少必填字段而报错
- 期望: 创建时自动填充所有必填字段（version, depth, phaseHistory, dependencies），确保 Schema 验证通过

**场景 B: 前后端分离项目拆分建议（P-002）**
- 当前: 用户描述"我需要一个博客平台，有前端和后端"，Discovery 未识别出拆分时机
- 期望: Discovery 识别到"前端/后端"关键词，输出拆分建议："建议拆分为 blog-platform-frontend 和 blog-platform-backend 两个子 Feature"

**场景 C: 树形嵌套功能验证（P-003）**
- 当前: blog-platform 测试项目是单层结构（root → leaf），无法验证树形嵌套能力
- 期望: 测试项目包含多层嵌套（root → parent → child1/child2 → grandchild），完整验证树形能力

**场景 D: 新用户学习拆分原则（P-004）**
- 当前: 新用户不知道何时应该拆分、如何拆分
- 期望: 提供拆分原则文档（何时拆、拆到什么粒度、父子关系规则）和树形示例项目

**场景 E: 跨子树依赖验证（P-005）**
- 当前: 跨子树依赖解析代码已实现，但未被实际测试验证
- 期望: 创建跨子树测试场景（child1 依赖 child2，两者在不同父级下），验证依赖解析正确性

### 2.3 用户旅程地图

```
当前旅程（痛点标注 🔴）:
用户调用 @sddu-discovery → 调用 @sddu-spec → 生成 Feature
  ├── state.json 缺少必填字段 🔴 下游验证失败
  ├── 前后端分离未触发拆分建议 🔴 用户需手动拆分
  ├── 测试项目单层结构 🔴 树形嵌套能力未验证
  ├── 缺少拆分原则文档 🔴 新用户上手困难

  ├── 树形嵌套 E2E 测试完整覆盖 ✅ 核心能力已验证
```
