## 1. 问题定义
> 概括核心问题及其业务影响，回答"为什么需要关注"

| 核心问题 | 业务影响 | 不解决的成本 |
|---------|---------|------------|
| SDDU 框架源码缺乏分层隔离 — 工作流核心代码与 opencode 平台扩展代码混放在 `src/` 下，维护者无法分辨模块职责边界 | 新增维护者上手困难，架构理解成本持续累积；模块耦合导致修改核心逻辑时可能意外破坏平台集成 | 未来每次架构决策都需全量理解整个 `src/`，平台无关的工作流能力无法被独立演进或复用 |
| 缺少显式的扩展接口/适配层 — 核心引擎对 opencode 的依赖散落在 `index.ts` 和各模块的隐式调用中，无契约边界 | 若未来需将 SDDU 移植到其他编辑器/平台，需 fork 源码并大规模重写集成逻辑 | 框架的可移植性为零，SDDU 工作流能力被锁定在单一平台 |
| 测试结构不一致且与源码耦合方式混乱 — 部分模块有 `__tests__/` 子目录，部分测试文件与源文件平级混放，无法按层独立执行测试 | 核心工作流逻辑的回归测试无法与平台集成测试分离，CI 反馈粒度粗糙 | 重构或移植时缺乏分层测试安全网，改动风险高 |

## 2. 用户画像
> 描述受影响用户角色及其场景，回答"谁遇到了什么问题"

| 用户角色 | 典型场景 | 关键痛点（用户原话） | 当前应对方式 |
|---------|---------|-------------------|------------|
| **框架维护者**（核心用户） | 阅读 `src/` 源码以理解某个 Bug 的根因或评估新特性的影响范围 | 「维护者看不清哪些是工作流核心源码，哪些是针对 opencode 的扩展源码」 | 靠个人经验记忆模块归属，无结构化导航手段 |
| **新贡献者** | 初次 clone 仓库后试图理解项目架构，寻找修改入口 | 「整体结构缺乏拓展思维」— 目录结构不传达设计意图 | 通读全部源码才能建立心智模型，入门成本高 |
| **架构决策者** | 规划框架长期演进方向，评估是否可移植到新平台 | 核心工作流逻辑与平台胶水代码纠缠，无法评估解耦代价 | 推迟跨平台计划，或接受大规模重写代价 |
| **插件用户**（间接影响） | 安装 SDDU 插件到 opencode 中使用 | 功能不变（约束要求）— 本次重组不影响最终用户体验 | 无感知 |

## 3. 问题清单
> 按影响程度分级梳理所有识别到的问题，每项赋予唯一编号 Q-xxx

### 3.1 核心问题
> 影响面大、频率高、用户强烈感知

| ID | 问题描述 | 影响范围 |
|----|---------|---------|
| Q-001 | **核心/扩展源码无隔离边界**：当前 `src/` 下所有模块（agents、commands、discovery、state、templates、utils）处于同一命名空间，无层级区分。例如 `agents/`（opencode 平台扩展）与 `discovery/`（工作流核心）是同级目录，维护者阅读时无法从文件路径判断模块的平台依赖性。 | 所有维护者，每次阅读源码 |
| Q-002 | **缺少显式扩展接口层**：`src/index.ts`（645 行）同时承担插件入口、工具注册、事件监听、类型导出四重职责，核心引擎对 opencode 的依赖（`@opencode-ai/plugin` 的 `tool()` API）直接嵌入在入口文件中，无抽象接口契约。若需适配新平台，无法替换平台层而不触及核心逻辑。 | 框架演进决策者，任何需要评估移植性的场景 |
| Q-003 | **测试组织缺少分层一致性**：`src/errors.test.ts` 和 `src/types.test.ts` 与源文件平级混放在 `src/` 顶层；`state/` 和 `discovery/` 模块使用 `__tests__/` 子目录模式；`agents/` 模块的测试文件（`registry.test.ts`）与源文件同目录。三种组织方式并存，无法按"仅核心"或"仅扩展"维度选择性执行测试。 | 所有开发者，每次运行测试或 CI 反馈 |

### 3.2 次要问题
> 影响面中等、或为核心问题的衍生问题

| ID | 问题描述 | 影响范围 |
|----|---------|---------|
| Q-004 | **模板目录职责不清**：`src/templates/agents/` 存放 Agent 指令模板（行为定义），`src/templates/config/` 存放配置，顶层又有 `subfeature-templates.ts`（模板管理逻辑）。模板的"内容（配置）"与"引擎（管理逻辑）"未分离，且 agent 模板的内容属于"SDDU 业务定义"还是"opencode 平台配置"的归属模糊。 | 模板维护者和需要定制 Agent 行为的用户 |
| Q-005 | **构建产物的扁平化映射**：当前 `tsconfig.json` 和构建流程将 `src/` 编译为扁平的 `dist/` 目录结构，不保留源码的分层语义。这意味着即使源码做了架构分层，构建产物仍需额外配置才能反映分层。 | 构建流程维护者，以及作为 npm 包使用 SDDU 的下游消费者 |

### 3.3 潜在问题
> 目前影响小但可能恶化，或信息不足待验证

| ID | 问题描述 | 影响范围 |
|----|---------|---------|
| Q-006 | **`utils/` 模块的平台依赖性待审计**：`src/utils/` 目录下的工具函数目前未区分"纯逻辑工具（平台无关）"与"opencode 平台工具（依赖 opencode SDK）"。随着工具函数积累，此目录将成为下一个耦合重灾区。 | 长期维护 |
| Q-007 | **`errors.ts` 和 `types.ts` 的跨层引用风险**：这两个顶层文件被核心模块（discovery、state）和扩展模块（agents、commands）共同引用。如果未来核心层与扩展层需要不同的类型定义，当前单一文件结构将成为瓶颈。 | 架构演进时暴露 |

## 4. 竞品参考
> 记录竞品对类似问题的处理方式，回答"别人怎么做的、我们有什么不同"

| 竞品 | 是否处理过类似问题 | 处理方式 | 与我们场景的差异 |
|------|-------------------|---------|----------------|
| **VSCode Extension API** | 是 | 采用明确的 `ExtensionContext` 接口作为扩展与宿主之间的契约，核心编辑器能力通过 `vscode.*` 命名空间暴露，扩展代码通过 `activate(context)` 函数接入 | VSCode 的"核心"是闭源的编辑器本体，扩展是第三方开发；SDDU 则是"核心开源 + 平台适配"一体发布，需要同时维护两端 |
| **LangChain** | 是 | 采用 `langchain-core`（纯逻辑）、`langchain-community`（第三方集成）、`@langchain/openai` 等平台包的分包策略，核心定义抽象接口（如 `BaseLLM`），平台包实现具体适配 | LangChain 是多包分发，SDDU 是单包插件，分包策略需要评估是否适合单仓结构 |
| **Jest** | 是 | 核心 runner 与平台适配器（`jest-cli`、`jest-worker` 等）分子包管理，测试按包组织，各包有独立的 `__tests__/` | Jest 是独立测试工具，SDDU 是 opencode 插件，分包的发布复杂度更高 |
| **Prettier** | 是 | 核心格式化引擎（`prettier` 包）与编辑器插件（VS Code、Vim 等）完全分离，插件通过调用 Prettier API 实现集成 | Prettier 的核心是 CLI 工具，SDDU 的核心是工作流引擎，二者对外暴露的接口形式不同 |

> ⚠️ **注意**: 以上竞品信息仅描述事实，不做方案推荐。具体采用何种架构风格将在 spec 和 plan 阶段结合 SDDU 的实际约束（单包插件、openCode 平台依赖、现有用户基础）进行评估。

## 5. 假设与风险
> 记录问题挖掘过程中识别的假设和风险，供后续阶段验证和关注

### 5.1 关键假设
> 记录我们对问题理解所基于的假设，标注待验证项

| # | 假设内容 | 验证方式 |
|---|---------|---------|
| A-001 | SDDU 核心工作流引擎（discovery、state、templates 逻辑）可以完全与 opencode SDK 解耦，即不存在对 `@opencode-ai/plugin` 的硬依赖 | 逐模块审查 import 语句，验证核心模块是否能编译为无 opencode 依赖的纯 TypeScript |
| A-002 | 现有的 opencode 插件集成行为（工具注册、事件监听）可以通过适配层完整保留，功能不变 | 在重组后运行现有测试套件和 E2E 验证，确认所有 `sddu_*` 工具和事件监听器功能不变 |
| A-003 | 分层架构不会显著增加构建复杂度或 npm 包体积 | 对比重组前后的 `dist/` 大小和 `npm pack` 产物 |
| A-004 | 当前所有维护者理解并认可"核心/扩展分层"的设计方向 | spec 阶段与团队确认架构愿景 |
| A-005 | `.opencode/*`、`opencode.json`、`.sddu/*` 不在本次修改范围，这些目录由用户安装插件后自行管理 | 边界约束已在用户要求中明确 |

### 5.2 主要风险
> 识别可能影响问题判断或后续决策的风险因素

| # | 风险描述 | 影响程度 |
|---|---------|---------|
| R-001 | **过度设计风险**：用户要求"超前设计，不急但一定要设计好"，若分层过于激进（如过度抽取抽象接口），可能导致当前简单场景下的代码复杂度不必要地膨胀 | 中 |
| R-002 | **回归风险**：重组 `src/` 目录结构涉及大量 import 路径变更，可能在重组过程中引入隐蔽的引用错误 | 高 |
| R-003 | **团队共识风险**：若其他贡献者对分层方案的必要性或具体切分方式有异议，可能拉长决策周期 | 中 |
| R-004 | **类型导出断裂风险**：`src/index.ts` 当前导出的公共 API（如 `StateMachine`、`DiscoveryWorkflowEngine` 等）被下游消费者或内部测试引用，重组后需保证导出路径兼容或提供迁移路径 | 中 |

## 6. 下一步建议
> 给出后续工作的优先级建议，回答"接下来优先做什么"

| 优先级 | 事项 | 说明 |
|--------|------|------|
| 高 | 聚焦 **Q-001 核心/扩展隔离**：定义清晰的分层模型（如 `core/` 工作流引擎层 + `opencode/` 平台适配层 + `shared/` 公共类型），确定每个现有模块的归属 | 这是最根本的架构问题，其他 Q-002~Q-007 都由此衍生 |
| 高 | 聚焦 **Q-002 扩展接口定义**：在分层基础上，设计核心层与平台适配层之间的显式接口契约（如 `SDDURuntime` 接口），使适配层可替换 | 这是"拓展思维"的关键体现 |
| 中 | 处理 **Q-003 测试分层**：在目录重组完成后，统一测试文件组织方式，确保可按层独立执行测试 | 依赖 Q-001 的分层方案先落地 |
| 中 | 审计 **Q-006 utils/ 模块**：在分层过程中逐文件标注每个工具函数的平台依赖性，为后续整理做准备 | 可并入 Q-001 工作中同步处理 |
| 低 | 调查 **Q-004 模板职责**：与 spec Agent 协作，明确模板内容的归属边界 | 可在 spec 阶段进一步细化 |
| 低 | 评估 **Q-005 构建产物**：调研 tsconfig paths 映射或 subpath exports 是否能支持分层构建产物 | 属 plan 阶段技术方案的一部分 |
