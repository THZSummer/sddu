## 2. 上下文
> 回顾问题背景和目标用户

SDDU 框架当前的核心问题是：**源码组织不传达业务架构意图**。虽然经过审计 46/47 个源文件已是纯逻辑代码（零 opencode SDK 依赖），但目录结构不区分「SDDU 方法论业务」与「openCode 平台适配」，导致：

- **框架维护者**：阅读源码时无法从文件路径判断模块的平台依赖性，靠个人经验记忆模块归属
- **新贡献者**：首次接触代码库需通读全部源码才能建立心智模型，入门成本高
- **架构决策者**：无法评估 SDDU 工作流引擎移植到其他 LLM 平台的可行性和代价

本 Feature 的目标是在不修改任何功能行为的前提下，重组源码架构，使代码组织传达业务设计意图。

**核心约束**：`.opencode/*`、`opencode.json`、`.sddu/*` 不在本次修改范围。

## 3. 目标与非目标
> 明确需求范围，防止范围蔓延

### 3.1 目标 (Goals)
> 明确本次要达成的业务目标

| # | 目标描述 |
|---|---------|
| G-001 | SDDU 方法论的核心业务逻辑（工作流引擎、状态管理、需求挖掘、模板生成、任务编排、Agent 行为定义）在代码组织上形成明确的业务域边界，与平台适配代码清晰分离 |
| G-002 | openCode 平台的适配代码（Agent 注册、工具注册、生命周期钩子、平台配置）作为独立的适配层，通过契约依赖核心业务，而非核心业务依赖平台 |
| G-003 | 公共类型、错误定义、状态模式等跨业务域共享的约定，对任何平台零依赖 |
| G-004 | 测试按业务域组织，支持「仅核心业务」「仅平台适配」「全量」三种粒度的独立执行 |
| G-005 | 平台适配层可被替换——未来若适配其他 LLM 平台/编辑器，核心业务逻辑零改动 |
| G-006 | 构建产物保留业务分层语义，下游消费者可按业务域引用 |

### 3.2 非目标 (Non-Goals)
> 明确本次不涉及的范围，防止需求蔓延

| # | 明确不做 |
|---|---------|
| NG-001 | 不修改任何功能行为——所有 sddu_* 工具、事件监听器、Agent 行为与重组前一致 |
| NG-002 | 不改动 `.opencode/` 目录 |
| NG-003 | 不改动 `opencode.json` |
| NG-004 | 不改动 `.sddu/` 目录 |

## 4. 用户故事
> 以用户视角描述功能需求

| # | 作为… | 我想要… | 以便… |
|---|-------|---------|-------|
| US-001 | 框架维护者 | 理解 SDDU 方法论中每个业务概念（工作流/状态/Agent/模板）的边界，以及它们与 openCode 平台的依赖关系 | 修改核心逻辑时明确影响范围，不会意外破坏平台集成 |
| US-002 | 架构决策者 | 清晰地看到 SDDU 工作流引擎的哪些能力是平台无关的，哪些是对 openCode 的适配 | 评估迁移到其他 LLM 平台时，明确哪些业务逻辑可复用、哪些适配需要重写 |
| US-003 | 新贡献者 | 从代码组织就能理解 SDDU 的业务架构——有哪些业务域、它们如何协作 | 首次接触代码库时快速建立心智模型，而非通读全部源码 |
| US-004 | 框架维护者 | 对「SDDU 核心业务」和「openCode 平台适配」分别独立运行测试 | 修改一方时快速验证，不被另一方的测试噪音干扰 |

## 5. 功能需求 (FR)
> 每个需求必须有唯一标识符且可测试

| ID | 需求描述 | 验收标准 | 优先级 |
|----|---------|---------|--------|
| FR-001 | SDDU 方法论工作流核心（含：阶段流转规则、需求挖掘流程、特性树管理、状态一致性校验、任务编排逻辑、规范模板生成规则）应作为独立的业务域，不依赖任何特定 LLM 平台的 SDK | 核心业务域所有模块的 import 语句中不存在对 `@opencode-ai/plugin` 或任何平台特定 SDK 的引用 | P0 |
| FR-002 | SDDU 的 Agent 行为定义（每个阶段 Agent 的职责描述和提示词模板）应作为方法论的业务资产独立管理，与特定平台的 Agent 注册机制分离 | Agent 行为定义（HBS 模板内容）不包含平台特定的注册逻辑或 API 调用；同一套方法论资产可被不同平台的适配层引用 | P0 |
| FR-003 | openCode 平台的适配（Agent 注册、工具注册、生命周期钩子、平台配置）应作为独立适配层，消费核心业务域暴露的能力，而非核心业务域依赖平台 | openCode 适配层可以独立替换（概念上），核心模块不受影响；适配层对核心的依赖是单向的 | P0 |
| FR-004 | 跨业务域共享的类型定义、错误码、状态模式定义应对任何平台零依赖 | shared 域所有文件无平台 SDK import，可被 core 和 opencode 域安全引用而不引入平台依赖传递 | P0 |
| FR-005 | 测试应按业务域组织，支持三种执行粒度——「仅核心业务」：只跑 SDDU 方法论工作流核心的测试；「仅平台适配」：只跑 openCode 平台适配的测试；「全量」：跑全部测试。每种粒度独立可运行且结果互不依赖 | 可分别执行 `test:core`、`test:opencode`、`test:all` 三种测试命令（或等效脚本），各粒度独立通过 | P1 |
| FR-006 | SDDU 的模板资产应区分「方法论文档模板」（定义每个阶段产出物的格式规范，如 spec 应包含什么章节）与「平台配置模板」（用于生成特定平台配置文件的内容，如 opencode.json） | 方法论文档模板与平台配置模板在代码组织中可明确区分，修改一方不影响另一方 | P1 |
| FR-007 | 构建产物应保留业务分层语义——下游消费者（如 npm 包使用者）可以通过构建产物的组织方式区分「SDDU 方法论核心 API」与「openCode 平台适配 API」 | npm pack 产物中可识别核心 API 和平台 API 的边界（具体方案由 plan 阶段确定） | P2 |
| FR-008 | 工具函数应按平台依赖性分类——纯逻辑工具（如任务解析、依赖检测、模板生成）与平台绑定工具（如有）在代码组织上可区分 | 纯逻辑工具所在模块无平台 SDK import；两类工具不混放在同一命名空间 | P1 |
| FR-009 | SDDU 工作流核心业务域应定义明确的对外能力边界——声明它对平台适配层暴露的能力集合和调用契约 | 核心对外暴露的能力有明确的类型定义或接口契约；平台适配层通过该契约消费核心能力，核心不感知具体平台 | P0 |
| FR-010 | SDDU 工作流核心应定义完整的多平台适配接口契约，覆盖工具注册与调用、Agent 注册与生命周期、平台事件监听、状态持久化，使得任何 LLM 平台可通过实现该契约接入 SDDU 方法论引擎 | 存在多平台适配接口的类型定义；核心层不感知具体平台实现 | P0 |

## 6. 非功能需求 (NFR)
> 性能、安全、可用性等跨切面需求

| ID | 类别 | 需求描述 | 验收标准 |
|----|------|---------|---------|
| NFR-001 | 正确性 | 重组后所有现有功能行为不变——所有 sddu_* 工具、事件监听器、Agent 行为与重组前一致，最终用户体验零变化 | 重组前后运行相同的 E2E 验证场景，行为一致 |
| NFR-002 | 回归安全 | 现有全部测试套件在重组后通过，无回归 | 重组后执行全量测试套件，通过率 100% |
| NFR-003 | 兼容性 | 公共 API（index.ts 导出的约 30 个类型/类/函数）在 v4.0.0 中允许 breaking change，但须提供清晰的新导入路径和迁移文档 | 存在迁移文档，逐项列出旧路径→新路径映射 |
| NFR-004 | 构建正确性 | 重组后项目可正常构建，npm pack 产物可正常安装和使用 | `npm run build` 成功，`npm pack` 产物可被下游项目安装并正常调用 |
| NFR-005 | 性能 | 重组不引入运行时性能退化——不应因增加抽象层调用而降低执行效率 | 重组前后的关键路径（状态更新、工作流执行）耗时无明显增加（差异 < 5%） |

## 7. 边界情况 (EC)
> 异常场景和边界条件的处理方式

| ID | 场景 | 处理方式 |
|----|------|---------|
| EC-001 | 循环依赖：重组过程中如果核心业务域与平台适配层之间形成循环依赖（如共享类型引用平台类型），应被检测和阻断 | 在 plan 阶段设计依赖方向规则（core → shared ← opencode），通过构建或 lint 工具在 CI 中强制检查 |
| EC-002 | 模板路径引用断裂：HBS 模板中可能存在对文件路径的引用（如 Agent 定义中的 promptFile 路径），重组目录后需确保模板中的路径引用不失效 | plan 阶段需审计所有 HBS 模板中的路径引用，重组时同步更新或通过构建配置保持兼容 |
| EC-003 | 硬编码路径：源码中可能存在硬编码的目录路径（如 `.sddu/`、`specs-tree-root/`），重组后需确保这些路径语义不变化 | plan 阶段需 grep 审计全量硬编码路径，确认重组前后路径解析结果一致 |
| EC-004 | 协同冲突：若有并行开发的特性分支，目录重组产生的 import 变更可能与正在开发的分支产生大量冲突 | plan 阶段需评估变更影响面，建议在团队无活跃特性分支的时间窗口执行重组 |
