## 1. 问题陈述 (已更新)

### 1.1 背景

当前 SDD 工作流插件的状态管理在 **sdd-multi-module (v1.2.11)** 中已实现分布式 State 管理，但仍有部分功能需要完善。

### 1.2 已解决的问题 ✅

以下问题已通过 `sdd-multi-module` Feature 解决：

| 问题 | 状态 | 解决方案 |
|------|------|----------|
| 1. 单点故障风险 | ✅ 已解决 | 分布式 state.json，每个 Feature 独立存储 |
| 2. 并发冲突概率高 | ✅ 已解决 | 单个 Feature 状态更新不影响其他 Feature |
| 3. 目录结构不自然 | ✅ 已解决 | state.json 与 Feature 文档同目录 |
| 4. 删除 Feature 时状态残留 | ✅ 已解决 | 删除目录即删除状态文件 |

### 1.3 仍需完善的问题 ⚠️

以下问题**部分解决**或**未解决**，需要本 Feature 继续完善：

| 问题 | 状态 | 说明 |
|------|------|------|
| 5. 状态转换不透明 | ⚠️ 部分解决 | StateMachine 已实现但未完全集成到 Agent 工作流 |
| 6. 缺少状态验证 | ⚠️ 部分解决 | Schema 验证存在但缺少自动验证触发 |
| 7. 状态更新手动 | ⚠️ 部分解决 | 需要确认是否支持 session.idle 事件自动触发 |
| 8. 缺少状态历史 | ⚠️ 部分解决 | history 字段存在但需要验证是否自动记录 |
| 9. 依赖关系不清晰 | ⚠️ 部分解决 | dependencies 字段存在但缺少依赖状态检查器 |

### 1.4 新增问题 (基于实际使用反馈)

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 10. 跨 Feature 状态聚合查询性能 | P1 | 根目录 state.json 需要手动维护，未自动聚合 |
| 11. 状态 Schema 版本不统一 | P1 | 发现 v1.2.5、v1.2.11、v2.0.0 多个版本混用 |
| 12. 旧格式兼容性 | P0 | 部分 Feature 仍使用旧格式 state.json |

## 4. 功能需求

### 4.1 状态机集成 (FR-001)

**FR-001: 系统必须将 StateMachine 完全集成到所有 Agent 工作流**

**当前状态**: machine.ts 已存在但未完全集成

**验收标准**:
- [ ] AC-001.1: @sdd-spec 完成后自动更新状态为 `specified`
- [ ] AC-001.2: @sdd-plan 完成后自动更新状态为 `planned`
- [ ] AC-001.3: @sdd-tasks 完成后自动更新状态为 `tasked`
- [ ] AC-001.4: @sdd-build 完成后自动更新状态为 `building`
- [ ] AC-001.5: @sdd-review 完成后自动更新状态为 `reviewed`
- [ ] AC-001.6: @sdd-validate 完成后自动更新状态为 `validated`
- [ ] AC-001.7: 状态转换必须通过 StateMachine 验证

### 4.2 状态自动更新 (FR-002)

**FR-002: 系统必须支持 session.idle 事件触发状态更新**

**当前状态**: 需要实现

**验收标准**:
- [ ] AC-002.1: 检测到文件创建/修改时自动触发状态检查
- [ ] AC-002.2: session.idle 事件触发时检查并更新状态
- [ ] AC-002.3: 状态更新失败时提供明确错误提示
- [ ] AC-002.4: 支持手动状态更新作为备用

### 4.3 状态历史记录 (FR-003)

**FR-003: 系统必须自动记录状态变更历史**

**当前状态**: history 字段存在但需要验证自动记录

**历史记录结构**:
```json
{
  "history": [
    {
      "timestamp": "2026-04-05T10:00:00Z",
      "from": "specified",
      "to": "planned",
      "triggeredBy": "@sdd-plan",
      "actor": "agent-name",
      "comment": "技术规划完成"
    }
  ]
}
```

**验收标准**:
- [ ] AC-003.1: 每次状态变更自动追加历史记录
- [ ] AC-003.2: 历史记录包含时间戳、触发源、操作者
- [ ] AC-003.3: 历史记录不可篡改（只追加）
- [ ] AC-003.4: 支持查询完整状态历史

### 4.4 依赖状态检查器 (FR-004)

**FR-004: 系统必须在状态变更前检查依赖状态**

**当前状态**: 需要实现

**依赖规则**:
- 如果 Feature A 依赖 Feature B，则 B 的状态必须 ≥ A 的目标状态
- 依赖检查在状态前进时强制执行
- 依赖检查在状态回退时警告提示

**验收标准**:
- [ ] AC-004.1: 状态前进前验证所有依赖就绪
- [ ] AC-004.2: 依赖未就绪时阻止状态变更并提示
- [ ] AC-004.3: 支持循环依赖检测
- [ ] AC-004.4: 提供依赖关系可视化（可选）

### 4.5 跨 Feature 聚合查询 (FR-005)

**FR-005: 系统必须自动聚合所有 Feature 状态**

**当前状态**: 根目录 state.json 存在但需要手动维护

**聚合查询机制**:
- **自动扫描**: 递归扫描 `.sdd/specs-tree-root/` 下所有 state.json
- **深度限制**: 递归深度限制为 10 层
- **循环检测**: 跳过符号链接避免无限循环
- **统计聚合**: 计算各状态/阶段的 Feature 数量分布
- **依赖图构建**: 根据 dependencies.on 字段构建依赖关系图
- **阻塞检测**: 识别因依赖未就绪而无法推进的 Feature

**验收标准**:
- [ ] AC-005.1: 支持自动扫描并聚合所有 Feature 状态
- [ ] AC-005.2: 聚合查询响应时间 < 500ms（100 个 Feature 以内）
- [ ] AC-005.3: 支持按状态/阶段/层级过滤查询结果
- [ ] AC-005.4: 提供阻塞 Feature 列表及原因

### 4.6 State Schema 统一 (FR-006)

**FR-006: 系统必须统一所有 Feature 使用 v2.0.0 State Schema**

**当前状态**: 多个 Schema 版本混用

**Schema v2.0.0 定义**:
```json
{
  "feature": "specs-tree-feature-id",
  "name": "Feature 名称",
  "version": "2.0.0",
  "status": "specified",
  "phase": 1,
  "level": 1,
  "parent": null,
  "files": {
    "spec": "spec.md",
    "plan": "plan.md",
    "tasks": "tasks.md",
    "readme": "README.md"
  },
  "dependencies": {
    "on": ["dependent-feature-id"],
    "blocking": []
  },
  "history": [],
  "createdAt": "2026-04-05T00:00:00Z",
  "updatedAt": "2026-04-05T00:00:00Z"
}
```

**验收标准**:
- [ ] AC-006.1: 提供 Schema 迁移工具
- [ ] AC-006.2: 支持从 v1.2.5/v1.2.11 迁移到 v2.0.0
- [ ] AC-006.3: 迁移过程保持向后兼容
- [ ] AC-006.4: 所有 Feature 迁移后验证通过

### 4.7 状态查询接口 (FR-007)

**FR-007: 系统必须提供状态查询接口**

**验收标准**:
- [ ] AC-007.1: 支持通过工具命令查询单个 Feature 状态
- [ ] AC-007.2: 支持查询所有 Feature 状态概览

### 4.8 状态报告 (FR-008)

**FR-008: 系统必须生成状态报告**

**报告内容**:
- 各状态 Feature 数量统计

- 长期停滞 Feature 告警（>7 天无状态变更）

**验收标准**:
- [ ] AC-008.1: 支持生成文本格式报告

## 8. 边界情况

### 8.1 错误处理 (EC-001)
- **EC-001**: state.json 格式错误时返回解析错误并提示修复
- **EC-002**: Feature 目录不存在时拒绝状态更新并提示创建
- **EC-003**: Schema 版本不匹配时提供迁移建议

### 8.2 并发处理 (EC-004)
- **EC-004**: 并发状态更新时使用文件锁防止数据损坏
- **EC-005**: 锁获取超时（>5s）时返回繁忙错误并建议重试

### 8.3 极端情况 (EC-006)
- **EC-006**: Feature 数量超过 1000 时聚合查询性能降级告警
- **EC-007**: 历史记录超过 10000 条时自动归档旧记录
- **EC-008**: 聚合查询超时（>2s）时返回部分结果并提示重试

### 8.4 依赖场景 (EC-009)
- **EC-009**: 检测到循环依赖时阻止状态前进并报错
- **EC-010**: Feature 目录被删除时自动从聚合查询结果中移除
- **EC-011**: Feature 被移动或重命名时自动更新依赖引用

### 8.5 多层嵌套场景 (EC-012)
- **EC-012**: 嵌套层级超过 10 层时返回警告并建议扁平化结构
- **EC-013**: 递归扫描时遇到符号链接跳过避免无限循环
- **EC-014**: 检测到循环嵌套时报错并拒绝操作
- **EC-015**: 递归扫描深度达到限制时返回已扫描结果并提示
