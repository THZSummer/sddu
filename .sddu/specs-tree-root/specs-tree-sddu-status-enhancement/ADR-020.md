# ADR-020: 两字段隔离模型 — Phase + Status 完全独立

## 状态
PROPOSED

## 背景

### 问题
当前 SDDU v2.1.0 的状态系统存在根本性设计缺陷：

1. **单字段混用**：`status` 字段（`WorkflowStatus`）同时承载两类语义：
   - 阶段语义（feature 在 SDDU 工作流的哪个阶段：specified/planned/tasked）
   - 流转语义（feature 是否可以继续、是否已终止等：suspended/terminated）

2. **双宇宙映射**：内部使用 `FeatureStateEnum`（9 状态：drafting/discovered/specified/planned/tasked/implementing/reviewed/validated/completed），但 schema 只接受 `WorkflowStatus`（6 状态：specified/planned/tasked/building/reviewed/validated）。`machine.ts`、`auto-updater.ts`、`dependency-checker.ts`、`tree-state-validator.ts` 各有一份映射逻辑，导致：
   - `drafting`、`discovered`、`completed` 无法存入 schema，被强制映射为 `specified` 或 `validated`，丢失语义
   - 映射逻辑分散在 4 个模块中，维护成本高，容易出现不一致

3. **不支持流转状态**：用户无法将 feature 标记为 suspended/terminated/merged

### 约束
- 无需向后兼容：旧格式 state.json 由 R5 一致性检测在用户升级后处理
- **干净切换**：不做增量过渡，不做历史逻辑兼容
- 必须区分"阶段推进"和"流转状态"两类操作

## 决策

采用**两字段隔离模型**：

1. **`phase`**（8 个阶段，统一 -ed 形态）：标记 SDDU 工作流阶段，由系统自动推进，单向不可逆
   - `registered → discovered → specified → planned → tasked → builded → reviewed → validated`

2. **`status`**（5 个流转状态）：标记 feature 的生命周期状态，由用户或系统设定
   - `tracked`（默认）、`completed`（系统自动）、`suspended`（用户）、`terminated`（用户）、`merged`（用户）

3. **完全独立**：两个字段互不推导、互不依赖。`@sddu 状态` 仅推荐 `status === "tracked" && phase !== "validated"` 的 feature 继续。

4. **删除双宇宙映射**：移除 `FeatureStateEnum`、`OldFeatureStateEnum`、`AgentWorkflowStateEnum` 及所有映射函数（`mapInternalStateToWorkflowStatus`、`mapFeatureStateToInternal`、`getStatePhase`）。代码中直接使用 `phase` 和 `status`。

## 后果

### 正面

1. **语义清晰**：`phase` 只管阶段，`status` 只管流转，各司其职
2. **消除映射层**：不再有 9↔6 的双宇宙翻译，代码复杂度显著降低
3. **扩展性**：新增 `phase` 或 `status` 值只需在一处（`schema-v3.0.0.ts`）定义，所有引用方自动生效（NFR-006）
4. **用户可操作**：通过 `@sddu 标记` 命令，用户可以直接管理 feature 的 suspended/terminated/merged 状态
5. **全新 feature 默认正确**：新建 feature 自动获得 `phase: "registered"`, `status: "tracked"`，无需手动设置

### 负面

1. **Schema 版本不兼容**：v2.x 的 state.json（单 `status` 字段）与 v3.0.0（`phase` + `status` 双字段）不兼容。读时需做转换，写时统一为 v3.0.0
2. **machine.ts 重构风险**：删除双宇宙映射需要仔细处理所有引用点，遗漏可能导致运行时错误
3. **Agent 模板大改**：sddu agent 的输出逻辑需要从"基于单一状态推导"改为"基于 phase + status 联合推导"

### 风险缓解

1. **R5 一致性检测**：插件版本升级后首次 `@sddu 状态` 自动触发检测，发现不符合新规则的 state.json → 请示用户 → 获同意后自动修复
2. **读时兼容**：StateLoader 读取时支持 v2.x 格式（自动推导 phase），不阻断
3. **FR-008 保护**：一致性检测修复时，不覆盖用户已设定的 `suspended`/`terminated`/`merged` 状态
4. **充分测试**：单元测试覆盖所有映射点、边界条件、回归 path

## 替代方案

### 方案 B：在现有 `status` 字段上扩展
在 `WorkflowStatus` 中增加 `suspended`、`terminated`、`merged`，保持单字段模型。

**被拒绝原因**：单字段混用正是当前问题的根源。扩展后 `validStatuses` 将膨胀到 9+ 个值（specified + planned + tasked + building + reviewed + validated + suspended + terminated + merged），语义更加混乱。`@sddu 状态` 的输出逻辑需要复杂的 if-else 分支来区分"阶段"和"流转状态"。

### 方案 C：三字段模型
增加 `lifecycleState` 字段，保留 `status`（阶段）和 `phase`（编号），形成三字段。

**被拒绝原因**：过度设计。`phase`（编号 1-8）和 `status`（阶段名如 "specified"）存在一对一映射关系，保留两者是冗余的。两字段（phase 字符串 + status 流转状态）足以表达所有语义。
