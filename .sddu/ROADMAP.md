## 执行摘要 (前 20%)

### 愿景陈述

SDDU 当前多 Feature 支持存在架构缺陷：specs-tree-root 缺少顶层规划文档、Feature 采用水平拆分而非树形结构、状态管理分散、文件命名不统一、Build 阶段文档缺失。本 Roadmap 旨在通过 5 个阶段的系统性优化，建立完整的树形 Feature 管理体系，实现父子状态自动汇聚、统一命名规范、完整的 6 阶段工作流支持，为 SDDU 的多 Feature 开发提供坚实的基础设施。

### 现状分析

#### 1. specs-tree-root 现状

```
.sddu/specs-tree-root/
├── README.md              # ✅ 存在
└── specs-tree-plugin-rename-sddu/  # ❌ 水平拆分
    └── validation-result.json
```

**缺失**:
- ❌ spec.md - 顶层规范
- ❌ plan.md - 技术规划
- ❌ tasks.md - 任务分解
- ❌ state.json - 全局状态

#### 2. Feature 结构现状

当前是**水平拆分**模式：
```
specs-tree-root/
├── specs-tree-feature-a/
├── specs-tree-feature-b/
└── specs-tree-feature-c/
```

期望的**树形结构**模式：
```
specs-tree-root/
└── specs-tree-online-bookstore/
    ├── README.md
    ├── state.json
    └── sub-features/
        ├── specs-tree-user-module/
        │   ├── state.json
        │   └── sub-features/
        │       ├── specs-tree-login/
        │       └── specs-tree-profile/
        └── specs-tree-commerce-module/
            └── sub-features/
                ├── specs-tree-cart/
                └── specs-tree-order/
```

#### 3. 状态管理现状

- `multi-feature-manager.ts` - 已有状态聚合逻辑，但仅支持水平聚合
- `subfeature-manager.ts` - 支持子 Feature 管理，但缺少嵌套支持
- **缺失**: 父子状态自动汇聚机制

#### 4. 文件命名现状

| 阶段 | 当前命名 | 目标命名 | 状态 |
|------|----------|----------|------|
| Discovery | discovery.md | 0-discovery.md | ⚠️ 不统一 |
| Spec | spec.md | 1-spec.md | ⚠️ 不统一 |
| Plan | plan.md | 2-plan.md | ⚠️ 不统一 |
| Tasks | tasks.md | 3-tasks.md | ⚠️ 不统一 |
| Build | build.md | 4-build.md | ⚠️ 不统一 |
| Review | review.md | 5-review.md | ⚠️ 不统一 |
| Validate | validation.md | 6-validate.md | ⚠️ 不统一 |

#### 5. Build 阶段现状

- ✅ `sddu-build.md.hbs` Agent 模板存在
- ❌ 缺少 `4-build.md` 文档模板
- ❌ 缺少 `4-build-summary.md` 总结模板
- ❌ Agent 模板缺少 Build 阶段详细说明

#### 状态汇聚逻辑

```typescript
// 父 Feature 状态自动计算规则
// 父 Feature 状态 = 最慢子 Feature 的状态

状态优先级 (从慢到快):
specified < planned < tasked < building < reviewed < validated

汇聚规则:
1. 所有子 Feature validated → 父 validated
2. 有子 Feature building → 父 building
3. 有子 Feature tasked → 父 tasked
4. 有子 Feature planned → 父 planned
5. 所有子 Feature specified → 父 specified
```
